# The Defier《逆命者》下一大版本策划案 V8

## 0. 文档定位

本文为《逆命者》`V10.0《天命对弈》` 的大版本策划案。

本版本只聚焦一件事：把当前“天道榜 PVP”从 `残影 / 镜像演武 + 赛后回执`，升级为真正的 `真人同场、服务端权威、双方体验都成立` 的 PVP 竞技系统。

它不是继续给现有 PVP 页面加更多解释面板，也不是把残影包装得更像真人。V10 的核心目标是：

> 让玩家能和另一个真实玩家进入同一场公平对局，在可读、可反制、可复盘的回合制节奏中证明自己的构筑理解，而不是被先手秒杀或被客户端上报胜负决定结果。

本文覆盖：

1. 当前 PVP 诊断
2. V10 版本定位
3. 设计支柱
4. 玩家模式结构
5. 核心对局规则
6. 反先手秒杀与 PVP 专属规则层
7. 服务端权威设计边界
8. 双方体验与 UI 包装
9. 赛季、奖励、残影与回放关系
10. 开发切片、测试门禁与上线验收
11. 风险、反目标与后续扩展

## 1. 当前 PVP 诊断

### 1.1 现在的 PVP 本质

当前 PVP 不是真人同场对局。

现有链路是：

1. 服务端 `/api/pvp/match` 从榜单和防守快照中挑一个对手。
2. 返回 `matchTicket`、对手 `rank`、`ghost`、`battleData`。
3. 前端把对手快照实例化为 `GhostEnemy`。
4. 本地战斗系统推进完整战斗。
5. 赛后前端尝试上报 `didWin`。
6. 服务端生产默认不信任客户端胜负上报，未启用权威结算时返回 `server_authority_unavailable`。
7. 前端降级为本地演武回执或本地练习账本。

所以当前系统已经拥有很好的 PVP 外围能力：

- 排行榜
- 防守快照
- 对手档案
- 风险画像
- 焦点约战
- 镜像演武兜底
- 赛后复盘
- PVP 商店与经济
- 结算回执文案

但它缺少真正 PVP 的核心：

- 没有 live match 房间
- 没有双方 presence
- 没有服务端战斗状态
- 没有服务端行动裁定
- 没有服务端事件日志驱动的胜负结算
- 没有断线重连、超时、投降、无效局等真人对局生命周期

### 1.2 当前 PVP 的优势

这些内容应保留并复用：

1. **榜单与经济层**
   `pvp_ranks`、`pvp_economy`、`pvp_match_history` 已经能承载段位、胜负、天道币、历史记录。
2. **对手阅读层**
   对手档案、DRI 风险、约战单、复盘结果已经形成强读题体验。
3. **练习兜底层**
   残影与镜像演武适合继续承担练习、离线体验、断线托管、回放对照。
4. **测试和门禁基础**
   当前 Node、后端、浏览器、移动端审计已经覆盖 PVP 页面、结算页和本地/服务端降级链路。

### 1.3 当前 PVP 的核心缺口

1. **不是真人同场**
   玩家对战的是本地 `GhostEnemy`，不是另一个实时行动的玩家。
2. **胜负不应来自客户端**
   服务端现在正确地默认拒绝客户端胜负上报。V10 要沿这个方向继续做服务端权威，而不是恢复信任客户端。
3. **PVE 爽感机制会破坏 PVP 公平**
   当前战斗有大量首击增伤、首回合抽牌、首回合灵力、低血乘区、破绽平伤、护盾转伤、额外回合等 PVE 爽点。它们直接进入排位会造成先手秒杀和非互动局。
4. **旧文案把残影兜底放在主 PVP 叙事里**
   V10 必须把“正式真人 PVP”和“镜像练习”拆开，否则玩家会误解正式榜单的可信度。

## 2. V10.0 版本定位

### 2.1 版本名

`V10.0《天命对弈》`

### 2.2 一句话定位

> 真正的真人回合制 PVP：服务端裁定、首轮防爆、构筑可读、失败可复盘。

### 2.3 玩家承诺

V10 向玩家明确承诺：

1. **你面对的是真人**
   正式 PVP 不再把残影当主对手。匹配成功后，双方进入同一场 live match。
2. **你不会被无互动秒杀**
   先手和后手都有明确补偿、伤害预算和资源上限，至少有一次读题与反制窗口。
3. **你输赢由服务器裁定**
   客户端只提交行动，不提交胜负。所有随机数、抽牌、触发、伤害和结算以服务端日志为准。
4. **你能看懂为什么赢或输**
   赛后给出关键回合、伤害来源、失误窗口、构筑短板和练习建议。
5. **PVE 爽感不被硬削**
   PVE 仍保留爆发、压血、护盾、额外行动等爽点；PVP 通过专属规则层隔离公平性。

### 2.4 版本主目标

1. 建立服务端权威 PVP 对局基础。
2. 建立 PVP 专属战斗规则，解决先手秒杀和非互动控制。
3. 把正式排位、好友约战、镜像练习、战局回放拆成清晰模式。
4. 复用现有 PVP 档案、DRI、赛后复盘、经济和赛季天道盘。
5. 给后续赛季规则、禁限表、观战、锦标赛留下扩展点。

## 3. 设计支柱

### 支柱 A：真人同场，不再伪装

正式 PVP 的最低标准是：

- 两名在线玩家进入同一个 `liveMatchId`
- 服务端保存 authoritative battle state
- 双方行动都通过服务端校验
- 服务端广播状态更新
- 胜负由服务端写入榜单、经济和历史

残影仍然保留，但它只能叫练习、演武、托管或回放对照，不能再被当成正式 PVP 的主对手。

### 支柱 B：公平比爆炸更重要

PVE 可以允许玩家做出“这把太爽了”的高爆发，但 PVP 必须避免“对方还没操作就结束”的非游戏体验。

V10 的公平底线：

- 首回合不能直接稳定斩杀。
- 所有一回合致胜都必须有前置 setup、可读信号和对手反制窗口。
- 回合控制不能剥夺对方完整行动权。
- 防守资源不能无上限转化为秒杀。
- 随机结果必须可复现、可回放、可审计。

### 支柱 C：保留构筑身份，但压平局外数值

正式排位不直接继承 PVE 后期膨胀属性。

排位应保留：

- 角色身份
- 卡组主轴
- 少量 PVP 合法命格 / 誓约 / 命途标签
- 外观、称号、赛季身份
- 对手档案和读题信息

排位应压平：

- 生命上限膨胀
- 永久力量膨胀
- PVE 临时事件 buff
- 章节/远征专属加成
- 不可验证的客户端快照数值

### 支柱 D：双方体验都要有“参与感”

体验目标不是只让胜者爽，而是让败者也觉得：

- 我至少行动过
- 我看到了对手意图
- 我知道哪里被抓住
- 我能通过练习或改构筑解决

因此 V10 必须包含：

- 等待反馈
- 出牌确认
- 服务端 ack/reject 提示
- 对手行动状态
- 回合倒计时
- 重连窗口
- 认输
- 再战
- 赛后复盘
- 回放

### 支柱 E：先做稳定竞技骨架，再做花哨赛季

首版不要同时引入大量新卡、新技能、新赛季机制。

首版优先级：

1. 服务端权威战斗能跑通。
2. PVP 规则层能防止先手秒杀。
3. 双客户端对局、断线、超时、重连、投降都可靠。
4. 排位和经济结算可信。
5. 再做赛季规则轮换和更多竞技内容。

## 4. 玩家模式结构

### 4.1 天命排位

正式竞技模式。

规则：

- 必须登录。
- 必须在线匹配真人。
- 不允许残影兜底。
- 使用 `pvp-live-v1` 规则版本。
- 服务端权威结算。
- 胜负影响段位、赛季积分、天道币和战绩。
- 断线和超时按正式规则处理。

适合：

- 想冲榜的玩家
- 想验证构筑强度的玩家
- 想获得赛季荣誉的玩家

### 4.2 问道练习

保留当前残影 / 镜像演武能力。

规则：

- 可以离线或在线。
- 可以打榜单残影、焦点镜像、本地练习对手。
- 不影响正式排位。
- 可以给少量练习奖励或复盘样本。
- 使用同一套 PVP 规则层模拟，但结算标记为练习。

作用：

- 降低新手进入 PVP 的门槛。
- 给玩家练习对策。
- 在匹配不足时保证仍有内容可玩。
- 作为真人 PVP 的教学和复盘对照。

### 4.3 好友约战

房间码或邀请制对战。

规则：

- 双方真人在线。
- 服务端权威。
- 默认不计排位分。
- 可选择标准规则或练习规则。
- 可允许 rematch。

作用：

- 满足社交对战。
- 方便测试、直播、展示。
- 为后续小型赛事打基础。

### 4.4 战局回放

基于服务端事件日志的回放系统。

规则：

- 不录屏。
- 回放来自 `pvp_live_match_events`。
- 可以快进、暂停、查看关键回合。
- 赛后复盘可以直接跳到关键回合。

作用：

- 解释胜负。
- 支撑仲裁和反作弊。
- 产出可分享内容。
- 让失败变成训练材料。

## 5. 核心对局流程

### 5.1 排位流程

1. 玩家进入天道榜。
2. 点击 `天命排位`。
3. 客户端提交入队请求。
4. 服务端匹配相近分段玩家。
5. 双方收到 `match_found`。
6. 双方进入准备界面。
7. 服务端发放初始手牌、先后手、规则摘要。
8. 双方可在准备阶段执行一次 `调息换牌`。
9. 双方 ready 后服务端锁定斗法谱与初始状态。
10. 对局按回合推进。
11. 玩家提交行动。
12. 服务端校验并广播结果。
13. 胜负产生后服务端结算。
14. 双方进入赛后复盘。

### 5.2 对局内回合流程

每个玩家回合：

1. 服务端触发回合开始效果。
2. 服务端抽牌、刷新灵力、处理持续效果。
3. 服务端广播可行动状态与回合倒计时。
4. 当前玩家提交行动。
5. 服务端校验费用、目标、卡牌位置、状态合法性。
6. 服务端把行动压入 effect queue。
7. 服务端按确定顺序结算效果。
8. 服务端广播 action ack 和新状态。
9. 当前玩家继续行动或结束回合。
10. 超时则服务端执行默认结束回合或自动防守。

### 5.3 默认节奏目标

V10 首版目标对局时长：

- 快速局：5-7 分钟
- 标准局：8-12 分钟
- 长局上限：15 分钟左右

首版回合倒计时：

- 准备阶段：30 秒
- 常规回合：75 秒
- 连续复杂行动后最多加时 15 秒
- 断线重连保护：60 秒
- 一局最多触发 2 次断线保护

### 5.4 非游戏防线

真人 PVP 的最低体验不是“绝对平均”，而是双方都必须真实参与过对局。

首版必须满足四条硬线：

1. **首轮不可无交互斩杀**
   在对手没有进入过自己的行动阶段前，任何伤害结算都不能把对手生命降到 0。超出的伤害按预算或最低生命保护截断，并记录为 `prevented_by_budget`。
2. **爆发必须有可读前兆**
   第 3 回合后可以出现高爆发，但需要来自前一回合的可见 setup，例如破绽、蓄势、姿态、手牌数异常或公开 buff。没有可见 setup 的单回合爆发仍受单效果封顶。
3. **控制不能剥夺完整回合**
   排位内不允许跳过对方整回合、清空整手牌、无限费用税或无限减速。控制应表现为减伤、虚弱、临时费用 +1、少量弃牌或公开延迟威胁。
4. **长局必须自然收束**
   第 9 回合开始进入 `天劫压境`：每个整轮结束后双方各受到递增的不可护盾转化压力伤害，数值为 2 / 4 / 6 / 8，上限 8。该伤害只用于防拖局，不触发吸血、反伤、护盾转伤、低血增伤或击杀奖励。

## 6. PVP 标准化与构筑来源

### 6.1 标准化原则

正式排位的公平性来自两个层面：

1. 规则公平：同一套 PVP 规则版本。
2. 数值公平：同一生命、灵力、手牌、资源曲线基线。

### 6.2 排位基础数值

`pvp-live-v1` 首版基线：

- 生命：50
- 初始手牌：5
- 调息换牌：准备阶段可选择 0-2 张手牌洗回牌库并补抽等量
- 每回合基础抽牌：5
- 基础灵力：3
- 手牌上限：10
- 卡组：20 张
- 单卡复制上限：2
- 先手首回合不可获得额外抽牌或额外灵力
- 后手获得 `后发护印`：第一回合开始获得 6 点不可转伤护盾

### 6.3 PVP 构筑槽

新增 `斗法谱` 作为 PVP loadout。

内容：

- 角色
- PVP 合法卡组
- 主动技能等级，排位内归一到指定等级
- 1 个 PVP 身份槽：命格 / 誓约 / 命途标签三选一
- 外观与称号

不包含：

- 当前 PVE 血量
- 当前 PVE 灵石
- 当前章节临时 buff
- 冒险事件的短期准备
- 未经服务端校验的客户端属性

身份槽原则：

- 首版排位只允许 1 个 PVP 身份槽，固定为 PVP 版 T1。
- 章节、远征、法宝、法则、灵契、传承、地图事件 buff 不进入排位数值。
- 身份槽只能定义风格，不能同时叠加首回合抽牌、首回合灵力、首击增伤和低血乘区。

### 6.4 PVE 与 PVP 的关系

PVE 继续提供：

- 卡牌理解
- 角色熟练度
- 外观
- 称号
- 回放样本
- 练习目标

PVE 不直接提供：

- 排位数值压制
- 更高生命
- 更高初始灵力
- 无上限首轮爆发

这样能避免老玩家靠 PVE 长线数值碾压新玩家，同时保留主线与 PVP 的学习关系。

## 7. 反先手秒杀规则

### 7.1 为什么必须做 PVP 专属规则层

当前 PVE 战斗有大量高爽感机制：

- 首击增伤
- 首回合抽牌
- 首回合灵力
- 低血增伤
- 破绽平伤
- 易伤追击
- 护盾增幅
- 护盾转伤
- 额外回合
- 交换血量
- 硬眩晕
- 高额反射

这些机制在 PVE 中能制造爽点，但在 PVP 中会制造非游戏体验。

因此 V10 不应全局削弱卡牌，而应增加 `PVP Rule Override`：

- PVE 使用原始效果。
- PVP 排位使用 PVP 覆写规则。
- 问道练习默认也使用 PVP 覆写规则，帮助玩家练正式环境。

### 7.2 先后手规则

先手：

- 正常抽 5。
- 第一回合基础灵力 3。
- 第一回合额外抽牌为 0。
- 第一回合额外灵力为 0。
- 第一回合对玩家造成的总伤害受首回合伤害预算限制。

后手：

- 正常抽 5。
- 第一回合开始获得 `后发护印`。
- `后发护印` 提供 6 点护盾。
- 该护盾不能被 `blockBurst`、反击或其他转伤效果消耗成伤害。
- 后手第一回合若未受伤，护印在回合开始后消失，不转化为优势。

### 7.3 首两回合伤害预算

首版：

- 第一名玩家第 1 回合对敌方玩家造成的总伤害上限：18
- 第二名玩家第 1 回合对敌方玩家造成的总伤害上限：22
- 双方第 2 回合单回合对玩家造成的总伤害上限：28
- 从第 3 回合开始取消硬预算，但继续保留单效果封顶和禁限表

预算结算口径：

- 预算按“行动方本回合对敌方玩家造成的最终伤害”统计。
- 易伤、破绽、力量、首击、穿透等修正先结算，再进入预算截断。
- 溢出部分不转化为治疗、护盾、反伤、吸血或其他收益。
- 服务端必须记录 `damage_prevented_by_budget`，赛后复盘要能显示“本回合被规则压制了多少爆发”。
- 预算只限制玩家生命伤害；抽牌、回灵、上状态、获得护盾仍按各自上限处理。

目的：

- 防止稳定 T1 斩杀。
- 保留强开局压力。
- 让对手至少获得一次操作和一次反制机会。

### 7.4 关键机制覆写表

| 机制 | PVE | PVP |
| --- | --- | --- |
| `firstAttackBonusPerBattle` | 多来源可叠 | 只取最高来源，或总上限 +6 |
| `firstTurnDraw` | 多来源可叠 | 首版排位禁用，首回合额外抽牌为 0 |
| `firstTurnEnergy` | 多来源可叠 | 首版排位禁用，首回合额外灵力为 0 |
| `lowHpDamageBonusPct` | 可多来源乘算 | 只取最高来源，上限 25%，第 2 回合后生效 |
| `mark` | 层数可直接转平伤 | 被消耗时平伤上限 4 |
| `vulnerable` | 可叠加提高伤害 | 层数上限 2 |
| `penetrate` | 可直接穿盾扣血 | 单效果上限 10 |
| `blockBurst` | 可高比例转伤 | 消耗上限 10，倍率不超过 1.0 |
| `retainBlock` | 可跨回合留盾 | 跨回合护盾上限 10 |
| `overhealToBlockRatio` | 可高比例转盾 | 上限 0.5 |
| `thorns` / `reflect` | 可高额反伤 | 单次返伤上限 8，满额免伤改为减伤 |
| `extraTurn` | 额外回合 | 禁用，改为抽牌或减伤 |
| `swapHpPercent` | 交换血量百分比 | 禁用 |
| 硬 `stun` | 跳过对手回合 | 禁用，改为 `weak` 或费用税 |

### 7.5 高风险卡牌与效果处理方向

不在策划阶段逐张重写卡牌文本，而是按 effect type 建规则。

首版必须覆盖：

- `consumeAllEnergy`
- `penetrate`
- `executeDamage`
- `blockBurst`
- `debuff: vulnerable`
- `buff: extraTurn`
- `swapHpPercent`
- `stun`
- `reflect`
- `thorns`
- `draw`
- `energy`
- `freeCard`

### 7.6 PVP 禁限表

首版禁用：

- `extraTurn`
- `swapHpPercent`
- 硬跳过对手回合的 `stun`
- 满额 `reflect` 免伤反弹

首版限制：

- 0 费抽牌
- 0 费回灵
- 低费大额穿透
- 护盾转伤
- 高额反伤
- 多来源首击加成
- 多来源低血乘区

首版允许：

- 低费节奏牌
- 有上限的爆发
- 有 setup 的斩杀
- 有反制窗口的控制
- 有封顶的护盾反击
- 有代价的过牌和回灵

### 7.7 斗法谱合法性与反退化

斗法谱不是把 PVE 最强卡组直接搬进排位。首版需要用合法性检查保证卡组有交互、有曲线、有弱点。

基础构筑约束：

- 20 张卡。
- 单卡复制上限 2。
- 0 费卡最多 4 张，其中 0 费抽牌 / 回灵 / 高伤害牌合计最多 2 张。
- 费用曲线必须至少包含 8 张 1-2 费牌，避免起手无法行动。
- 直接伤害牌最多 10 张，纯防守或纯过牌牌也不得超过 10 张。
- 至少 4 张非伤害交互牌，例如护盾、虚弱、清状态、姿态、防爆发、低额移除护盾。

反退化规则：

- 任何能把费用降到 0 的效果，不能作用于本回合已被标记为 `highImpact` 的伤害牌。
- 任意检索、随机生成、临时卡复制必须来自 PVP 合法池，且单回合最多生成 2 张可打出的牌。
- 额外抽牌、额外灵力、复制出牌、回收弃牌不能形成闭环；同一张牌同一回合最多被重复结算 1 次。
- 如果一个效果没有写入 `pvp-live-v1` 支持矩阵，排位默认拒绝或转写为安全等价效果，不能按 PVE 原文兜底执行。

## 8. 服务端权威设计边界

### 8.1 必须服务端裁定的内容

正式 PVP 中，下列内容必须服务端裁定：

- 匹配
- 先后手
- 洗牌
- 抽牌
- 起手手牌
- 行动合法性
- 费用支付
- 目标合法性
- 随机结果
- 效果触发顺序
- 伤害
- 护盾
- 状态
- 胜负
- 段位
- 货币
- 赛季任务

客户端只负责：

- 展示当前服务端状态
- 提交玩家行动
- 播放预测动画
- 处理输入和提示
- 接收服务端 ack/reject

### 8.2 战斗引擎形态

必须新增或抽离一个纯逻辑引擎：

`server/pvp-live/engine`

引擎特征：

- 不访问 DOM
- 不依赖浏览器 timer
- 不直接播放动画
- 不读取客户端全局对象
- 使用显式输入 state + action
- 输出 nextState + events
- 所有随机来自 seed
- 每个事件有 sequence
- 可在 Node 测试中重放

### 8.3 状态与事件

服务端保存两类数据：

1. 当前状态 snapshot
2. 顺序事件 log

snapshot 用于：

- 快速重连
- 服务端恢复
- 当前 UI 渲染

event log 用于：

- 回放
- 复盘
- 仲裁
- 调试
- 反作弊

### 8.4 单实例首版约束

当前后端使用 SQLite。V10 首版可以明确限制为单实例服务：

- live room 在单 Node 进程内管理
- SQLite 记录 durable match/event/history
- 重启时未完成对局按无效局或补偿规则处理

后续如果要多实例部署，需要引入：

- Redis presence
- Redis pub/sub 或专用 room service
- sticky session 或统一 WebSocket 网关

首版不要把多实例分布式当必要条件，否则范围会过大。

## 9. 双方体验设计

### 9.1 匹配体验

匹配界面必须显示：

- 当前模式
- 预计等待
- 匹配范围
- 可取消按钮
- 找到对手后的确认态
- 对手准备状态
- 规则版本

匹配超过一定时间：

- 排位：提示扩大匹配范围或继续等待，不切残影。
- 练习：可切残影。
- 好友约战：提示等待对方进入。

### 9.2 对局 HUD

对局中必须清晰显示：

- 我方生命、护盾、灵力、手牌、弃牌、抽牌堆
- 对方生命、护盾、手牌数、剩余牌数、状态
- 当前回合方
- 回合倒计时
- 对方是否在线
- 本回合伤害预算剩余
- 服务端同步状态
- 行动提交中 / 已确认 / 被拒绝

### 9.3 行动反馈

客户端可以做本地预演，但不能先行改变权威状态。

推荐流程：

1. 玩家拖牌或点击出牌。
2. 客户端做合法性预检查。
3. UI 显示 `待裁定`。
4. 客户端发送 action。
5. 服务端 ack 后播放确定动画。
6. 服务端 reject 时回滚预演并显示原因。

### 9.4 断线和超时

断线规则：

- 掉线后保留 60 秒重连窗口。
- 对手看到 `对方重连中`。
- 一局最多 2 次重连保护。
- 超过保护次数或时间，判定投降或自动托管。

超时规则：

- 常规回合超时自动结束回合。
- 若玩家本回合未行动且处于高压状态，可自动打出默认防御策略。
- 连续 2 次超时后提示将判负。
- 连续 3 次超时判负。

### 9.5 赛后复盘

赛后必须给双方：

- 胜负
- 段位变化
- 货币变化
- 关键回合
- 最大伤害来源
- 首轮压力是否被预算压制
- 对手核心构筑标签
- 我方失误 / 优势标签
- 推荐练习残影
- 回放入口
- 再战 / 返回 / 查看榜单

失败文案要避免羞辱玩家，重点解释：

- 你输在哪里
- 对方赢在哪里
- 下次怎么应对

## 10. 赛季、奖励与长期循环

### 10.1 与赛季天道盘关系

V10 不推翻 V9 的赛季天道盘。

它把其中的 `PVP 验算线` 从残影验证升级为：

- 真人排位验证

- 残影练习验证
- 战局回放验证

### 10.2 排位奖励原则

排位奖励以荣誉和展示为主：

- 段位
- 称号
- 法相
- 卡背
- 对局边框
- 回放铭牌
- 赛季纪念

谨慎给予：

- PVE 数值增强
- 直接影响 PVE 通关的强卡
- 会让强者更强的战斗资源

### 10.3 天道币

天道币继续保留，但来源拆分：

- 排位胜利：主要来源

### 10.4 残影的新定位

文案必须明确：

- 残影练习不影响正式排位。
- 镜像演武不是真人排位。
- 服务端权威回执只来自 live match。

## 12. 验收标准

### 12.1 玩法验收

必须满足：

- 两个真人账号能进入同一场 live match。
- 任意一方不能在对方未操作前稳定秒杀。
- 后手至少有一次有效反制机会。
- 出牌、抽牌、随机、伤害、胜负都能从服务端日志重放。
- 断线能重连，超时能收口。
- 败方能看到可解释的复盘。

### 12.2 技术验收

必须满足：

- 客户端不提交胜负。
- 服务端结算段位和经济。
- 重复 action 被拒绝。
- 非当前回合 action 被拒绝。
- 非法卡牌 / 费用 / 目标被拒绝。
- 随机数可复现。
- event log 能重放出同样结果。
- 断线重连能恢复到最新 revision。

### 12.3 平衡验收

首版至少通过：

- 首回合 OTK 压测
- 先后手胜率模拟
- 高爆发构筑模拟
- 护盾转伤模拟
- 破绽 / 易伤爆发模拟
- 低血反杀模拟
- 控场非互动模拟

硬门槛：

- 10,000 场标准构筑模拟中，先手总胜率必须在 47%-53%。
- 任意基准对阵对先手胜率不得超出 45%-55%。
- 后手第一次行动前死亡次数必须为 0。
- 对局时长 P95 <= 12 分钟，P99 <= 15 分钟。
- 第 14 整轮后仍未结束的对局数量为 0。

### 12.4 文案验收

必须完成：

- 正式 PVP 不再称残影为正式对手。
- 残影被明确标为练习。
- 结算回执能区分排位、练习、好友、无效局。
- 玩家介绍页不再把“镜像演武兜底”写成正式 PVP 的主承诺。

### 12.5 发布验收

本地开发完成后，至少需要：

- `npm run build:pages`
- `npm run test:node`
- live PVP backend E2E
- two-client browser audit
- PVP mobile audit
- PVP result mobile audit
- release gate marker 覆盖 true PVP

生产部署前按正式线上规则额外验证：

- `curl -I https://080305.xyz/`
- `curl -sS https://080305.xyz/api/health`
- 注册、登录、存档、PVP live match smoke
- `ssh cloud119 'systemctl is-active the-defier-backend; nginx -t'`

## 13. 风险与反目标

### 13.1 最大风险

最大风险不是匹配，而是战斗内核迁移。

如果战斗仍然只在浏览器 `battle.js` 中本地推进，服务端只收结果，那就不是真正 PVP。

### 13.2 技术风险

1. `battle.js` 与 DOM、动画、timer、客户端对象耦合重。
2. 当前 SQLite 后端不适合多实例实时房间。
3. WebSocket 引入新的连接、重连、部署、测试复杂度。
4. 规则覆写若分散写在卡牌上，后续难维护。
5. 旧 PVP 文案和测试会与新模式冲突。

### 13.3 玩法风险

1. 规则限制过重会让 PVP 没有爽感。
2. 规则限制过轻会回到先手秒杀。
3. 排位若吃 PVE 长线数值，会变成老玩家碾压。
4. 控制流若不限制，会制造“不能玩”的挫败。
5. 匹配人数不足时，排位等待可能变长。

### 13.4 反目标

V10 首版不做：

- 大规模新卡包
- 多人混战
- 观战直播
- 锦标赛
- 公会战
- 跨服分布式房间
- 复杂 ban/pick
- 全卡池完全无禁限排位
- 用客户端胜负上报恢复正式榜单结算

这些可以作为后续版本扩展。

## 15. 最终版本目标

`V10.0《天命对弈》` 完成后，玩家应能这样描述 PVP：

> 这不是打系统残影，而是真人斗法。先手有压力但秒不掉人，后手有反制窗口；每一次出牌都由服务器裁定，输了能看回放知道哪一回合被抓住，赢了也能把战局沉淀成自己的赛季履历。

这就是 V10 的核心成功标准。
