# The Defier Codex 开发蓝图 (2026 Q1-Q2)

## 1. 目标
- 把游戏迭代成“可持续更新”的策略肉鸽：每周有新增可玩内容，每次更新可控回归风险。
- 核心体验优先级：
  1. 战斗可读性
  2. 流派构筑深度
  3. 局外成长驱动力

## 2. 里程碑

### M1 (当前周期，1-2 周)
- 战斗日志分级 + 历史面板（便于玩家复盘）
- 首局新手引导（主菜单 + 首场战斗）
- 自动化回归稳定（Node checks + Playwright 场景）

**验收标准**
- 玩家可通过快捷键打开日志历史并按类型过滤。
- 新玩家首次进入可获得关键操作提示，提示不重复刷屏。
- `tests/run_node_checks.sh` 和浏览器审计脚本全通过。

### M2 (2-4 周)
- 新增 2 套完整流派（含核心卡、支撑卡、终结卡）
- 新增 8-12 个事件（含高风险高收益分支）
- 试炼/锻炉节点收益权重再平衡

**验收标准**
- 每套流派至少 15 张可用卡，且可通关前 5 重。
- 事件选择形成明确短期/长期 tradeoff。

### M3 (4-8 周)
- 局外传承系统 V1（可重置）
- 赛季化异步 PvP 规则面板
- 经济平衡看板（产出、消耗、留存）

**验收标准**
- 失败后仍有稳定成长反馈。
- PvP 规则版本可展示且可追溯。

## 3. 模块拆解

### 3.1 战斗可读性
- 日志分类：`damage/status/system/reward/warning`
- 历史面板：最近 120 条，支持筛选
- 可视化提示：关键机制触发需有明确文案

### 3.2 新手引导
- 主菜单一次性提示：从“新的轮回”开始
- 首场战斗三段提示：看意图 -> 出牌 -> 结束回合/复盘
- 本地状态持久化，避免重复打扰

### 3.3 内容生产线
- 卡牌/事件/敌人均采用模板化字段
- 提交前执行字段与引用校验
- 每个版本新增内容必须附可玩路径与回归点

## 4. 开发工作流（供 Codex 持续执行）
1. 先写最小可用改动，避免大爆改。
2. 每个功能点必须附一个“可验证证据”：
   - 脚本输出
   - 截图
   - `render_game_to_text` 状态
3. 发现新错误先修第一处阻断错误，再继续扩展。
4. 变更后统一跑：
   - `tests/run_node_checks.sh`
   - Playwright 审计场景
5. 在 `progress.md` 追加“变更 + 验证 + 后续事项”。

## 5. 风险与约束
- 云端鉴权与游客模式必须可并行，不得互相阻断。
- 新增机制不得破坏移动端可点击区域。
- 任何版本上线前必须通过关键流程：主菜单 -> 新游戏 -> 地图 -> 战斗 -> 奖励。
