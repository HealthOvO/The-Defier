# The Defier《逆命者》V9.0《命盘战争》功能需求文档 V6

## 1. 文档用途

本文是 `designer_major_upgrade_planning_v7.md` 的配套需求文档。

本文不写工时与排期，只定义一件事：

> `V9.0《命盘战争》` 首发必须交付哪些内容，哪些能力可以放到第二阶段，以及哪些工程边界不能越界。

## 2. V9.0 必做范围

`V9.0` 至少要交付以下 `5` 大板块：

1. **天道敕令周循环板**
2. **命盘谱系**
3. **界痕抉择 / 契约后效**
4. **跨模式供血与赛季题面联动**
5. **数据结构与自动化门禁扩展**

缺任何一块，`V9.0` 都会从“大版本结构升级”退化成“系统深化合集”。

## 2.1 首发范围（建议）

首发建议优先承诺以下内容：

1. 天道敕令周循环板的最小可用版
2. 命盘谱系的记录层
3. 界痕抉择 / 契约后效的最小模板集
4. 洞府 / 奖励页 / payload 的长期字段同步
5. Node 合同测试与浏览器审计扩线

这五块已经足够闭合出一条完整的主循环：

`本周题面 -> 章节押注 -> 章末结算 / 追账 -> 长期谱系 -> 下一轮目标`

## 2.2 第二阶段预留

建议第二阶段再继续放大以下能力：

1. 赛季天道盘的完整阶段制包装
2. 周结算 / 季押卷 / 结业验证链
3. 谱系奖励层的更多展示物与轻量特权
4. 更多界痕模板与更长持续章数
5. 更深的 PVP / 无尽终局验证

原因是这些能力会显著放大版本表达，但不是让 `V9.0` 主循环成立的第一优先级。

## 2.3 首发后优先扩展（建议）

如果首发四层结构已经稳定成立，下一轮最值得优先扩的不是新页面，而是把当前的周目标、议程后果与季盘阶段推进收束成一条更硬的结算链：

1. 周结算后形成 `正卷 / 险卷 / 欠卷`
2. 欠卷沉淀成可追踪的 `研究债账包`
3. `seasonBoard ranking` 只负责把债账组织成 `验证状`
4. 玩家把当前路线投向挑战 / PVP / 无尽去做真正验证
5. 结果再回写到谱系、后效与下一轮敕令目标

这块扩展的目标，是把“长期结构”从组织层推进成真正可经营的押注层，而不是再加一层摘要 UI。

## 3. 玩家主流程需要升级成什么样

`V9.0` 后，玩家主流程需要演进为：

1. 在观察站 / 赛季入口读取当前题面
2. 从天道敕令中选择本周修行方向
3. 带着目标进入章节与洞府议程
4. 在章内推进关键节点、处置、契约与偏题纠正
5. 在章末得到结题 / 残卷 / 追账 / 界痕结果
6. 把结果转成谱系成长和下一轮目标
7. 在 PVP / 无尽 / 镜像验证中检验这条路线

## 4. 天道敕令周循环板

### 4.1 要做什么

做一个跨模式周目标系统，给玩家明确的每周修行面板。

### 4.2 功能要求

首发必须支持至少三条任务线：

- `远征线`
- `训练线`
- `对抗线`

每周至少给出 `6-9` 条任务。

每条任务至少包含：

- 标题
- 所属线路
- 进度
- 完成条件
- 奖励预览

### 4.3 行为要求

1. 任务进度必须自动读取既有玩法结算，不允许玩家手动补录。
2. 周刷新后必须保留上一周结论摘要。
3. 旧任务失效、周切换与未领奖状态必须可恢复。

### 4.4 UI 要求

首发至少应在以下位置可见：

- 洞府总览
- 观察站入口或赛季入口
- 奖励页 / 本周总结摘要

## 5. 命盘谱系

### 5.1 要做什么

把玩家的长期玩法倾向沉淀成可追踪、可展示的专精档案。

### 5.2 功能要求

首发至少记录以下四类谱系：

- 角色谱系
- 流派谱系
- 节点谱系
- 研究谱系

研究谱系首发必须记录：

- 议程类型
- 处置风格
- 契约风格
- 结题 / 残卷倾向

### 5.3 行为要求

1. 同一局同一条谱系不得重复计数。
2. 记录层和奖励层必须可拆分。
3. 首发可以只有展示和检索收益，不强制给重数值奖励。

## 6. 界痕抉择 / 契约后效

### 6.1 要做什么

让本章议程结果能够影响后续章节，而不是在章末立即归零。

### 6.2 功能要求

首发至少支持 `3` 类模板：

1. 路线偏置
2. 敌情 / 风险偏置
3. 归卷 / 档案收益偏置

每条后效至少包含：

- 来源
- 持续章数
- 正向收益
- 负向代价
- 当前状态

### 6.3 行为要求

1. 契约兑现与契约未兑现必须产生不同后效。
2. 残卷回收可生成较轻后效，但不能等同成功结题。
3. 跨章后效必须支持清档、迁移、战败和章节结束后的正常收口。

### 6.4 UI 与投影要求

这些字段至少要同步到：

- 洞府总览
- 章节摘要 / 远征辅助区
- 奖励页
- `render_game_to_text` / payload

## 7. 跨模式供血与赛季题面联动

### 7.1 要做什么

定义“观察站 / 远征 / 洞府 / PVP / 无尽”之间的共同目标关系。

### 7.2 功能要求

首发至少满足：

1. 观察站样本可以成为周目标或赛季目标来源
2. 远征议程结果可以推进周目标或谱系
3. PVP / 无尽结果至少能作为一条验证线推进目标
4. 至少一条奖励页摘要会明确说明“这次结果如何服务当前主轴”

### 7.3 首发限制

1. 首发不强制做完整赛季阶段 UI
2. 首发不强制做大量新数值资源
3. 首发重点是“目标统一”和“结果回流”，不是另起新模式

### 7.4 首发后优先扩展：周结算 / 季押卷 / 结业验证

这一轮扩展至少应满足：

1. 周结算后会生成明确的 `押卷结果`，而不是只刷新任务完成度。
2. `押卷结果` 至少区分：
   - `正卷`
   - `险卷`
   - `欠卷`
3. `欠卷` 至少记录：
   - 来源
   - 欠账主题
   - 清账窗口
   - 建议验证入口
   - 清账收益
   - 失败追加负担
4. `seasonBoard` 在 `ranking` 阶段至少给出 `1` 个主验证状与 `1` 个可选旁验证状。
5. 结业验证必须继续复用现有高压入口，不额外要求首发级独立模式：
   - 挑战验证
   - PVP 焦点验证
   - 无尽镜像验证
6. 结业验证结果至少影响：
   - 赛季阶段推进或定榜结论
   - 谱系证明记录
   - 欠卷清账或降级
7. owner 必须拆清：
   - `agenda` 生成债账
   - `seasonBoard` 组织验证状
   - `aftereffect` 持续施压
   - `lineage` 记录清账风格

## 8. 数据结构与自动化门禁扩展

### 8.1 必须扩的字段层

1. 周目标状态
2. 谱系记录状态
3. 界痕 / 契约后效状态
4. 跨模式结算所需的最小摘要字段

### 8.2 Node 门禁要求

至少补以下合同测试：

1. 周目标结算
2. 谱系计数不重复
3. 后效跨章持续
4. 存档迁移不丢字段
5. 契约兑现 / 未兑现 / 残卷三路分叉
6. `正卷 / 险卷 / 欠卷` 三类结算结果的状态迁移
7. 冲突场景下 owner / precedence matrix 只保留单一驱动者

### 8.3 浏览器门禁要求

至少补以下 probe：

1. 洞府页看到长期目标和后效信息
2. 奖励页看到主轴摘要
3. 成功与失败路径都能维持字段一致
4. 桌面与移动端都不挤爆布局
5. 验证状、欠卷与押卷结果在洞府 / 奖励页 / payload 三处一致

## 9. 不在首发承诺的内容

`V9.0` 首发不建议承诺：

1. 全面重写 battle runtime
2. 完整重做地图节点生态
3. 大规模 PVP 赛制翻新
4. 单独新开一个局外模式

这些方向不是不能做，而是不该和 `V9.0` 的“长期结构搭建”抢主线。

## 10. 首发验收标准

如果 `V9.0` 首发成功，应满足：

1. 玩家知道本周练什么
2. 玩家知道自己长期主修什么
3. 玩家知道自己欠了什么账、留下了什么后果
4. 玩家能感受到不同模式在推进同一条主轴
5. 自动化门禁能稳定证明上述状态在 UI 和 payload 中一致

如果以上五点有两点以上无法稳定成立，则说明 `V9.0` 仍然停留在“信息很多”，还没有成为真正的大版本结构升级。
