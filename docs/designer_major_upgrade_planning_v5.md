# The Defier《逆命者》下一大版本策划案 V5

## 0. 文档定位

本文为《逆命者》`V7.0《命途裂变》` 的完整策划案。
目标是把现有 `V6.1` 迭代线中已经建立的命途、赛季、挑战、远征、PVP 经济体系，升级为“高重玩 + 高可读 + 高复盘 + 高稳定”的 7.0 发布级框架。

本策划案包含：

1. 版本目标与定位
2. 玩法与内容升级方案
3. 难度综合评定与重标定方案
4. 开发切片与交付节奏
5. QA 门禁与“无阻塞缺陷”验收标准
6. 风险与回滚策略

## 1. 版本定位（V7.0《命途裂变》）

### 1.1 一句话定位

> 让每一局都有“中局目标与后果”，让每一次失败都能产出“可复盘价值”，让每个模式都互相供给动机与成长资源。

### 1.2 版本主目标

1. **玩法目标**
   从“有很多系统可玩”升级为“系统之间会互相拉扯决策”。
2. **内容目标**
   增加不是简单体量堆叠，而是增加“路线分流、后果分流、结局分流”。
3. **可玩性目标**
   同角色连续 10 局，玩家仍能清晰感知局中目标不同、节奏不同、结算价值不同。
4. **稳定性目标**
   关键流程达到“自动化回归 + 手工场景复核 + 无新增阻塞缺陷”的上线门禁。

## 2. 当前版本问题诊断（为 V7.0 定义升级靶点）

### 2.1 现状优势

1. 命途样本板、章节适配、构筑摘要已经具备策略基础。
2. 无尽轮回赛季化（赛季词条、季签、账本）已形成长期目标框架。
3. 挑战与观察站具备复盘入口，PVP 经济链路具备持续驱动。
4. Node 与浏览器审计链已比较完整，具备持续迭代条件。

### 2.2 关键瓶颈

1. **局中目标深度仍不稳定**
   目前主要是“通关导向”，中局目标（悬赏/追猎/势力后果）还不够强制影响决策。
2. **章节后果表达不够集中**
   玩家完成章节后，对“我为什么是这个结局”的理解仍偏分散。
3. **难度感知与真实危险不总一致**
   玩家在部分场景会感到“看起来不危险但突然暴毙”，需要更明确危险分层和预警。
4. **模式联动可继续做深**
   主线、挑战、无尽、PVP 虽有关联，但“切模式动机”仍可更明确。

## 3. V7.0 设计支柱

### 支柱 A：每章都要有“主动承接”的中局目标

章节开始时必须出现可选择目标（悬赏/追猎/势力委托），并影响后续路线与奖励。

### 支柱 B：决策要有后果，后果要能复盘

每个关键抉择都要在命盘与章节结算中留下可见记录。

### 支柱 C：难度要有梯度，也要有解释

玩家不仅要被挑战，还要看懂“为什么难、怎么应对”。

### 支柱 D：模式之间必须互相供血

主线、无尽、挑战、PVP 都要为其他模式提供真实价值（素材、线索、策略模板、奖励倍率）。

### 支柱 E：复杂度上升，但可读性必须同步上升

每新增一个系统，至少提供一个“摘要面板 + 风险提示 + 结果回看”出口。

## 4. 核心玩法升级（必做模块）

## 4.1 章节悬赏 2.0（主驱动模块）

### 功能描述

每章开始提供 `3` 条悬赏，承接 `1-2` 条，类型覆盖战斗/路线/构筑/极限。

### 设计细则

1. 悬赏必须展示：
   完成条件、失败条件、奖励、推荐流派、潜在冲突（如与势力关系冲突）。
2. 悬赏允许事件改写：
   特定事件可“改签悬赏”，提升收益但增加条件。
3. 悬赏完成后即时反馈：
   章节内完成时立即弹出“完成收益 + 命盘评分增益”。

### 对可玩性的提升

从“打过就行”升级为“我这章在做什么任务”。

## 4.2 势力关系与敌意系统 2.0

### 功能描述

每章至少 `2-3` 个活跃势力，态度分为支持/中立/敌对，影响节点和战斗。

### 设计细则

1. 势力态度变化必须附理由（最近一次变化日志）。
2. 势力必须影响至少三项：
   商店池、事件权重、战斗增援/伏击。
3. 势力关系进入章节结算摘要：
   玩家能看到“本章核心关系转折”。

### 对可玩性的提升

路线不再只是收益选择，还变成政治立场选择。

## 4.3 仇敌追猎系统（主叙事驱动）

### 功能描述

特殊精英进入“追猎对象池”，可逃逸、复现、投靠势力或成为 Boss 护卫。

### 设计细则

1. 首发目标：`8-12` 名记名仇敌。
2. 每名仇敌至少有 `2` 条战斗变体和 `1` 条事件线索。
3. 命盘记录仇敌结局：
   击破、放走、交易、仇敌进阶。

### 对可玩性的提升

单局会产生“个人宿敌故事线”，提高记忆点与分享欲。

## 4.4 命途样本板强化（决策中枢）

### 功能描述

在现有样本推荐上增加“章节风险镜像”和“补件优先级队列”。

### 设计细则

1. 新增“下一章高危标签”：
   如灼烧压制、资源税、多段控制等。
2. 推荐从“给方向”升级到“给顺序”：
   明确 1/2/3 优先补件项。
3. 增加“误配告警”：
   当前构筑若与样本主场偏差过大，给出红色预警。

### 对可玩性的提升

减少盲目拿牌，提升中局纠错能力。

## 4.5 无尽轮回赛季 2.0（长期目标升级）

### 功能描述

新增赛季分段目标和赛季挑战链，保证中高轮次也有阶段目标。

### 设计细则

1. 赛季目标分为：
   基础（所有人可完成）、进阶（中层玩家）、极限（高层玩家）。
2. 赛季账本增加“崩盘原因统计”：
   让玩家理解失败主因。
3. 季签加入“可控风险指令”：
   让玩家主动选择更高压或更稳妥路线。

### 对可玩性的提升

把“刷轮次”升级为“冲阶段目标 + 对比打法质量”。

## 4.6 挑战 / 观察站 / 远征联动包

### 功能描述

挑战模式产出的优秀命盘可以进入观察站并生成远征加成线索。

### 设计细则

1. 挑战结算增加“命盘精选标签”。
2. 观察站新增“同主题对比视图”。
3. 远征入口读取精选命盘，提供额外选项（仅小幅度，不破坏平衡）。

### 对可玩性的提升

让“打挑战”不仅是成就，也能反哺主流程策略。

## 4.7 PVP 赛季化对抗 2.0

### 功能描述

在现有段位/经济基础上，引入赛季规则轮换与策略预览。

### 设计细则

1. 每赛季提供 1 条环境规则（如控制抗性上调、爆发衰减）。
2. 匹配前显示“当前赛季环境提示”。
3. 商店新增“赛季主题外观/称号任务”并与战绩挂钩。

### 对可玩性的提升

减少 PVP 固定套路，增加赛季迁移策略乐趣。

## 4.8 新流派与桥接组件（内容增量）

### 功能描述

新增 `2` 套主流派 + `1` 组跨流派桥接组件。

### 首发建议

1. **镜渊流**：围绕复制、映照、延迟触发。
2. **誓罚流**：围绕誓约惩罚反转收益。
3. **桥接组件**：用于把旧流派平滑转向新流派，降低成型断档。

### 对可玩性的提升

增加开局变化与中局转轴空间，减少“拿不到核心就废局”。

## 4.9 章节事件库重构（组合化生产）

### 功能描述

把事件拆为“触发条件 + 代价模板 + 奖励模板 + 后果模板”组合生成。

### 设计细则

1. 每章必须有独特主题事件簇。
2. 事件必须支持至少一种“短期赚 / 长期亏”或“短期亏 / 长期赚”抉择。
3. 事件后果写入命盘，允许后续章节追溯引用。

### 对可玩性的提升

事件不再是一次性奖励点，而是路线策略的一部分。

## 4.10 UI/UX 可读性升级（信息层）

### 功能描述

新增统一“局势总览条”和“章节风险卡”。

### 设计细则

1. 局势总览条固定展示：
   当前核心标签、风险等级、悬赏进度、势力倾向。
2. 章节风险卡展示：
   当前章高危机制、推荐防御策略、推荐资源预留。
3. 保持移动端可读：
   小屏必须保证两步内可看到关键风险信息。

### 对可玩性的提升

复杂系统不再隐藏在多层菜单里，决策效率更高。

## 4.11 难度重标定系统（DRI：Danger Rating Index）

### 功能描述

建立统一难度评定指标，给主线、无尽、挑战、PVP 提供统一语言。

### 指标维度（0-100）

1. 爆发威胁（Burst Threat）
2. 续航压力（Sustain Pressure）
3. 控制压制（Control Denial）
4. 资源税负（Resource Tax）
5. 纠错窗口（Recovery Window，反向分）

### 输出规则

1. 每章/每节点/每特殊战斗都有 DRI 标签。
2. UI 以“低/中/高/极高”四档展示。
3. 命盘记录“最高压力点”和“崩盘维度”。

### 对可玩性的提升

玩家能提前识别高危节点，挑战感保留但挫败感下降。

## 4.12 稳定性工程与防御性编程扩展

### 功能描述

继续把核心链路做成“可回归、可定位、可追溯”。

### 必做项

1. 关键系统统一空值防御与结构归一化。
2. 异步链路统一并发锁/超时/重试策略。
3. 新系统必须配套最小 Node sanity + 浏览器审计脚本。

## 5. 难度综合评定方案（版本级）

## 5.1 难度分层目标

1. **新手层**
   首次通关体验应强调可读性，失败原因可解释。
2. **熟练层**
   有明显中局优化空间，靠决策可提升胜率。
3. **挑战层**
   能通过承接高压悬赏、季签、追猎目标获得更高回报。

## 5.2 平衡目标（建议值）

1. 主线普通难度通关率：`45% - 60%`
2. 主线高难度通关率：`20% - 35%`
3. 无尽中层（季签中压）稳定推进率：`30% - 45%`
4. 高压目标完成率：`10% - 20%`

> 注：上线前必须基于自动化战斗回放与人工样本复验再做最终校准。

## 5.3 难度校验门禁

1. DRI 高于阈值的节点必须提供至少 1 条清晰应对信息。
2. 同章节内不得出现“连续三场无恢复窗口”的死锁组合。
3. 高压奖励必须与风险对等，避免“高风险低收益”。

## 6. 开发切片与交付节奏

## 6.1 切片原则

1. 每个切片都要“可运行 + 可测试 + 可回退”。
2. 先做框架，再堆内容，避免后期重构风暴。

## 6.2 建议切片

1. **Slice A：核心框架**
   章节悬赏 2.0 + 势力态度日志 + DRI 数据结构。
2. **Slice B：玩法联动**
   仇敌追猎 + 命盘扩展 + 观察站联动。
3. **Slice C：长线模式**
   无尽赛季 2.0 + 挑战精选命盘 + 远征加成线索。
4. **Slice D：PVP 升级**
   赛季规则轮换 + 匹配前策略提示 + 赛季商店任务。
5. **Slice E：可读性与收口**
   局势总览条 + 风险卡 + 平衡回调 + 全量回归。

## 7. QA 与“无阻塞缺陷”上线标准

## 7.1 测试矩阵

1. **语法层**
   新增/修改脚本必须通过 `node --check`。
2. **逻辑层**
   每个新模块至少新增 1 个 `sanity_*` 脚本。
3. **浏览器层**
   对应 UI 模块至少 1 个 `browser_*_audit` 断言。
4. **Playwright 冒烟**
   使用统一客户端脚本完成交互 + 截图 + 状态输出。
5. **人工抽检**
   每个核心流程至少抽检 1 张桌面图 + 1 张移动端图。

## 7.2 上线门禁（必须全部满足）

1. Node 门禁全通过。
2. 浏览器审计全通过（允许非阻断截图降级，但断言必须全绿）。
3. 冒烟截图中无关键 UI 缺失或重叠。
4. 控制台无新增高优先级错误。
5. 关键流程（新开局、章节推进、结算、存档、PVP）无阻塞性缺陷。

## 8. 版本验收标准（产品视角）

1. 玩家能在章节开始明确看到并承接中局目标。
2. 玩家的路线选择会对后续遭遇与结算产生可见后果。
3. 玩家能在局内看懂当前风险、在局外看懂失败原因。
4. 玩家在不同模式间切换时能感知明确收益（不是“为了换个界面玩”）。
5. 版本上线后，核心流程无 P0/P1 阻塞缺陷。

## 9. 风险与回滚策略

## 9.1 主要风险

1. 系统联动过多导致平衡波动。
2. 章节后果链路复杂导致边界条件遗漏。
3. UI 信息增量过大导致小屏拥挤。

## 9.2 控制策略

1. 所有新系统加 `feature flag`，支持按模块开关。
2. 高风险数值改动保留上一版参数快照。
3. 关键数据结构变更提供迁移与回退兼容逻辑。

## 9.3 回滚预案

1. 先回滚高风险模块（悬赏改写、仇敌复现）。
2. 保留已验证稳定的 UI 总览与日志链路。
3. 若赛季规则波动过大，回退到基础赛季模板并保留奖励记录。

## 10. 里程碑结果定义

`V7.0《命途裂变》` 成功的标志不是“加了很多内容”，而是：

1. 玩家能清楚回答“我这章在做什么目标”。
2. 玩家能清楚解释“我为什么赢/为什么输”。
3. 玩家愿意在主线、挑战、无尽、PVP 之间主动切换。
4. 版本在自动化与人工复验下保持稳定可迭代。
