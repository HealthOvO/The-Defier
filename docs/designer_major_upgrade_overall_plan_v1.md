# The Defier《逆命者》V10.0 整体方案 V1

## 0. 方案定位

本文是 V10 大版本的整体方案，用来保存当前产品、玩法、PVP、公平性、内容边界和验收方向。

本阶段不锁定具体修改文件、模块拆分和代码 owner。后续进入开发前，再根据当时仓库结构补实施计划。

配套文档：

- `docs/designer_major_upgrade_planning_v8.md`：完整策划推演。
- `docs/designer_major_upgrade_requirements_v7.md`：规则、接口、状态、测试合同。
- `docs/designer_major_upgrade_pvp_content_pack_v1.md`：首版合法牌池和基准斗法谱。
- `docs/designer_major_upgrade_pvp_ui_copy_samples_v1.md`：首版玩家文案样例。
- `docs/designer_major_upgrade_pvp_balance_fixtures_v1.md`：平衡仿真 fixture 合同。

文档优先级：

1. 若玩法、规则、公平性、上线门槛或证据阻断口径冲突，以本文为准。
2. 若首版合法牌池、基准斗法谱、身份槽或仿真输入冲突，以内容包为准，但不得突破本文禁用机制。
3. 若玩家可见文案冲突，以 UI 文案样例为素材，以本文合同为裁定边界。
4. 若测试 fixture 或报告字段冲突，以 fixture 合同为素材，以本文证据模板为准入边界。
5. 若 `requirements_v7` 中的具体文件路径、模块名或接口草案与开发时仓库结构不一致，进入开发前必须重新生成实施计划；不能把旧路径当成已冻结事实。

## 1. 大版本目标

V10 的核心目标是把《逆命者》从“单机成长 + 残影练习”推进到“真人可对抗、可复盘、可长期平衡”的版本。

玩家应该获得三种新体验：

- 真实对手：排位和约战必须是两名真实在线玩家，而不是客户端打残影后上报胜负。
- 公平对局：后手必须有行动权，不能出现先手一套秒杀、硬控整回合、不可读随机闪避决定胜负。
- 可学习失败：输局后能看到关键回合、最大爆发来源、预算拦截、公开 setup 和下一局调整建议。

本版本优先证明真人 PVP 的竞技骨架成立，不追求一次性继承全部 PVE 卡牌、全部命格、全部流派。

## 2. 核心产品结构

V10 的 PVP 页面分成六个体验区：

| 区域 | 定位 |
| --- | --- |
| 天命排位 | 真人在线排位，服务端裁定，写入正式积分和赛季记录 |
| 问道练习 | 残影或托管练习，不进入正式排位结算 |
| 好友约战 | 指定玩家对战，不影响正式积分 |
| 战报回放 | 查看关键回合、预算拦截、最大爆发、败因建议 |
| 天道商店 | 保留现有 PVP 经济和兑换，但收益来自服务端正式结算 |
| 防守残影 | 作为练习素材和展示，不再暗示它是真人排位对手 |

最重要的口径：排位只认真人 live match，练习只认练习。两条链路可以共享 UI 入口，但不能共享胜负结算。

## 3. 真人 PVP 对局体验

一局排位的玩家路径：

1. 选择斗法谱和一个 PVP 身份槽。
2. 进入匹配，系统只匹配真实在线玩家。
3. 双方进入准备房间，查看规则版本、本方构筑锁定状态、对手粗粒度公开画像和连接状态。
4. 调息换牌，每名玩家最多换 0-2 张，且每局一次。
5. 进入回合制战斗。
6. 服务端裁定每次行动，客户端只展示结果。
7. 对局结束后进入结算和复盘。

体验原则：

- 每次行动都要有即时反馈：pending、成功、拒绝原因。
- 对方手牌和牌库顺序不可见，但手牌数量、牌堆数量、公开状态必须可见。
- 高爆发必须先有公开 setup，例如破绽、易伤、蓄势或异常手牌数。
- 控制只能降低效率，不能让对手完整一回合无事可做。
- 断线和超时要给恢复窗口，但不能被用来拖局。

### 3.1 匹配公平与等待策略

真人 PVP 的底线不是“匹配到一个真人”就结束，而是双方都认为这一局有得打、有得学。

匹配请求必须包含以下非文件级字段：

| 字段 | 用途 | 玩家可见性 |
| --- | --- | --- |
| 规则版本 | 确保双方使用同一套 `pvp-live-v1` 真值 | 可见 |
| 赛季 ID | 避免跨赛季积分和规则混排 | 可见 |
| 构筑快照 hash | 确认入队后不能换谱、换身份或绕过禁限表 | 可见摘要 |
| 匹配评分 | 服务端用于衡量玩家当前竞技水平 | 不直接展示数字 |
| 段位区间 | 用于控制玩家对局预期 | 可见 |
| 样本量标签 | 区分新玩家、低样本、高置信玩家 | 不直接展示数字 |
| 连接健康 | 避免高延迟或频繁掉线玩家拖累对局 | 只展示连接状态 |
| 入队时间 | 控制等待扩圈和超时提示 | 可见 |

硬性匹配条件：

- 双方必须是真实在线玩家。
- 双方规则版本、赛季 ID 和排位合法池必须一致。
- 双方都没有正在进行的正式排位。
- 双方构筑快照已锁定且校验通过。
- 任一方连接健康不达标时不能进入正式排位；只能提示重试或进入练习。
- 匹配不到真人时不能自动塞残影、不能自动判胜、不能发正式奖励。

等待扩圈策略：

| 等待时间 | 匹配范围 | UI 口径 | 不允许 |
| --- | --- | --- | --- |
| 0-20 秒 | 同段位或相邻小段，匹配评分差小 | 正常匹配中 | 不跨规则版本 |
| 20-60 秒 | 扩到相邻段位，优先同样本量玩家 | 正在扩大搜索范围 | 不匹配高风险连接 |
| 60-120 秒 | 允许更大评分差，但必须记录跨度原因 | 可能匹配到经验不同的真人玩家 | 不隐藏跨度，不伪装成均势 |
| 120 秒后 | 提供继续等待、取消、进入练习三选一 | 真人较少，可继续等或先练习 | 不自动切残影，不写正式积分 |

首版匹配策略参数：

| 参数 | 首版合同 | 审计要求 |
| --- | --- | --- |
| 严格近分 | `0-99` | 标记为 `good / strict_rating / near_0_99`，优先选择候选池中评分差最小的真人 |
| 长等待扩圈 | `100-199` | 只有等待超过扩圈阈值才自动匹配，标记为 `expanded / expanded_100_199 / fair_100_199` |
| 宽跨度评分差 | `200-399` | 当前首版不自动匹配；未来若开放必须先有双方显式确认和结算解释 |
| 超宽跨度 | `>=400` | 首版不自动匹配；低在线池只能继续等待、取消或练习 |
| 近期重复匹配抑制 | 10 分钟内同一对手默认降优先级 | 低在线池低于 8 名候选时可放宽，但必须解释 |
| 连接健康通过 | 最近 60 秒 heartbeat 缺失不超过 1 次，重连次数为 0，RTT P95 不超过 1500ms | 只展示连接状态，不展示对手网络细节 |
| 连接健康高风险 | 最近 60 秒 heartbeat 缺失 2 次以上、RTT P95 超过 2500ms 或刚发生重连 | 不能进入正式排位，只能重试或练习 |
| 取消 / 未 ready 冷却 | 短时间多次触发时 30 秒、2 分钟、5 分钟递进 | 冷却不扣正式积分，不给对手胜场 |
| 宽跨度占比上限 | 一轮灰度中 `wide_but_accepted` 占比不超过 15% | 超过时先优化在线池、等待提示或入口分流 |

匹配质量必须写入审计摘要，至少包含：

- 双方规则版本和赛季 ID。
- 双方段位区间和匹配评分差。
- 双方样本量标签。
- 等待时间和扩圈阶段。
- 连接健康结果。
- 是否近期重复匹配。
- 是否触发新玩家保护。
- 是否使用入队时评分快照和最近候选选择。
- 最终匹配质量标签：`good`、`expanded`、`wide_but_accepted`、`rejected`。

Matchmaking snapshot 合同：

| 快照 | 生成时机 | 必须字段 | 用途 |
| --- | --- | --- | --- |
| 入队快照 | 玩家进入 `queueing` 时 | 规则版本、赛季 ID、构筑 hash、身份槽、匹配评分、段位、样本量标签、连接健康摘要、入队时间 | 证明玩家当时是否合法入队 |
| 配对快照 | 服务端选定双方时 | 双方评分差、扩圈阶段、候选池大小、近期重复匹配结果、连接健康分类、匹配质量标签、宽跨度原因 | 证明为什么匹到这个真人 |
| 玩家接受快照 | 宽跨度或异常提示被玩家确认时 | 提示类型、玩家选择、确认时间、是否继续等待过 | 证明玩家是否接受宽跨度匹配 |
| 结算解释快照 | 写正式结算前 | 终局类型、匹配质量标签、评分差、定级状态、积分公式版本、赛季积分变化原因 | 证明为什么这样改积分 |

`wide_but_accepted` 不是失败，但不能被当作常态。如果一轮测试中宽跨度匹配占比过高，优先调整等待提示、赛季入口和在线池策略，而不是降低战斗公平规则。

匹配质量分层门槛：

| 人群 / 时段 | 额外门槛 |
| --- | --- |
| 定级前 5 场 | `wide_but_accepted` 占比低于 8%，连续 2 场碾压败后必须推荐练习或继续等待，不暗改战斗数值 |
| 低段位低样本 | 等待 P95 可以更长，但不能牺牲同样本量保护；重复匹配同一强对手必须降优先级 |
| 高段位低在线 | 可以接受更长等待和更高重复匹配解释，但宽跨度确认必须显式展示 |
| 深夜 / 低在线池 | 允许进入继续等待 / 取消 / 问道练习三选一；不能自动塞残影 |
| 高频重赛对手 | 10 分钟内重复匹配需要审计原因；若同一对手连续碾压，不得继续默认匹配 |

匹配质量报告除了总量，还必须按段位带、样本量标签、在线池时段输出 `waitP95`、`wideMatchRatio`、`rematchRatio`、`stompRate` 和取消后冷却样本。总量通过但任一脆弱分层被打穿时，不能扩大灰度。

### 3.1.1 先手、座位与开局信息合同

先后手公平不能只靠护命预算，还必须能证明座位分配没有被客户端、重赛、约战或社交关系操控。

排位座位合同：

| 项 | 首版口径 | 审计字段 |
| --- | --- | --- |
| 分配时机 | `match_created` 时由服务端一次性分配 seat A / seat B，并写入配对快照 | `seatAssignmentSource`, `assignedAt`, `matchId`, `queueSnapshotHash` |
| 客户端权限 | 客户端不能请求先手、换座位或在看到对手信息后重抽座位 | `seatChangeRequested=false`, `seatLocked=true` |
| 长期公平 | 同一玩家在足够样本内的先手率必须接近 50%，首版警戒区间为 `45%-55%` | `firstSeatRateByPlayer`, `firstSeatRateByDivision`, `sampleSize` |
| 重复匹配 | 近期重复对手不允许通过取消重排来刷先手 | `recentOpponentPair`, `cancelAfterRevealCount`, `seatRepeatCount` |
| 争议复查 | 任何座位争议必须能从配对快照和事件日志复查 | `seatAuditTrail`, `ruleVersion`, `matchQualityTag` |

好友约战座位合同：

| 场景 | 口径 |
| --- | --- |
| 单局约战 | 默认服务端随机座位；双方可在房间内显式选择“交换座位”，但必须在锁定前双方确认 |
| 固定规则 Bo3 | 第 1 局随机，第 2 局交换座位，第 3 局优先让前两局后手次数较少的一方先手；若完全相同则服务端随机 |
| 同谱复现 | 复现关键回合时保留原对局座位，只作为练习，不写正式积分 |
| 争议处理 | 约战座位争议不影响排位积分；只影响约战记录和战报解释 |

开局前对手信息揭示矩阵：

| 阶段 | 排位可见 | 排位隐藏 | 离开 / 未 ready 处理 |
| --- | --- | --- | --- |
| `queueing` | 自己的规则版本、构筑合法性、等待时间、扩圈阶段 | 未匹配对手的一切身份和构筑信息 | 可取消，不冷却；高频取消可轻度限排 |
| `matched` | 已匹配到真人、规则版本一致、连接健康摘要、匹配质量标签 | 对手完整用户名、精确评分、完整斗法谱、卡组明细、身份槽细节 | 此时离开不让对手得胜，只记录未 join / cancel |
| `setup` | 对手席位、段位/匹配质量区间、粗粒度公开画像、连接状态、是否 ready、公开战斗计数 | 对手真实 userId、显示名、构筑 hash、构筑标签、身份槽、完整牌表、起手、换牌结果、隐藏身份细节 | 看见信息后取消、未 ready 或断线不写正式胜负；高频触发进入排队冷却 |
| `active` | 公开状态、手牌数量、牌库数量、公开 buff、公开 setup | 对手手牌、牌库顺序、随机种子、未公开意图 | 按正式断线、超时、投降规则处理 |
| `finished` | 脱敏战报、公开事件、自己当时可见信息、复盘标记 | 对手未公开手牌历史、牌库顺序、未公开 RNG | 可举报、分享脱敏战报或进入练习 |

对手信息揭示的底线：

- 排位不能在 `matched` 时暴露足以让玩家按对手构筑选择性 dodge 的信息。
- `setup` 阶段只给不能反推出构筑的公开画像，例如“流派待观察 / 公开战况观察中”和匹配质量区间；快攻、控制、护盾、连击等判断只能在 `active` 后由公开行动逐步形成，不能在开局前给出。
- 任何“看见信息后取消”的样本都必须进入匹配质量和风控审计；处罚只针对高频模式，不把单次弱网或误触当作恶意。
- 结算页可显示更完整的公开战报，但仍不能泄露对手当时隐藏手牌、牌库顺序或未公开随机种子。

### 3.2 新玩家与低样本保护

新玩家保护只能改善匹配和解释，不能暗中改变战斗规则。

首版保护规则：

- 前 5 场正式排位为定级样本，优先匹配其他低样本或相邻低段玩家。
- 定级样本可以让匹配评分变化更快，避免强玩家长期停留在低段。
- 定级样本的可见积分涨跌要更平滑，降低新玩家第一次尝试 PVP 的挫败。
- 保护不能增加生命、抽牌、灵力、伤害、起手质量或隐藏爆率。
- 保护不能把真人排位替换为残影。
- 保护不能让对手承担未知惩罚；若匹配跨度较大，结算公式必须公开处理。

低样本异常处理：

- 低样本玩家若连续快速获胜、频繁制造压倒性局面，应更快提升匹配评分，减少对低段玩家的压制。
- 低样本玩家若连续无有效行动线失败，应优先给练习入口、基准谱建议和复盘解释，不暗中匹配弱对手。
- 频繁取消已匹配准备、准备超时或断线的玩家，可以进入短时间排队冷却；冷却不能扣正式积分，除非已进入对局终局判负。

新玩家保护的成功标准不是让新玩家必胜，而是让前几局都能看懂局面、完成有效行动、知道下一局该调哪里。

### 3.3 对局手感、等待体验与低干扰社交

真人 PVP 的公平规则只是底线。双方还必须觉得这一局有手感、有节奏、有对手存在感，但不能让社交、动画或等待干扰裁定。

对局内手感目标：

| 场景 | 玩家应该感受到 | 必须有的反馈 | 禁止事项 |
| --- | --- | --- | --- |
| 我方提交行动 | 行动已经送达，等待服务端裁定 | 按钮进入 pending，显示本次行动摘要和可取消范围 | 客户端先播胜负、先扣血、先抽牌或自行补算 |
| 行动被接受 | 我的选择改变了局面 | 资源、手牌、生命、公开状态和事件日志同步变化 | 只播放动画但不更新权威状态版本 |
| 行动被拒绝 | 我知道为什么不能这么做 | 稳定拒绝码、短文案、可改正入口 | 只显示“失败”或让玩家猜是网络、规则还是资源不足 |
| 对方回合等待 | 对手正在思考，局面仍可读 | 对方思考计时、连接状态、公开 setup、我方可查看信息 | 空白等待、隐藏计时、用假动画制造对方行动 |
| 长动画与连锁展示 | 看得懂发生了什么，但不拖节奏 | 关键事件可跳过动画，复盘保留完整事件 | 动画阻塞出牌、跳过动画跳过规则解释 |
| 弱网或重连 | 我知道当前状态是否可信 | 同步中、已恢复、需要刷新、行动待确认四类状态 | 让本地旧状态覆盖服务端状态 |
| 移动端操作 | 关键按钮可点，手牌不误触 | 出牌确认、目标高亮、撤销未提交选择 | 小屏误触直接提交不可逆关键行动 |

首版计时原则：

- 准备阶段使用独立 ready 计时；进入对局后使用行动计时。
- 行动计时必须展示给双方；不能隐藏剩余时间。
- 首版每个行动窗口默认给足思考时间，但连续超时会缩短后续宽限或进入托管判负流程。
- 进入最后 10 秒必须有明显提示，但提示不能遮挡手牌、目标和结束回合按钮。
- 客户端卡顿、切后台和网络抖动不能直接等同恶意拖局；只有服务端记录的连续超时、重复重连和异常行动序列才进入风控。
- 托管动作只能做低影响保底选择，不能打出决定性高爆发，也不能替玩家提交社交表情或投降。

首版计时 SLA：

| 计时项 | 首版值 | UI 要求 | 争议字段 |
| --- | --- | --- | --- |
| 准备 ready | 45 秒 | 显示双方 ready 状态和剩余时间 | `readyDeadlineAt`, `readyTimeoutCount` |
| 调息换牌 | 包含在 ready 计时内 | 换牌确认后不可再次调息 | `mulliganUsed`, `readyConfirmedAt` |
| 行动窗口 | 75 秒 | 双方可见当前行动方和倒计时 | `turnDeadlineAt`, `actionTimeoutCount` |
| 最后提示 | 剩余 10 秒 | 强提示但不遮挡手牌、目标和结束回合 | `lastWarningShownAt` |
| heartbeat | 服务端下发 `heartbeatIntervalMs`；当前默认 5 秒，最小 1 秒 | 弱网时显示同步状态，不伪装成对手思考 | `lastHeartbeatAt`, `missedHeartbeatCount` |
| 断线判定 | 服务端 `heartbeatStaleMs`；当前默认 15 秒未收到 heartbeat | 标记重连中 | `disconnectedAt` |
| 当前行动方 grace | 服务端 `reconnectGraceMs`；当前默认 30 秒 | 暂停行动计时并展示 grace 倒计时 | `graceStartedAt`, `graceDeadlineAt` |
| 非行动方断线 | 不冻结当前行动方计时 | 对方重连中提示 | `nonActorDisconnectedAt` |
| settling 等待 | P95 目标 3 秒，硬上限 15 秒 | 显示正在结算，禁止继续提交行动 | `settlingStartedAt`, `settlementDeadlineAt` |

当前 S4A 最小实现已把 active 行动窗口锚定到独立 `turnTiming`：合法出牌只更新 match 最近变更时间，不重置当前回合 `startedAt/deadlineAt`；只有开战、结束回合换手和 timeout 托管换手会启动下一行动席位的新倒计时。旧持久化行或坏数据缺失 `turnTiming` 时也不能回退到 `match.updatedAt` 推导倒计时，只能回退到 battle start / match createdAt / 已超时锚点。后续若接入 WebSocket / 多实例队列，推送层必须沿用该锚点。

当前 S4B 最小实现已让浏览器端 heartbeat timer 消费 `connectionReport.heartbeatIntervalMs`，不再在 PVP scene 中固定 `5000ms`。HTTP fallback 阶段仍会立即发送一次 heartbeat，但后续周期跟随服务端连接报告；后续 WebSocket 推送必须继续以该连接报告作为客户端节奏和弱网提示的单一输入。

若超过 settling 硬上限，客户端不能自行判胜负；必须进入 `settlement_pending`、`match_invalidated` 或重试读取结算的明确状态。

低干扰社交只做三类：

| 功能 | 首版允许 | 首版不做 |
| --- | --- | --- |
| 预设表情 | 开局问候、称赞、思考中、精彩、再来一局；可静音 | 自由文本聊天、挑衅文案、连续刷屏 |
| 结算互动 | 再战邀请、加好友入口、分享战报、举报异常 | 结算页强制弹社交、失败方无法关闭 |
| 战报传播 | 分享公开脱敏战报和关键回合摘要 | 分享隐藏手牌、随机种子、未脱敏 payload、实时观战窥牌 |

社交安全底线：

- 表情只影响表达，不影响行动计时、匹配评分、赛季积分或奖励。
- 每名玩家必须能静音对手表情；静音不通知对手，避免二次冲突。
- 同一玩家连续表情必须限频；限频不影响出牌。
- 举报、拉黑、静音不能成为逃避失败结算的手段。
- 拉黑只影响后续社交和优先避免重复匹配，不保证永远不匹配；若在线池过小，系统必须解释匹配原因。
- 首版不开放实时观战。只允许对局结束后的脱敏战报分享，避免外部语音或旁观者泄露隐藏信息。
- 再战默认进入好友约战或练习约战，不直接写正式积分；若未来支持正式再战，必须重新经过匹配和快照锁定。

## 4. 首版战斗规则

首版以 `pvp-live-v1` 作为规则基线。下表是当前设计基线，进入开发前必须通过资源节奏冻结条件；若反脚本化证据失败，必须先调整抽牌、卡组或首发内容包，再生成具体实施计划。

| 项 | 首版值 |
| --- | --- |
| 生命 | 50 |
| 卡组 | 20 张 |
| 单卡复制 | 最多 2 张 |
| 起手 | 5 张 |
| 每回合抽牌 | 5 张 |
| 初始灵力 | 3 |
| 手牌上限 | 10 |
| 调息换牌 | 每人 0-2 张，每局一次 |
| 先手首回合伤害预算 | 18 |
| 后手首回合伤害预算 | 22 |
| 第二回合伤害预算 | 28 |
| 后手护印 | 6 点不可转伤护盾 |
| 平均目标回合 | 6-9 整轮 |
| 长局收束 | 第 14 整轮强制判定 |

反非游戏防线：

- 后手在首次真实行动窗口前不能死亡。
- 首两整轮禁止完整回合剥夺。
- 首回合不提供额外抽牌和额外灵力。
- 0 费牌不进入首版排位合法池。
- 复制、造牌、额外回合、硬控、随机闪避、无限债账、局外资源伤害不进入首版排位。
- 单回合超过 30 点玩家伤害，上一行动窗口必须已有公开 setup。

### 4.1 单一规则真值

后续所有实现、仿真、UI、复盘都只能读取同一套 `pvp-live-v1` 规则真值。若其他文档、旧实现、旧 PVP 逻辑与下表冲突，以下表为准。

| 规则项 | 首版唯一口径 | 冲突处理 |
| --- | --- | --- |
| 排位对手 | 只能是真实在线玩家 | 匹配不到真人时保持等待、取消或进入练习入口，不能自动切残影 |
| 胜负来源 | 服务端结算 | 客户端胜负上报只能用于练习或本地展示，不能改正式积分 |
| 0 费牌 | 首版排位合法池为 0 张 | 任何 0 费牌进入排位构筑都拒绝 |
| 首回合额外抽牌 | 0 | 身份槽、buff、卡牌效果都不能突破 |
| 首回合额外灵力 | 0 | 身份槽、buff、卡牌效果都不能突破 |
| 后手首次行动前死亡 | 0 次允许 | 伤害预算和护印必须截断致死线 |
| 完整回合剥夺 | 首版禁用 | stun、extraTurn、skipTurn、硬冻结全部拒绝或预转写后再入池 |
| 高爆发 | 第 3 回合后才允许，且上一行动窗口必须有公开 setup | 无 setup 的大爆发被预算拦截并写入复盘 |
| 长局 | 第 14 整轮强制判定 | 不允许通过治疗、留盾、反伤形成无限拖局 |
| 练习模式 | 不影响正式积分 | 可保留旧残影，但文案和结算必须与排位分离 |

裁定顺序必须固定：

| 顺序 | 裁定内容 | 说明 |
| --- | --- | --- |
| 1 | 验证行动窗口 | 不是当前行动方、对局未激活、客户端版本过旧时直接拒绝 |
| 2 | 验证资源和目标 | 检查手牌、灵力、目标、费用、一次性限制 |
| 3 | 套用 PVP 内容映射 | 原样合法、固定转写、禁用、仅测试、未来候选五类先裁定 |
| 4 | 结算基础效果 | 护盾、治疗、抽牌、灵力、状态、伤害按固定顺序进入待结算结果 |
| 5 | 套用首回合资源闸 | 第 1 回合额外抽牌和额外灵力均为 0，任何来源都不能突破 |
| 6 | 套用公开 setup 与单效果上限 | 破绽、易伤、穿透、处决、低血等都必须进入同一上限链 |
| 7 | 套用回合伤害预算 | 预算截断最终玩家生命伤害，并记录拦截事件 |
| 8 | 检查死亡和终局 | 若未终局，推进事件日志和状态版本；若终局，进入结算 |
| 9 | 生成座位视图 | 只向各自玩家发送脱敏后的可见状态 |

若上述顺序与支撑资料冲突，以本表为准。

### 4.1.1 资源节奏与反脚本化冻结条件

当前 `20` 张卡组、起手 `5`、每回合抽 `5` 的基线强调可读性，但存在把前 3-5 轮打成固定脚本的风险。进入开发前必须先用内容包和仿真证明它不会过度确定；否则不能把该资源节奏写入实施计划。

反脚本化指标：

| 指标 | 阻断阈值 | 说明 |
| --- | --- | --- |
| `earlyPublicStateRepeatRate` | 同一 archetype 前 3 整轮公共局面重复率 `>0.18` 阻断 | 防止玩家每局都看到同一开局路线 |
| `mirrorBranchEntropyP50` | 同构筑镜像前 5 回合分支熵 `<1.4` 阻断 | 防止镜像局只有一条显然最优路线 |
| `mirrorBranchEntropyP10` | `<0.8` 阻断 | 防止低分位样本极度脚本化 |
| `coreDeckExposureByRound5` | 任一 archetype 第 5 整轮前核心牌曝光率 `>0.85` 阻断 | 防止 4-5 轮几乎看完整副牌 |
| `highRepeatRouteRate` | 任一 archetype 高重复路线占比 `>0.12` 阻断 | 与 fixture 中开局重复簇联动 |
| `functionSlotDiversity` | 同一 archetype 的可替换功能位少于 2 组阻断 | 防止只有模板微调，没有构筑探索 |

若上述任一指标失败，调参顺序为：

1. 先调整首发内容包的功能位和同 archetype 替换池。
2. 再调整每回合抽牌数、卡组张数或调息范围，优先评估 `20/5/4`、`24/5/4`、`24/4/4` 三个候选。
3. 再调整长局收束和节奏目标。
4. 最后才调整伤害预算和后手保护；调整后必须重跑双方体验公平、隐藏信息、平衡和娱乐性审计。

### 4.2 回合预算解释

伤害预算不是“削弱所有伤害”，而是防止非游戏击杀的保险丝。

- 预算只在玩家生命伤害结算时生效。
- 护盾、治疗、抽牌、弱化仍正常结算。
- 被预算拦截的伤害要生成公开事件，用于复盘解释。
- 预算不能被穿透、处决、易伤、破绽、低血增伤绕过。
- 预算不能触发吸血、反伤、击杀奖励或其他二次收益。

预算命中后的玩家感受应是：“这次爆发被规则保护拦住，但对方已经建立了压力”，而不是“我的牌被系统没收了”。

护命窗口唯一时序：

| 窗口 | 覆盖对象 | 预算值 | 结束条件 |
| --- | --- | --- | --- |
| 先手首行动窗口 | 先手第 1 次 `action` 阶段造成的玩家生命伤害 | 18 | 先手结束第 1 次行动 |
| 后手首行动窗口 | 后手第 1 次 `action` 阶段造成的玩家生命伤害 | 22 | 后手结束第 1 次行动 |
| 第二行动窗口 | 双方各自第 2 次 `action` 阶段造成的玩家生命伤害 | 28 | 该座位结束第 2 次行动 |
| 后手首次行动前保护 | 后手完成第 1 次真实行动前，先手不能造成致死结果 | 与先手首行动预算和后手护印共同生效 | 后手第 1 次行动被 accepted 或主动结束回合 |

“首两整轮保护”只表示双方前两次行动都受预算和完整回合剥夺禁令约束，不表示前两整轮所有伤害都被无条件免死。第 3 次行动开始，预算不再按首两行动窗口自动生效，但仍受单效果上限、公开 setup、长局公式和合法池限制。

### 4.3 回合结构与行动窗口

为了让仿真、服务端裁定、浏览器表现和复盘都按同一节奏运行，首版必须冻结回合结构。

术语定义：

| 术语 | 含义 |
| --- | --- |
| 整轮 | 先手完成一个回合，后手完成一个回合，合称 1 个整轮 |
| 回合 | 一个座位从回合开始到回合结束的完整行动周期 |
| 行动窗口 | 当前行动方可以出牌、结束回合、投降或恢复 pending 结果的时间段 |
| 真实行动窗口 | 玩家本人在线且可操作的行动窗口；托管窗口不算首次真实行动 |
| 公开 setup | 上一行动窗口已经写入公开事件、双方都能看到的破绽、易伤、蓄势、异常手牌数或其他爆发前兆 |

每个回合固定按以下阶段结算：

| 顺序 | 阶段 | 服务端处理 | 玩家体验 |
| --- | --- | --- | --- |
| 1 | 回合开始 | 记录 `turn_started`，推进回合号和当前行动方 | UI 高亮当前行动方 |
| 2 | 起始状态结算 | 结算公开的持续伤害、治疗、护盾衰减和倒计时；所有效果必须先写事件再改状态 | 玩家能看到上一回合遗留效果如何生效 |
| 3 | 抽牌 | 从权威牌库按顺序抽至本回合抽牌数；若手牌达到 10 张，剩余抽牌跳过并记录 `draw_capped` | 玩家知道自己因手牌上限少抽了几张，不触发弃牌或额外收益 |
| 4 | 灵力重置 | 将本回合可用灵力设置为规则值；首回合额外灵力始终为 0 | UI 显示本回合可用资源 |
| 5 | 行动窗口 | 接受出牌、结束回合、投降、重连补发；每个 action 必须带幂等 ID | 玩家看到 pending、成功或稳定拒绝原因 |
| 6 | 回合结束 | 结算回合末公开状态、移除到期状态、记录 `turn_ended` | UI 明确进入对方回合 |
| 7 | 终局检查 | 击杀、投降、超时、断线判负、长局判定都在统一终局检查中进入 `settling` | 双方收到同一终局原因 |

首版不提供响应链、打断栈和对手回合即时出牌。这样做不是降低深度，而是先保证真人 PVP 的节奏、公平、移动端可读性和回放确定性。反制主要通过公开 setup、弱化、净化、护盾、治疗、预算拦截和下一行动窗口决策完成。

行动窗口底线：

- 每个玩家自己的行动窗口至少允许一次合法非投降选择，除非该玩家已投降、超时判负或对局已终局。
- 出牌后如果触发预算拦截，行动仍算被接受；只截断玩家生命伤害，不回滚整张牌。
- 非当前行动方提交出牌一律拒绝，不进入 pending。
- 重复 action ID 只能返回首次结果，不能重复抽牌、扣灵力、造成伤害或发奖励。
- 回合结束是显式行动；玩家超时后的托管结束回合必须写入公开事件。
- 手牌上限只限制继续抽牌，不触发随机弃牌、自动弃牌或隐藏弃牌收益。

后手公平检查按“后手第一个真实行动窗口”计算，而不是按后手第一次收到状态消息或托管动作计算。

### 4.4 长局强制判定

第 14 整轮结束后，如果双方都未死亡，必须进入长局强制判定。它的目标是收束拖局，不是惩罚防守流派。

长局分数只使用公开、可复盘、不可隐藏刷取的指标：

| 指标 | 计分 | 上限 | 说明 |
| --- | --- | --- | --- |
| 剩余生命差 | `clamp(本方生命 - 对方生命, -20, 20)` | 20 | 体现压血成果，但不能让单一治疗/自伤决定全部 |
| 有效伤害 | `floor(对敌方生命造成的未被预算截断伤害 / 3)` | 18 | 只计算真实生命伤害，不计算被预算拦截部分 |
| 有效防守 | `floor(阻止或恢复的生命伤害 / 4)` | 12 | 护盾、治疗、净化都可计入，但上限低于伤害 |
| 公开 setup 转化 | 每次公开 setup 在下一己方行动窗口转化为伤害、治疗、弱化或护盾 +2 | 10 | 鼓励可读计划，不奖励隐藏爆发 |
| 资源效率 | 每个整轮结束时未浪费灵力且至少产生一次有效行动 +1 | 8 | 鼓励行动取舍，不鼓励空过拖局 |
| 预算拦截惩罚 | 本方无公开 setup 的伤害被预算拦截每次 -2 | -10 | 惩罚不可读爆发尝试 |
| 托管惩罚 | 本方托管行动每次 -2 | -8 | 防止断线拖局成为策略 |

判定流程：

1. 第 14 整轮后先结算当轮所有公开状态和终局检查。
2. 若双方仍存活，计算双方长局分数。
3. 分差大于等于 5 时，分数高者以 `round14_score` 获胜。
4. 分差小于 5 时，进入 `round14_draw`，不发高额奖励，不大幅改变段位。
5. 复盘必须展示各项分数来源，让双方知道为什么判胜、判负或判平。

长局判定禁止项：

- 不计算隐藏手牌质量、牌库未来抽牌、随机种子或未公开计划。
- 不允许治疗、护盾或反伤无限刷分；所有防守分都有上限。
- 不允许通过空过攒资源获得高分；资源效率必须绑定有效行动。
- 不允许把预算拦截的虚高伤害计入有效伤害。
- 不允许用长局判定绕过断线、超时、投降或服务端无效局规则。

如果仿真显示第 14 整轮频繁进入平局，优先调整内容池收束能力和基准谱结构，再考虑改长局公式。

## 5. 首版内容边界

首版只开放 5 个内容流派和 3 套系统基准。

| 类型 | 目标 |
| --- | --- |
| 流血压制 | 连续小伤害和公开流血，制造压力但不秒杀 |
| 破绽节奏 | 先 setup 后 payoff，让爆发有可读前兆 |
| 雷策轻连击 | 中速连击爽点，受预算和公开状态约束 |
| 治疗消耗 | 对抗快攻，提供慢速恢复和消耗体验 |
| 护盾反击 | 稳定防守并反打，不能无限拖局 |
| 快攻压迫 | 系统压力基准，验证先后手公平 |
| 过牌中速 | 系统资源基准，验证抽牌上限 |
| 控制弱化 | 系统控制基准，验证不剥夺行动权 |

暂不进入首版排位：

- 回响复制链。
- 誓债清算链。
- 造牌 / 随机生成。
- 弃牌换手主轴。
- 状态牌塞入手牌或牌库。
- 跨回合留盾牌。
- 硬控和额外回合。

这些机制可以保留为后续赛季候选，但必须先经过服务端规则、UI 可读性、复盘解释和平衡仿真验证。

### 5.1 内容映射准则

每张牌、每个身份槽、每个状态在进入排位前，都必须被归入一个明确类别。

| 类别 | 含义 | 首版处理 |
| --- | --- | --- |
| `legal` | 可直接按 PVP 规则结算 | 进入排位合法池 |
| `translated` | PVE 原效果不直接进入，但有固定 PVP 转写 | 只按转写后的公开效果进入 |
| `disabled` | 会造成非游戏体验、隐藏爆发或协议复杂度 | 拒绝进入排位 |
| `test_only` | 可用于引擎单测，但不开放给玩家排位 | 不进入排位构筑 |
| `future_candidate` | 后续赛季候选，需要额外 UI/复盘/仿真证明 | 不进入首版 |

映射原则：

- 不允许“运行时临场猜测”：入队前必须已经知道这张牌属于哪一类。
- 不允许静默降级：如果 PVE 效果被转写，玩家文案和复盘必须显示 PVP 版语义。
- 不允许隐藏生成：排位中不能凭随机、复制、造牌得到未在入队快照里的牌。
- 不允许资源绕行：费用降低、回灵、抽牌不能绕过首回合和 0 费限制。
- 不允许不可读胜负：随机闪避、直接换血、瞬时胜利、额外回合不进入首版。

每个内容条目进入排位前必须完成归一化记录，至少包含：

| 字段 | 用途 |
| --- | --- |
| 内容 ID | 卡牌、身份槽或状态的稳定标识 |
| 原始效果摘要 | 来自 PVE 或旧实现的原语义 |
| PVP 类别 | `legal` / `translated` / `disabled` / `test_only` / `future_candidate` |
| PVP 结算语义 | 首版实际执行的公开规则；禁用项写拒绝原因 |
| 公开解释标签 | UI 和复盘展示给双方的短标签 |
| 隐藏信息边界 | 哪些字段只在服务端存在，不能进入对方视图 |
| 快照参与字段 | 哪些字段进入入队快照和构筑校验 hash |
| 仿真标签 | 用于区分伤害、防御、过牌、控制、治疗、setup、payoff |

没有归一化记录的内容，不允许进入排位。

### 5.2 首版 Meta 生态目标

首版要形成一个小而稳定的生态，而不是把全部 PVE 流派搬进排位。

| 对局关系 | 设计目标 |
| --- | --- |
| 快攻 vs 护盾 | 快攻能压血线，护盾能争取反击窗口 |
| 快攻 vs 治疗 | 快攻有前期优势，但不能在恢复体系启动前秒杀 |
| 破绽 vs 控制 | 破绽有中段爆发，控制能通过弱化和净化延缓 |
| 连击 vs 中速 | 连击爆发更高，中速稳定性更好 |
| 治疗 vs 护盾 | 双方都能拖，但第 14 整轮必须收束 |
| 低血反打 vs 快攻 | 低血有戏剧性，但前两回合不能靠自伤即时反杀 |

首版不追求每套构筑都有 50% 胜率，但不允许出现一个构筑压过整个生态。

## 6. 服务端权威原则

V10 排位的胜负必须由服务端决定。

服务端拥有：

- 随机数种子和抽牌顺序。
- 双方完整手牌、牌库、弃牌堆。
- 行动合法性。
- 伤害、护盾、状态、死亡判定。
- 事件日志、回放、结算。
- 段位、经济、历史、赛季记录。

客户端只负责：

- 展示按座位脱敏后的状态视图。
- 提交带幂等 ID 的玩家意图。
- 显示 pending、成功、reject、重连和复盘。

禁止回退：

- 排位不能使用客户端上报 `didWin`。
- 排位不能在匹配失败时自动切残影。
- 排位不能信任客户端重新提交的斗法谱。
- 排位不能把隐藏手牌、牌库顺序或随机数状态发给浏览器。

### 6.1 抽象协议状态机

整体方案只冻结状态语义，不锁具体实现文件。

| 状态 | 玩家可见体验 | 允许行为 | 退出条件 |
| --- | --- | --- | --- |
| `idle` | 未排队 | 选择斗法谱、进入排队 | 提交有效排队请求 |
| `queueing` | 匹配中 | 取消排队、查看预计等待 | 匹配到真人或取消 |
| `matched` | 匹配成功 | 建立连接、进入准备 | 双方进入准备房间 |
| `setup` | 准备 / 调息 | 调息换牌、确认准备、重连 | 双方准备完成或准备超时 |
| `active` | 正在对战 | 出牌、结束回合、投降、重连 | 击杀、投降、超时判负、长局判定 |
| `settling` | 正在结算 | 等待结果，不可再提交行动 | 结算成功或判定无效局 |
| `finished` | 结果和复盘 | 查看复盘、再来一局 | 返回入口 |
| `invalidated` | 无效局 | 查看原因和补偿说明 | 返回入口 |

协议不变量：

- 任意时刻只能有一个权威状态。
- 同一玩家同一场对局的同一行动 ID 只能结算一次。
- 服务端已接受的行动，重连后必须能重放或补发结果。
- 结算只能写一次；重复请求只能返回同一结算结果。
- 客户端本地 pending 状态不能覆盖服务端状态。
- HTTP 和实时连接同时存在时，以服务端版本号更高的状态为准。

状态映射合同：

| 玩家态 | 持久态 | 协议 / 事件态 | 是否可重连 | 是否允许结算 | 说明 |
| --- | --- | --- | --- | --- | --- |
| 未排队 | `idle` | 无活跃 match | 否 | 否 | 玩家可改斗法谱和身份槽 |
| 匹配中 | `queueing` | queue heartbeat / cancel | 否 | 否 | 匹配失败只能继续等待、取消或进练习 |
| 匹配成功 | `matched` | `match_created` / `match_found` | 是 | 否 | 已锁快照，但尚未进入准备完成 |
| 准备调息 | `setup` | `snapshot_locked` / `mulligan_submitted` / `ready_submitted` | 是 | 仅允许无效局 | 未开局前不写正式积分 |
| 正在对战 | `active` | `battle_started` 后的 turn / action 事件 | 是 | 是 | 只有服务端事件日志可推进状态 |
| 正在结算 | `settling` | `match_finished` / `settlement_written` | 是 | 幂等写一次 | 不再接受出牌、调息或投降 |
| 结果复盘 | `finished` | replay / settlement read | 是 | 只读 | 重复请求只返回同一结果 |
| 无效局 | `invalidated` | `match_invalidated` / optional compensation | 是 | 否 | 可给非强度补偿，但不改正式积分 |

`created`、`ready`、`turn.phase`、`presence` 这类实现态只能作为上述持久态的内部细分，不能成为玩家、测试和结算各自解释的第二套真值。

### 6.2 终局结果矩阵

所有终局都必须用同一套结果语义表达。

| 终局类型 | 最终状态 | winner | 是否写排位 | 是否有回放 | 客户端结果 |
| --- | --- | --- | --- | --- | --- |
| `lethal` | `finished` | 胜者座位 | 是 | 是 | 胜负结算 |
| `surrender` | `finished` | 未投降方 | 是 | 是 | 认负结算 |
| `timeout_forfeit` | `finished` | 未超时方 | 是 | 是 | 超时判负 |
| `disconnect_forfeit` | `finished` | 未断线方 | 是 | 是 | 断线判负 |
| `round14_score` | `finished` | 分数高者 | 是 | 是 | 长局判定胜负 |
| `round14_draw` | `finished` | `draw` | 当前首版不写正式匹配评分、赛季积分、奖励或正式历史；仅保留本局公开复盘 | 是 | 长局平局 |
| `double_lethal_draw` | `finished` | 空 | 写历史和回放；匹配评分按平局样本；赛季积分和主要奖励为 0 | 是 | 同次结算双方同时死亡 |
| `ready_timeout` | `invalidated` | 空 | 否 | 可选 | 准备超时，无效局 |
| `server_invalidated` | `invalidated` | 空 | 否 | 可选 | 服务异常，无效局 |

终局不变量：

- `winner` 允许为空，但只允许出现在 `double_lethal_draw` 或 `invalidated` 类结果；当前 `round14_draw` 使用 `winner=draw` 且不强造 loser。
- `round14_draw` 是已完成对局，不是无效局；当前首版只进入公开复盘，不写正式匹配评分、正式历史、赛季积分、主要奖励或任务进度。
- `finished` 可以有正式结算；`invalidated` 不能改变正式积分。
- 每个终局只能写一次结算记录。
- 客户端收到重复终局消息时必须展示同一结果，不能二次播报奖励。
- 回放事件统一使用 `match_finished`，不再使用旧文档中的 `battle_finished` 命名。
- 伤害预算命中统一写公开事件 `budget_clamped`，不作为 action reject；旧版预算 reject code 不能进入新 wire contract。

复合终局优先级：

| 冲突场景 | 裁定口径 | Golden 覆盖 |
| --- | --- | --- |
| 同一 effect queue 内双方同时生命归零 | `double_lethal_draw` | `double_lethal_same_queue` |
| 当前行动致死与对手反伤同时发生 | 若双方同一结算队列死亡，`double_lethal_draw`；若只有对手死亡，`lethal` | `thorns_double_lethal` |
| 行动超时与已提交合法致死 intent 竞争 | 以服务端接受 intent 的时间和事件日志顺序为准；已 accepted 的致死先结算，否则 `timeout_forfeit` | `timeout_vs_accepted_lethal` |
| 断线发生在 `match_finished` 之后 | 不改变已写终局，只继续 `settling` 或读取结果 | `disconnect_after_match_finished` |
| `match_finished` 后结算写入失败但状态可信 | 保持 `settling`，重试幂等结算；不允许客户端改结果 | `settlement_retry_after_finished` |
| `match_finished` 后发现状态不可信或隐藏信息泄露 | 进入 `server_invalidated`，不改正式积分，写争议证据包 | `settling_invalidated_untrusted_state` |
| 投降与致死同 tick | 以服务端先接受的 intent 为准；同一 revision 内投降优先于后续未处理行动 | `surrender_vs_pending_lethal` |

开局前结果矩阵：

| 分支 | 最终状态 | 是否写排位 | 是否进历史 | 是否冷却 | 是否补偿 | 玩家口径 |
| --- | --- | --- | --- | --- | --- | --- |
| 匹配后任一方未 join | `invalidated` | 否 | 只写轻量异常记录 | 可按频率冷却未 join 方 | 否或少量非强度补偿 | 对手未进入准备，本局未开始 |
| 准备阶段一方取消 | `invalidated` | 否 | 只写轻量异常记录 | 高频取消方冷却 | 否 | 对方取消，本局未开始 |
| 准备阶段一方未 ready | `invalidated` | 否 | 只写轻量异常记录 | 高频 timeout 方冷却 | 可给等待方少量非强度补偿 | 准备超时，本局不计分 |
| 准备阶段一方投降 | `invalidated` | 否 | 只写轻量异常记录 | 可按取消处理 | 否 | 对局尚未开始，不作为胜负 |
| 准备阶段断线后未恢复 | `invalidated` | 否 | 只写轻量异常记录 | 高频断线方冷却 | 可给等待方少量非强度补偿 | 对手连接中断，本局不计分 |
| 规则版本或快照校验失败 | `invalidated` | 否 | 写审计记录 | 视为系统或构筑问题，不默认冷却 | 否 | 规则或斗法谱已更新，请重新确认 |

正式胜负只从 `active` 后的终局产生。`setup` 前后的异常不能给另一方刷胜场、刷任务或刷正式奖励。

### 6.3 断线与托管底线

断线处理目标是“保护正常玩家”，不是替玩家自动打最优解。

- 短断线：保留行动窗口，允许重连恢复。
- 首次超时：优先执行低风险防御或结束回合。
- 多次超时：判负或无效局，防止拖局。
- 托管不能主动打出处决、低血反杀、高爆发或决定性连击。
- 托管动作必须写入复盘，让双方知道不是玩家主动选择。

断线和超时不能成为第三种胜负策略。

统一计数模型：

| 计数 | 增加条件 | 清零条件 | 达到上限 |
| --- | --- | --- | --- |
| `disconnectGraceUsed` | 当前行动方断线并进入 grace | 对局结束 | 超过 2 次后不再冻结计时，按托管或判负处理 |
| `turnTimeoutCount` | 行动窗口自然超时 | 对局结束 | 第 3 次超时判负 |
| `readyTimeoutCount` | 准备阶段未 ready | 离开当前匹配 | 高频触发进入短排队冷却 |
| `automationActionCount` | 服务端执行托管动作 | 对局结束 | 进入复盘风险标签，不能打决定性爆发 |

关键阶段定义：

- `setup`：只允许无效局、重新匹配或练习分流，不允许写正式胜负。
- `active` 当前行动方：断线进入 grace，grace 内恢复行动权；grace 外先托管保底，再按次数判负。
- `active` 非当前行动方：不冻结当前行动方计时；断线方重连后只能拉取权威 StateView。
- `settling`：不恢复行动权，只允许读取结算和复盘。

托管白名单：

- 允许：结束回合、最低风险防御、保留资源、可公开解释的净化。
- 禁止：处决、低血反杀、高爆发、复杂连击、投降、社交表情、改变斗法谱或任何局外选择。

### 6.4 行动意图与拒绝码合同

首版不先锁定具体接口文件，但所有通道提交的玩家动作必须被归一化为同一类行动意图。HTTP、实时连接、重连补发和测试夹具都应使用同一套语义。

行动意图最小字段：

| 字段 | 含义 | 约束 |
| --- | --- | --- |
| `intentId` | 玩家本次意图的幂等 ID | 同一场对局、同一玩家内唯一；重复提交返回首次结果 |
| `matchId` | 对局 ID | 只能指向当前玩家参与的对局 |
| `seatId` | 玩家座位 | 必须与服务端绑定的玩家身份一致 |
| `intentType` | 动作类型 | 只能来自首版允许列表 |
| `ruleVersion` | 客户端所知规则版本 | 与服务端不一致时拒绝或要求刷新 |
| `stateVersion` | 客户端看到的状态版本 | 低于服务端时返回权威状态和可恢复提示 |
| `payload` | 动作参数 | 只包含动作必要字段，不能携带权威结果 |
| `submittedAt` | 客户端提交时间 | 只用于延迟观察，不能作为裁定时间源 |

首版允许的 `intentType`：

| 类型 | 可用状态 | 必要 payload | 成功结果 |
| --- | --- | --- | --- |
| `queue_join` | `idle` | 构筑快照引用、身份槽、规则版本 | 进入 `queueing` |
| `queue_cancel` | `queueing` | 无 | 返回 `idle` |
| `mulligan` | `setup` | 要换的手牌 ID，最多 2 张 | 更新本方手牌视图，记录一次性调息已用 |
| `ready` | `setup` | 无 | 标记本方准备完成 |
| `play_card` | `active` | 手牌 ID、目标、可选模式 | 按规则裁定，推进事件日志和状态版本 |
| `end_turn` | `active` | 无 | 结束当前行动方回合 |
| `surrender` | `setup` / `active` | 无 | 进入投降或无效局判定 |
| `reconnect_sync` | 任意非终局状态 | 最近收到的状态版本 | 返回权威状态、已接受行动结果和可继续动作 |
| `replay_request` | `finished` | 回放 ID、视角 | 返回脱敏回放 |

行动结果统一为四类：

| 结果 | 含义 | 客户端表现 |
| --- | --- | --- |
| `accepted` | 意图已被服务端裁定并改变或确认状态 | 清 pending，播放事件，更新状态版本 |
| `rejected` | 意图非法，未改变权威状态 | 清 pending，显示稳定拒绝原因 |
| `duplicate` | 该 `intentId` 已处理过 | 返回首次结果，不重复播放奖励 |
| `sync_required` | 客户端状态过旧或规则版本不一致 | 拉取权威状态，提示刷新当前局面 |

稳定拒绝码：

| 拒绝码 | 触发条件 | 玩家口径 |
| --- | --- | --- |
| `not_in_match` | 玩家不属于该对局 | 当前对局已失效 |
| `match_not_active` | 对局状态不允许该动作 | 现在还不能这么做 |
| `not_your_turn` | 非当前行动方提交行动 | 还没轮到你 |
| `stale_state` | 客户端状态版本落后 | 局面已更新，正在同步 |
| `rule_version_mismatch` | 规则版本不一致 | 规则版本已更新，请刷新局面 |
| `snapshot_locked` | 入队后尝试改谱或换身份 | 排位快照已锁定 |
| `invalid_card` | 手牌 ID 不存在或不属于玩家 | 这张牌当前不可用 |
| `card_disabled` | 内容映射为禁用或仅测试 | 该内容不进入首版排位 |
| `insufficient_energy` | 灵力不足 | 灵力不足 |
| `invalid_target` | 目标不存在、隐藏或不合法 | 目标不可选 |
| `mulligan_used` | 重复调息或超出换牌数量 | 本局调息次数已用完 |
| `duplicate_action_conflict` | 同一意图 ID 携带不同动作内容 | 行动编号冲突，请同步局面 |
| `loadout_illegal` | 入队构筑、身份槽或合法池快照未通过校验 | 斗法谱未通过排位校验 |
| `state_view_forbidden_field` | 座位视图或回放投影包含禁止字段 | 状态同步异常，已停止公开 |
| `action_timeout` | 行动窗口已超时 | 已超时，系统进入托管或判定 |
| `terminal_already_settled` | 对局已终局或结算已写入 | 对局已经结束 |

拒绝码不等于玩家文案。拒绝码必须稳定、可测试、可复盘；玩家文案可以本地化和润色，但不能改变拒绝语义。

行动意图底线：

- 客户端不能在 `payload` 中携带抽牌结果、伤害结果、胜负、奖励或对手隐藏信息。
- 服务端拒绝意图时不能局部扣资源、局部改血量或局部写回放。
- `sync_required` 只能要求玩家刷新权威状态，不能让客户端自行补算缺失事件。
- `duplicate` 返回的事件序列必须与首次 `accepted` 完全一致。
- 同一 `intentId` 如果携带不同 `payload`，必须返回 `duplicate_action_conflict`，不能覆盖首次意图，也不能按新意图重算。
- 任何正式结算相关意图都不能复用练习模式的本地胜负上报。

### 6.5 事件日志与回放可见性合同

真人 PVP 的复盘、重连、争议排查、平衡仿真都必须来自同一条权威事件日志。客户端动画、toast 和本地缓存只能消费事件，不能成为事实来源。

事件日志原则：

- 服务端事件日志是 append-only；已经写入的事件不能被静默改写。
- 每个事件必须有连续序号，缺号、重号或乱序都视为协议错误。
- 回放必须由初始快照和事件日志重建，不能由客户端录屏或客户端本地状态拼接。
- 事件日志可以在服务端保存完整信息，但发给玩家、回放、分享和审计时必须按视角脱敏。
- 随机结果可以记录，随机种子和未来牌库顺序不能进入任何玩家视图。

当前 S5B 最小实现已把 `pvp_live_match_events` 落成 append-only SQLite 事件表：服务端保存 match 时幂等补写事件流，replay API 优先从事件表恢复公开时间线，旧局缺少事件表记录时才回退 `state.events`。终局 replay 只允许使用连续 sequence 覆盖且包含 `match_finished` / `match_invalidated` 的完整事件源；如果事件表和 `state.events` 都不完整，API 必须拒绝生成回放，而不是返回截断时间线。这一步只解决回放 / 争议 / missed-event 的事实源基础，不等同于多实例共享队列、正式赛季入口或生产 smoke 已完成。

当前 S6A 最小实现已把单进程 WebSocket 推送接到上述事件真源：`join_match` 会先返回 seat-scoped `state_sync`，再按 `lastSeenRevision` 补发公开 `events_replay`；`heartbeat` 返回 `presence`，`intent` 返回 `intent_result` 并触发双方最新 StateView 广播。它仍是本机 / 单进程权威同步闭环，不等同于跨进程房间、共享队列、灰度监控或线上部署完成。

回放和争议留存周期：

| 数据层 | 首版保留期 | 赛季后处理 | 可见性 |
| --- | --- | --- | --- |
| `replay_self` | 至少保留 2 个完整赛季 | 超期可降级为关键回合摘要 | 仅本人可见 |
| `replay_public` | 分享链接默认 30 天，可由玩家撤销 | 超期失效或只保留公开高光摘要 | 公开脱敏 |
| `audit_safe` | 至少保留 2 个完整赛季加申诉窗口 | 超期只保留结论、风险标签和 hash | 玩家可见摘要和客服审计 |
| `server_full` | 按合规和运营审计需要保留，不进入玩家视图 | 可归档冷存；读取需审计 | 服务端和授权审计 |
| `drillScenario` | 来源 replay 有效期内可用；收藏 drill 至少 30 天 | 超期后只保留练习完成记录 | 本人或双方同意分享 |

任何回放降级都不能改变已完成结算；若争议仍在窗口期，相关 `audit_safe` 和必要 `server_full` 数据不能提前清理。

事件最小包络：

| 字段 | 含义 | 可见性 |
| --- | --- | --- |
| `eventId` | 单事件稳定 ID | 全视角可见 |
| `matchId` | 所属对局 | 全视角可见 |
| `sequence` | 对局内连续序号 | 全视角可见 |
| `eventType` | 事件类型 | 全视角可见 |
| `turnIndex` | 当前座位回合序号 | 全视角可见 |
| `roundIndex` | 当前整轮序号 | 全视角可见 |
| `actingSeat` | 行动或效果来源座位 | 全视角可见；无来源时为空 |
| `stateVersionBefore` | 事件前状态版本 | 协议审计可见；普通 UI 可隐藏 |
| `stateVersionAfter` | 事件后状态版本 | 协议审计可见；普通 UI 可隐藏 |
| `publicSummary` | 双方都能理解的一句话摘要 | 全视角可见 |
| `payload` | 事件参数 | 按可见性规则裁剪 |
| `visibility` | 事件默认可见层级 | 全视角可见 |
| `redactionTags` | 被裁剪字段类别 | 审计和测试可见 |
| `stateHashAfter` | 事件后状态摘要 | 服务端和测试可见；玩家不可见 |

首版事件类型：

| 事件类型 | 用途 | 可见性要求 |
| --- | --- | --- |
| `match_created` | 对局创建、座位、规则版本 | 不暴露对方隐藏构筑细节 |
| `snapshot_locked` | 入队构筑、身份槽、内容 hash 锁定 | 本方可见自己的锁定摘要；对手只见锁定已完成和粗粒度公开画像，不能看到构筑标签、身份槽或 hash；完整牌表仅本方和服务端可见 |
| `mulligan_submitted` | 调息换牌完成 | 对方只见换牌数量，不见换入换出卡 |
| `ready_submitted` | 玩家准备完成 | 双方可见 |
| `turn_started` | 回合开始 | 双方可见 |
| `status_ticked` | 持续状态结算 | 公开状态双方可见；隐藏状态首版排位不允许 |
| `cards_drawn` | 抽牌 | 本方见卡牌 ID；对方只见数量 |
| `draw_capped` | 手牌上限导致少抽 | 双方可见数量，不泄露未抽牌 |
| `energy_reset` | 灵力重置 | 双方可见当前灵力 |
| `card_played` | 出牌和目标 | 双方可见已打出的牌、目标和公开模式 |
| `intent_rejected` | 合法性拒绝 | 本方可见拒绝码；对方只在影响节奏时见公开摘要 |
| `damage_applied` | 生命伤害 | 双方可见实际伤害 |
| `shield_changed` | 护盾变化 | 双方可见公开护盾 |
| `heal_applied` | 治疗 | 双方可见实际治疗 |
| `status_applied` | 状态添加 | 双方可见公开状态、来源和持续时间 |
| `status_expired` | 状态到期 | 双方可见 |
| `setup_revealed` | 公开 setup 建立 | 双方可见来源、标签和可反制窗口 |
| `budget_clamped` | 伤害预算截断 | 双方可见原始伤害、实际伤害和截断原因 |
| `turn_ended` | 回合结束 | 双方可见 |
| `timeout_warning` | 行动窗口即将超时 | 当前行动方完整可见；对手只见对方连接状态 |
| `automation_action` | 托管动作 | 双方可见，并标记非玩家主动选择 |
| `match_finished` | 终局 | 双方可见终局类型、胜者和平局原因 |
| `settlement_written` | 正式结算写入 | 双方可见结果摘要；经济细节按玩家各自视角展示 |

可见性层级：

| 层级 | 允许内容 | 禁止内容 |
| --- | --- | --- |
| `server_full` | 完整手牌、牌库顺序、弃牌、随机状态、双方完整 payload | 不能直接发给浏览器 |
| `seat_self_live` | 本方手牌、本方可操作信息、双方公开状态 | 对方手牌、对方牌库顺序、随机种子 |
| `seat_opponent_live` | 对方公开行动、手牌数量、牌堆数量、公开状态 | 对方隐藏手牌、未来抽牌、未公开构筑细节 |
| `replay_self` | 本方当时可见信息、自己已做选择、公开事件和复盘标记 | 对方当时隐藏手牌、牌库顺序、随机种子 |
| `replay_public` | 可分享的公开时间线、公开 setup、伤害、预算、终局 | 任一方隐藏手牌、未公开选择、服务器内部 hash |
| `audit_safe` | 脱敏后的字段清单、可见性标签、泄露扫描结果 | 真实随机种子和完整隐藏状态 |

回放受众矩阵：

| 受众 | 默认层级 | 可解锁条件 | 必须证明 |
| --- | --- | --- | --- |
| 本人复盘 | `replay_self` | 对局结束且本人参与 | 只包含本人当时可见信息、公开事件、复盘标记和建议 |
| 对手本人 | `replay_self` | 对局结束且对手本人参与 | 与本方对称，但不会互相暴露隐藏手牌历史 |
| 分享链接 | `replay_public` | 玩家主动分享或复制战报 | 只能从公开事件、公开 setup、预算拦截、终局摘要推导 |
| 争议摘要 | `audit_safe` | 玩家提交异常反馈后生成 | 展示风险标签、字段路径和结论摘要，不泄露隐藏原文 |
| 内部审计 | `server_full` | 受控审计、漏洞排查或风控复核 | 不得直接进入浏览器 payload、分享链接或普通截图 |

公开战报推导合同：

| 输出 | 允许引用 | 禁止引用 | 必须携带 |
| --- | --- | --- | --- |
| 分享标题 | 终局类型、段位区间、公开流派标签 | 双方隐藏手牌、未来抽牌、未公开意图 | `publicEventIds` |
| 高光回合 | 公开 setup、实际伤害、预算拦截、长局分数来源 | 对手当时可选但未公开的手牌答案 | `turnIndex` / `roundIndex` |
| 失败建议公开摘要 | 公开事件、预算拦截、终局摘要、玩家自己当时可见信息 | 对手隐藏手牌质量、牌库顺序、RNG seed、未公开行动意图 | `derivationTags` |
| 调谱建议 | 本方斗法谱曲线、公开败因标签、已公开的对局节奏 | 对手未公开卡牌和内部评分 | `sourceVisibility` |
| 争议摘要 | 脱敏风险标签、字段路径、结论类型 | 原始设备、账号关联、隐藏状态原文 | `redactionTags` |

任何分享给第三方的建议都必须能通过 `replay_public` 或 `audit_safe` 重建推导链。若建议需要 `server_full` 才能解释，该建议只能出现在内部审计，不能出现在公开战报。

复盘标记：

| 标记 | 触发条件 | 玩家价值 |
| --- | --- | --- |
| `first_pressure` | 一方首次建立可持续血线、setup 或资源优势 | 告诉玩家局势从哪里开始倾斜 |
| `largest_burst` | 本局最大单行动窗口生命伤害 | 解释最大爆发来源 |
| `budget_saved` | 预算截断避免死亡或过量伤害 | 解释为什么没被秒或为什么伤害被拦 |
| `missed_answer_window` | 玩家有可支付净化、防御、治疗或弱化但未使用 | 给出可学习建议，不羞辱玩家 |
| `no_effective_action_risk` | 行动窗口接近无有效选择 | 作为公平风险证据进入仿真和复盘 |
| `long_game_swing` | 第 10 整轮后出现关键分差变化 | 解释长局判定前的转折 |
| `automation_impact` | 托管动作改变资源、血线或节奏 | 让双方知道结果受托管影响 |

回放输出底线：

- 默认回放只给 `replay_self`；分享链接只能使用 `replay_public`。
- 复盘建议只能引用玩家当时可见或战后公开的信息，不能用对手隐藏手牌来教育玩家。
- 当前 S5A 最小实现已提供参与者赛后 `GET /api/pvp/live/matches/:matchId/replay`：支持 `replay_self`、`replay_public`、`audit_safe`，以 `matchRef` 和 `replayHash` 替代 raw `matchId` 进入公开回放，并用 `hiddenScan` 阻断手牌、牌库、实例 id、原始 payload 和 RNG 泄露。S5B 已继续补齐独立 `pvp_live_match_events` 事件表长期真源，回放可在 `state_json.events` 损坏或被清空时从事件表恢复公开时间线。
- 若事件缺失、乱序或脱敏失败，该局回放不能作为封板证据。
- 任何隐藏信息审计失败都必须阻断真人排位对外开放，而不是只隐藏回放入口。
- 回放标记可以帮助学习，但不能改变结算、段位或奖励。

### 6.6 反作弊、滥用防护与争议处理合同

真人 PVP 的公平性不只来自规则，还来自玩家相信结果可信。首版必须能处理篡改、脚本、刷分、恶意断线和争议申诉，但不能用单个弱信号误伤正常玩家。

风险分类：

| 风险 | 判定依据 | 首版处理 |
| --- | --- | --- |
| 客户端篡改 | 客户端提交抽牌、伤害、胜负、奖励、隐藏信息或旧状态覆盖权威状态 | 拒绝意图，记录 `integrity_flagged`，不改正式积分 |
| 重复提交与重放 | 同一 action ID 反复提交、旧 stateVersion 重放、已结算终局重复写入 | 返回首次结果或 `sync_required`，不重复结算 |
| 自动化刷局 | 极短周期固定操作、长期相同出牌间隔、异常高频排队和取消 | 降低匹配优先级或进入排队冷却，不直接扣分 |
| 恶意拖局 | 反复 ready timeout、行动 timeout、断线重连后继续超时 | 按断线与托管合同处理，可进入冷却或判负 |
| 合谋刷分 | 固定账号高频互刷、异常投降链、宽跨度重复匹配、胜负循环互换 | 保留对局但暂停奖励或进入人工复核队列 |
| 规则漏洞利用 | 明显绕过禁限表、费用、预算、长局判定或结算幂等 | 该局进入 `server_invalidated` 或暂停正式结算 |
| 争议申诉 | 玩家认为对局异常、隐藏信息泄露、结算错误或对手利用漏洞 | 生成证据包，给出可复查结论 |

反作弊原则：

- 服务端权威优先于任何客户端输入；客户端提交的结果只能作为意图，不是事实。
- 不用单一弱信号直接封禁、扣分或没收奖励；硬处罚必须有可复查证据链。
- 连接差、移动端误触、短时后台切换不能自动等同作弊。
- 处罚不能帮助另一方刷分；可疑局优先暂停奖励、无效局或等待复核。
- 所有争议结论必须能追溯到规则版本、内容版本、事件日志、状态 hash 和结算记录。

争议证据包必须包含：

| 字段 | 用途 | 可见性 |
| --- | --- | --- |
| `matchId`、`seasonId`、`ruleVersion` | 锁定被申诉对局 | 双方可见 |
| 双方 seat、段位区间、匹配质量标签 | 判断是否匹配异常 | 脱敏可见 |
| 构筑快照 hash、合法池 hash | 判断是否绕过禁限表 | 摘要可见 |
| action 意图序列、拒绝码、状态版本 | 判断是否重放、重复提交或旧状态覆盖 | 审计可见 |
| 事件日志序列和 `stateHashAfter` 链 | 判断是否缺事件、乱序或篡改 | 审计可见 |
| RNG 结果摘要 | 判断随机结果是否来自服务端记录 | 审计可见；不暴露未来种子 |
| 终局原因、结算写入记录、奖励写入记录 | 判断是否重复结算或奖励污染 | 双方可见摘要 |
| 风险标签和处置状态 | 告诉玩家当前是正常、待复核、无效局或已处理 | 双方可见 |

申诉结论只允许以下类型：

| 结论 | 含义 | 结果处理 |
| --- | --- | --- |
| `normal_result` | 对局按规则正常完成 | 保持结算，给出关键规则解释 |
| `invalidated_bug` | 规则、协议、隐藏信息或结算出现系统异常 | 该局无效，不改正式积分；必要时补偿非强度奖励 |
| `abuse_confirmed` | 有足够证据证明篡改、刷分、恶意拖局或漏洞利用 | 撤销异常奖励，触发冷却、停排或更高等级处罚 |
| `insufficient_evidence` | 有疑点但证据不足 | 保持结算或暂停奖励复查，不能直接惩罚 |
| `duplicate_report` | 同一问题已被处理 | 返回已有结论和证据摘要 |

争议工单状态机：

| 状态 | 进入条件 | 玩家可见口径 | 积分 / 奖励处理 |
| --- | --- | --- | --- |
| `reported` | 玩家从结算页、战报页或复盘页提交异常 | 已收到反馈，系统会按事件日志复核 | 不立即改分，不重复发奖励 |
| `under_review` | 风控或系统规则命中需要复核 | 正在复核规则版本、事件日志和结算记录 | 高风险样本可暂挂异常奖励 |
| `reward_hold` | 存在重复结算、隐藏信息、刷分或漏洞利用疑点 | 奖励暂缓发放，胜负结论待复核 | 暂挂奖励或榜单展示，不静默扣分 |
| `resolved` | 产出五类申诉结论之一 | 展示结论、关键依据和是否影响积分 / 奖励 | 按结论保持、无效、撤销异常奖励或补偿非强度奖励 |
| `duplicate_closed` | 同一问题已有处理结果 | 返回已有结论和证据摘要 | 不产生新的奖励或积分变动 |

争议处理底线：

- 玩家可以从结算页和战报页提交异常反馈；反馈入口不能遮挡复盘和再战路径。
- 争议处理不应打断正常玩家的下一局，除非触发隐藏信息泄露、重复结算或服务端状态不可信。
- 已完成的正式积分不得静默改写；若需要撤销或补偿，必须留下可见说明和审计记录。
- 涉及隐私、设备、网络或账号关联的字段只能进入内部风险标签，不能出现在玩家可见回放里。
- 若争议样本暴露规则真值冲突，必须回到 V10-S0 修正规则快照，再重跑受影响切片。

## 7. 复盘与成长

复盘要把失败变成可学习内容，而不是只显示胜负。

每局至少产出：

- 胜负和终局原因。
- 最大单回合伤害。
- 最大公开 setup 来源。
- 伤害预算拦截次数。
- 首次优势建立回合。
- 长局判定结果和公开分数来源。
- 失败方下一局建议。

建议类型：

- 斗法谱曲线偏高：增加 1 费防御或过牌。
- 防爆不足：增加净化、虚弱或护盾。
- 自伤过早：降低低血牌密度。
- 收束不足：加入稳定中段伤害。
- 过牌过多：降低纯资源牌比例。

### 7.1 有效行动权定义

“后手没有被秒”不等于体验公平。后手首次行动必须是有效行动窗口。

后手首次行动同时满足以下条件，才算有效：

1. 生命大于 0，且本回合结束前不是必死状态。
2. 玩家本人仍在控制窗口内；托管行动不能算作玩家首次有效行动。
3. 手牌中至少存在一张当前可支付的牌，且该牌能产生正向局面变化。
4. 允许结束回合计为有效的唯一情况：结束回合会触发明确的防御、治疗、抽牌、持续伤害或倒计时收益，且不会立刻死亡。
5. 至少存在一种非投降选择能改变局面：防御、治疗、净化、弱化、过牌、建立 setup、造成伤害之一。
6. 没有被硬控、费用税、弃牌、冻结等效果压到“只能空过”。
7. UI 明确显示可以做什么，以及上一回合哪些公开事件造成当前压力。

无效行动窗口示例：

- 后手没死，但手牌全因费用税不可用。
- 后手能出牌，但所有牌都会被已锁定的延迟伤害无视。
- 后手只有结束回合，且结束后必死。
- 后手被托管自动打掉关键资源，玩家重连后已无法挽回。
- 后手能打出一张牌，但这张牌对生命、防御、手牌、setup、控制、伤害均无收益。

这些情况在仿真和浏览器验收中都应计入失败。

### 7.2 可玩性核心循环

V10 不能只靠“有真人 PVP”成立。玩家愿意持续打下去，需要每局都形成清楚的循环：

1. 开局前做选择：选择斗法谱、身份槽和预期对局节奏。
2. 对局中读信息：观察对方公开 setup、手牌数量、资源状态和血线压力。
3. 行动中有取舍：进攻、防守、过牌、净化、蓄势、保留资源都要有明确机会成本。
4. 终局后有解释：知道自己输在曲线、解牌、setup、资源或长局收束哪一项。
5. 下一局能调整：更换 2-4 张牌、切换基准谱、进入练习或直接再排位。

首版 PVP 的娱乐性来自“我看懂了、我有反应、我差一点、我能调整”，而不是来自隐藏随机、数值碾压或一回合秒杀。

### 7.3 日常、周常与赛季节奏

不同在线时长的玩家都应该有可完成目标。

| 场景 | 目标体验 | 设计边界 |
| --- | --- | --- |
| 10 分钟 | 打一局排位或练习，拿到清楚复盘 | 不能要求长时间刷日常才具备排位资格 |
| 20-30 分钟 | 打 2-3 局，按复盘调整一次斗法谱 | 调整成本要低，不能让玩家每局都重配整套构筑 |
| 当日回访 | 完成一个公开天命课题，例如使用某类 setup 获胜或完成一次长局反打 | 课题不能强迫玩家使用弱构筑掉分 |
| 每周目标 | 追求段位、流派熟练度、回放高光或好友约战记录 | 周目标给表达和荣誉，不给排位数值优势 |
| 赛季目标 | 完成定级、冲段、收集称号、沉淀代表流派 | 赛季重置不能抹掉玩家已学会的策略经验 |

赛季可以轮换推荐流派、复盘主题和外观奖励，但首版规则真值不能频繁漂移。规则变化必须可公告、可复盘、可回滚。

赛季循环 schema：

| 字段 | 含义 | 边界 |
| --- | --- | --- |
| `seasonId` | 当前赛季 | 只影响赛季积分、荣誉和展示 |
| `cycleType` | `daily_topic` / `weekly_goal` / `season_goal` | 不得要求必须打正式排位才能继续游戏 |
| `source` | 复盘主题、推荐练习、好友约战、排位表现或赛季公告 | 不能引用对手隐藏手牌、牌库顺序或 RNG |
| `recommendedMode` | 推荐进入真人排位、问道练习、好友约战或复盘 | 练习和约战不得写正式积分 |
| `completionCondition` | 完成条件 | 禁止强迫玩家用弱构筑掉分；失败参与奖励有上限 |
| `rewardType` | 荣誉、徽章、展示、称号、非强度货币 | 不能影响生命、伤害、抽牌、灵力、起手或匹配评分 |
| `dailyCap` / `weeklyCap` | 领取上限 | 防止刷失败奖励或互刷约战 |
| `dismissState` | 本日关闭、本周关闭、本赛季不再提示 | 关闭不影响排位资格 |
| `auditTags` | 奖励边界、模式隔离、复盘来源和跳过路径 | 进入赛季循环审计 |

### 7.4 奖励与成长边界

PVP 奖励必须服务长期参与，但不能破坏公平。

允许的奖励：

- 段位、赛季称号、头像框、回放徽章、流派熟练度。
- 天道商店中的外观、展示、称号、战报装饰和非排位强度物品。
- 练习素材、基准谱解锁、复盘筛选能力和好友约战展示。
- 失败局的小额参与奖励，但必须有每日或赛季上限。

禁止的奖励：

- 直接提高排位生命、伤害、抽牌、灵力或起手质量。
- 必须通过商店兑换才拥有的排位核心卡。
- 连败后暗中提高爆率、伤害或匹配弱对手的隐藏补偿。
- 用付费、肝度或局外养成绕过 `pvp-live-v1` 规则预算。

反刷场奖励合同：

| 奖励来源 | 首版口径 | 禁止 |
| --- | --- | --- |
| 赛季主奖励 | 按段位峰值、周度最佳表现、学习里程碑和代表回放发放 | 按总对局数线性堆奖励 |
| 非强度货币 | 有日 / 周上限和边际递减；用于外观、展示、战报装饰 | 通过刷失败、刷约战或刷无效局大量获得 |
| 失败参与奖励 | 只作为留存缓冲，每日上限清晰，可被关闭提示 | 多输多拿成为最优策略 |
| 好友约战 / 练习 | 可给学习进度、教学徽记、战报收藏 | 回流正式积分、正式赛季任务或排位核心资源 |
| `round14_draw` | 当前首版只给本局公开复盘和学习提示；匹配评分、赛季积分、主要奖励、非强度货币和正式历史均为 0 / 不写 | 被当成低风险刷任务路径 |

模式、终局与计数真值表：

| 模式 / 终局 | 匹配评分 | 赛季积分 | 主要奖励 | 非强度货币 | 日周任务 | 学习进度 | 历史 / 战报 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 排位 `lethal / surrender / timeout_forfeit / disconnect_forfeit / round14_score` | 按正式公式写入 | 按正式公式写入 | 按胜负和上限写入 | 有上限 | 只计正式排位类任务 | 可计 | 写正式历史和战报 |
| 排位 `round14_draw` | 不变 | 0 | 0 | 0 | 不计胜负类任务 | 只给本局公开复盘提示 | 不写正式历史；本局可读复盘保留在终局视图 |
| 排位 `double_lethal_draw` | 按平局样本写入 | 0 | 0 | 可给极小学习补偿且有上限 | 不计胜负类任务 | 可计 | 写正式历史和战报，标记平局 |
| 排位 `ready_timeout / server_invalidated / 开局前无效分支` | 不变 | 0 | 0 | 只允许少量非强度补偿 | 不计任务 | 可给异常提示，不计里程碑 | 只写异常摘要或争议证据 |
| 问道练习胜负 | 不变 | 0 | 0 | 可给学习上限内货币 | 只计练习类任务 | 可计 | 写练习记录，可收藏 |
| 好友约战胜负 | 不变 | 0 | 0 | 默认 0；教学徽记可有上限 | 不计正式日周任务 | 可计约战课题 | 写约战记录和双方同意的脱敏战报 |
| 关键回合复现 / drill | 不变 | 0 | 0 | 默认 0；可给首次完成徽记 | 只计学习类任务，且每日上限 | 可计 | 写练习完成记录，不写正式战绩 |
| 公开战报浏览 / 分享 | 不变 | 0 | 0 | 0 | 不计 | 不计，除非进入实际 drill | 只写分享审计和脱敏 payload hash |

反刷补充规则：

- 同一对手、同一 replay、同一 drillScenario 在日 / 周任务中只能按上限计一次，重复完成只保留学习记录。
- 所有奖励 ledger 必须包含 `mode`、`endReason`、`ruleVersion`、`seasonId`、`opponentPairHash`、`sourceReplayId` 或 `drillScenarioId` 中适用字段，保证幂等和复查。
- 好友约战、练习和 drill 的收益上限必须低于正式排位的表达性收益，且不能成为刷非强度货币的最优路径。
- 任何模式不得把失败参与奖励设计成“多输多拿”的最优策略。

赛季奖励快照与争议口径：

| 场景 | 首版裁定 |
| --- | --- |
| 赛季主奖励快照 | 默认按赛季内最高段位峰值和周度最佳表现发放，不按最后一天在线状态或总场次线性堆叠 |
| 快照时点 | 赛季结束时冻结榜单和奖励候选；冻结后只允许争议处理和补发 / 撤销异常奖励 |
| 申诉窗口 | 赛季结束后保留固定申诉窗口；窗口期内高风险奖励可进入 `reward_hold` |
| `under_review` 对局 | 暂不用于发放争议相关奖励；匹配评分和赛季积分按已写 ledger 展示风险标记 |
| 奖励释放 | 复核为正常时补发，不得要求玩家重新登录或重新完成任务 |
| 奖励撤销 | 只撤销异常奖励和异常榜单展示，不静默改写正常对局历史 |
| 玩家公告 | 补发、撤销、暂挂都必须有玩家可见摘要和证据类型，不泄露隐藏字段 |
| 榜单冻结 | 冻结榜单可显示“复核中”标记；不能把未复核异常样本当作最终荣誉 |

奖励审计必须能回答：本周玩家最高收益来自表现、学习和表达，还是来自单纯刷场次。若刷场收益更高，奖励配置不能封板。

成长感来自“理解和表达能力变强”，而不是“账号数值压过对手”。

### 7.5 练习、约战与复盘的再战动机

排位之外的模式要承担降低挫败和促进学习的功能。

| 模式 | 主要价值 | 不能做的事 |
| --- | --- | --- |
| 问道练习 | 验证改谱、练习反快攻、理解 setup 和预算拦截 | 不能写正式积分，不能伪装成真人排位 |
| 好友约战 | 低压力切磋、测试自定义约定、制造社交话题 | 不能绕过合法池生成正式战绩 |
| 战报回放 | 学习败因、分享高光、发现构筑缺口 | 不能泄露隐藏手牌、牌库顺序或随机种子 |
| 基准谱试炼 | 让新玩家理解五大流派和三套系统基准 | 不能变成必须刷完才能公平排位的门槛 |

复盘页的下一步应该是行动入口，而不是死板总结：再来一局、换 2 张牌、用基准谱练一次、约好友复现这一局，都应成为自然路径。

练习和约战的正向乐趣结构：

| 模式 | 乐趣结构 | 验收口径 |
| --- | --- | --- |
| 问道练习 | 按败因生成短课题：公开 setup 防守题、长局收束题、低血反打题、曲线修正题 | 课题可在 3-6 分钟内完成，完成后给一条可带回排位的非强度建议 |
| 好友约战 | 固定规则 Bo3、换边再战、同谱复现、赛后快速换 2-4 张测试 | 约战全程不写正式积分，但能保留双方同意分享的脱敏战报 |
| 关键回合复现 | 从战报提取公开状态和本方当时可见信息，生成一次练习场景 | 不能包含对手隐藏手牌、牌库顺序或 RNG seed |
| 教学约战 | 一方用基准谱演示反制窗口，另一方练操作或调谱 | 可给教学徽记和收藏，不给正式赛季收益 |

`drillScenario` 合同：

| 字段 | 含义 | 约束 |
| --- | --- | --- |
| `drillScenarioId` | 练习场景 ID | 由 replay 和公开事件派生，不能暴露隐藏信息 |
| `sourceReplayId` | 来源战报 | 只能来自本人 `replay_self` 或公开 `replay_public` 可推导内容 |
| `sourceEventIds` | 来源关键事件 | 指向 setup、预算拦截、最大伤害、长局分差或关键错过窗口 |
| `visibleStateSnapshot` | 玩家当时可见局面 | 只包含本方当时可见信息和公开状态 |
| `opponentPolicy` | 对手练习策略 | 只能使用公开可推导策略或基准 bot，不还原隐藏手牌答案 |
| `allowedPlayerChanges` | 玩家可调整范围 | 默认允许 0-4 张替换卡或选择基准谱，不允许改规则真值 |
| `objective` | 练习目标 | 防住 setup、压低爆发、完成收束、避免长局失分等单一目标 |
| `successCondition` | 成功条件 | 3-6 分钟内可完成，且能回到排位建议 |
| `maxSteps` | 最大步数 | 防止 drill 变成长时间刷奖励 |
| `rewardPolicy` | 奖励口径 | 只给学习进度、徽记或非强度记录，不写正式积分 |
| `sharePolicy` | 分享口径 | 只能分享脱敏摘要和完成结果，不分享隐藏 payload |

drill 的设计目标是让复盘建议变成一次可执行练习，而不是生成“知道对手隐藏答案”的谜题。若某个 drill 必须依赖对手当时隐藏手牌才能成立，它不能从玩家复盘入口生成，只能作为内部测试 fixture。

好友约战 `friendlySeries` 合同：

| 项 | 首版口径 |
| --- | --- |
| 系列格式 | 默认 Bo3；单局约战可选，但 UI 必须区分 |
| 规则版本 | 系列创建时锁定；系列中途规则更新不影响已创建系列 |
| 座位轮换 | 第 1 局随机，第 2 局交换，第 3 局按前两局先手次数少的一方先手；相同则随机 |
| 局间调谱 | 默认允许双方各换 0-4 张并重新确认；也可选择完全同谱复现 |
| 中断处理 | 中断只影响约战记录，不写正式积分；已完成单局保留脱敏战报 |
| 超时 / 断线 | 按单局记录，不自动判整个系列胜负；高频断线可结束系列 |
| 奖励 | 不写正式积分、正式日周任务或主要奖励；可给约战记录、教学徽记和收藏 |
| 分享 | 只分享双方同意的脱敏系列摘要、比分和公开高光 |

失败复盘的最短行动原则：

- 每次失败复盘只主推 1 个主败因、1 个最短修正动作、1 个下一局可验证目标。
- 如果败因是操作型，例如错过公开 setup 或预算窗口，主推“继续排位”或“关键回合复现”。
- 如果败因是构筑型，例如曲线过高、防爆不足或缺少收束，主推“调谱 + 问道练习”。
- 如果败因是匹配跨度、弱网或异常局，主推“查看解释 / 争议反馈 / 重新匹配”，不能把责任推给玩家操作。
- 复盘必须提供“差一点能翻”的公开证据位，例如错过的防守窗、预算拦截后仍可反打的回合、长局分数差来源。

### 7.5.1 娱乐性与再来一局合同

公平只证明 PVP 不坑玩家，娱乐性还要证明玩家愿意再开一局。首版每局都应有清晰的情绪节奏：

| 阶段 | 情绪目标 | 设计抓手 | 失败信号 |
| --- | --- | --- | --- |
| 入队前 | 我知道这局想试什么 | 斗法谱标签、今日课题、推荐练习、对局预计节奏 | 只为了清任务而排位 |
| 开局 1-2 整轮 | 我看得懂对手方向 | 公开 setup、手牌数、资源状态、首轮预算提示 | 不知道为什么被压制 |
| 中盘 3-6 整轮 | 我有取舍和反制 | 进攻 / 防守 / 净化 / 蓄势 / 过牌的机会成本 | 每回合只有唯一正确动作 |
| 终局前 | 我感到还有悬念或可学习 | 公开爆发前兆、长局分数来源、反打窗口 | 一方从头碾到尾且复盘无调整点 |
| 结算后 | 我知道下一局做什么 | 再排位、练习、调谱、好友复现、收藏高光 | 只看到输赢和奖励，没有下一步 |

娱乐性审计指标：

| 指标 | 首版目标 | 用途 |
| --- | --- | --- |
| `stompRate` | 非投降、非严重断线局中，单方从第 2 整轮后全程无反击窗口的比例 `<=0.15` | 防止碾压局成为常态 |
| `closeGameRate` | 终局前两整轮内双方都仍有有效行动线的样本比例 `>=0.35` | 证明局内悬念足够 |
| `leadChangeOrThreatShiftRate` | 优势方、最大威胁或长局分数领先发生变化的样本比例 `>=0.30` | 证明中盘不是线性展开 |
| `postGameActionCoverage` | 结算页为每个常见败因提供至少 1 个下一步行动入口 | 防止复盘只是总结 |
| `deckEditFollowThroughRate` | 首败或连续失败后进入调谱 / 练习 / 好友复现的路径可被记录 | 衡量学习循环是否被触发 |
| `rematchIntentRate` | 结算页再来一局、约战或继续排位点击率进入灰度监控 | 作为娱乐性趋势，不作为强制 KPI |

这些指标不能用来暗中操控匹配或给数值补偿。若娱乐性不足，优先调整公开反馈、推荐练习、复盘入口、构筑差异和赛季课题；最后才调整战斗规则真值。

### 7.5.2 流派探索与熟练度合同

首版 8 套基准谱不是让玩家永远照抄，而是给探索提供安全边界。每套谱必须有：

- 一个一句话爽点：例如快攻的压迫、防守的稳住反打、连击的公开蓄势。
- 一个明确技能测试：何时防守、何时蓄势、何时净化、何时保留资源。
- 一个公开弱点：被哪类 setup、净化、护盾、速度或长局收束克制。
- 至少 2-4 张可替换非核心卡位，复盘建议必须能落到这些卡位。
- 一个练习课题，允许玩家在问道练习或好友约战中低压力验证。

流派熟练度只能记录理解和表达，不能给排位强度：

| 熟练度产物 | 允许 | 禁止 |
| --- | --- | --- |
| 流派徽章 | 展示完成过的练习、代表回放、赛季高光 | 提供生命、伤害、抽牌、灵力或起手加成 |
| 高光收藏 | 收藏关键回合、预算拦截、反打和长局判定 | 暴露隐藏手牌、牌库顺序或随机种子 |
| 练习课题 | 推荐公开 setup 应对、曲线调整、收束补强 | 变成进入排位前的强制刷关 |
| 好友复现 | 用同规则快照复现关键局面和测试替换卡 | 写正式积分或绕过合法池 |

### 7.6 反挫败底线

高对抗 PVP 必须控制挫败感。

- 连败时给明确建议和练习入口，不暗中篡改规则。
- 匹配等待过长时提示取消、继续等待或进入练习，但练习不产生正式结算。
- 被预算拦截时双方都能看到原因，避免胜方觉得系统吞伤害、败方觉得莫名其妙。
- 对局失败文案只解释局面，不羞辱玩家。
- 新玩家前几局优先匹配规则版本、段位和构筑强度接近的真人；匹配不到真人时不自动塞残影。
- 强制长局判定要解释分数来源，避免治疗 / 护盾玩家觉得被突然没收胜利。

真正的目标不是让每个人都赢，而是让输局也能产生“我知道下一局该试什么”的动机。

### 7.7 段位、赛季积分与结算激励

段位系统的目标是制造长期目标和荣誉表达，不是制造数值碾压。

首版使用三层数据：

| 数据 | 用途 | 玩家可见性 | 边界 |
| --- | --- | --- | --- |
| 匹配评分 | 服务端匹配公平和强弱估计 | 不直接展示数字 | 不能影响战斗数值 |
| 赛季积分 | 玩家看到的本赛季进度 | 可见 | 只能来自正式排位结算 |
| 段位徽章 | 荣誉、目标和赛季展示 | 可见 | 不能提供排位强度 |

匹配评分变化使用稳定公式：

- 胜利记为 `1`，失败记为 `0`；当前首版 `round14_draw` 不写匹配评分样本，后续若接入正式平局评分再按独立赛季公告启用 `0.5`。
- 预期胜率使用 `expected = 1 / (1 + 10 ^ ((opponentRating - ownRating) / 400))`。
- 评分变化使用 `ratingDelta = round(K * (actual - expected))`。
- 普通局 `K = 32`。
- 定级前 5 局 `K = 48`，用于快速归位。
- 高样本稳定玩家 `K = 24`，降低随机波动。
- `invalidated`、`ready_timeout`、服务端异常不改变匹配评分。

赛季积分变化必须更容易被玩家理解：

| 终局 | 赛季积分 | 说明 |
| --- | --- | --- |
| `lethal` 胜利 | 增加 | 按对手强度和本局质量调整 |
| `surrender` 胜利 | 增加较少或正常增加 | 防止刷投降，同时保护正常认负 |
| `timeout_forfeit` / `disconnect_forfeit` 胜利 | 增加较少 | 不鼓励通过拖局逼对手掉线 |
| 正常失败 | 减少 | 低段和定级期可见损失更平滑 |
| `round14_draw` | `0` | 长局分差不足时不把治疗 / 护盾局误判为胜负 |
| `invalidated` | 不变 | 无效局不改变正式进度 |

首版可见积分参数：

| 情况 | 积分变化 |
| --- | --- |
| 正常胜利 | `+20` |
| 宽跨度匹配胜利 | `+12` 到 `+16`，按跨度降低 |
| 对手投降 / 超时 / 断线胜利 | `+12` 到 `+18`，按对局进度决定 |
| 正常失败 | `-12` |
| 宽跨度匹配失败 | `-4` 到 `-8`，按跨度降低损失 |
| 定级期失败 | 最多 `-6` |
| `round14_draw` | `0` |
| `invalidated` / `ready_timeout` / 服务端异常 | `0` |

首版段位结构使用 5 个大段，每段 100 赛季积分。段位名称不影响规则真值，可后续本地化，但每段必须满足：

- 晋段只给徽章、称号、头像框、回放徽章、商店外观资格。
- 不给生命、伤害、抽牌、灵力、起手质量、卡牌强度。
- 小段保护只能影响可见赛季积分，不影响匹配评分。
- 降段保护如果存在，必须公开写明次数和范围，不能成为隐藏补偿。

积分和奖励禁止项：

- 不允许连败后暗中提高玩家战斗数值或降低对手强度。
- 不允许用付费或商店兑换直接提高排位胜率。
- 不允许让练习、约战、残影胜负写入正式赛季积分。
- 不允许重复终局消息重复发积分、货币、称号或熟练度。
- 不允许宽跨度匹配失败时隐藏扣分原因。

每次正式结算必须给玩家三类解释：

1. 本局终局类型：击杀、投降、超时、断线、长局判定、平局或无效局。
2. 积分变化原因：对手强度、匹配跨度、定级状态、终局类型。
3. 下一步入口：复盘、调整 2-4 张牌、练习基准谱、继续排位或好友约战。

### 7.8 首战引导与赛季变更沟通

V10 不能只把真人 PVP 入口放出来。第一次进入排位的玩家必须知道“这是真人、会影响正式积分、练习不等于排位”；老玩家必须知道赛季规则、合法池和数值为什么变化。

首战路径按低干扰原则设计：

| 环节 | 玩家需要理解 | 允许的体验 | 禁止事项 |
| --- | --- | --- | --- |
| 排位入口确认 | 真人排位、练习、好友约战的区别 | 首次点击展示一张短确认卡，可跳过后回看 | 不能把练习伪装成排位，也不能隐藏正式积分影响 |
| 规则短卡 | 生命、起手、抽牌、灵力、伤害预算、后手保护、禁用机制 | 3-5 张可折叠短卡，后续可在规则页回看 | 不能用长篇说明墙阻断玩家首战 |
| 基准谱推荐 | 新玩家可以从低复杂度斗法谱开始 | 推荐攻击、中速、防守三类基准谱，每类只给 3 个关键词 | 推荐谱不能有隐藏数值加成，也不能绕过合法池 |
| 排位前演武 | 调息、预算拦截、公开 setup、结束回合、复盘入口 | 可选 1 局短练习，允许跳过，不写正式积分 | 不能用演武结果代替真人排位，也不能发排位强度奖励 |
| 首场排位提示 | 定级样本、匹配扩圈、连接状态和首战保护 | 对局前展示等待和匹配解释，首战后给复盘入口 | 不能隐藏宽跨度匹配、断线风险或正式结算口径 |
| 首败复盘 | 玩家知道下一局该改什么 | 给 1-2 条可执行建议，链接练习和调整斗法谱 | 不能羞辱玩家，不能泄露对手隐藏手牌或牌库顺序 |

首战有限状态机：

| 状态 | 触发 | 允许动作 | 退出条件 |
| --- | --- | --- | --- |
| `first_entry_unseen` | 玩家本赛季第一次点击真人排位 | 查看确认卡、查看三模式差异、进入规则短卡 | 继续排位、练习、好友约战或跳过 |
| `intro_seen` | 已看确认卡 | 进入规则短卡、查看推荐谱、跳过本赛季提示 | 选择下一入口 |
| `rule_cards_seen` | 已看规则短卡 | 回看完整规则、进入推荐谱、进入排位前演武 | 选择练习、排位或约战 |
| `practice_offered` | 玩家选择先演武或首败后被推荐练习 | 进入问道练习、跳过练习、返回排位 | 练习结束或跳过 |
| `first_ranked_queueing` | 首次进入真人排位队列 | 继续等待、取消、进入练习 | 匹配成功、120 秒长等待或取消 |
| `first_ranked_finished` | 第一场真人排位完成 | 查看复盘、继续排位、调谱、好友复现 | 复盘关闭或进入下一步 |
| `first_loss_reviewed` | 首败复盘已打开 | 练习、调谱、继续排位、好友复现 | 下一步动作被选择 |
| `season_dismissed` | 玩家选择本赛季不再提示 | 仅保留回看入口 | 新赛季或手动重置提示 |

首战状态机必须可回看、可跳过、可审计；跳过只关闭引导，不关闭规则入口和失败复盘。

引导、公告和回访的关闭状态必须分域持久化，不能互相覆盖：

| 持久化域 | 作用 | 关闭范围 | 不能影响 |
| --- | --- | --- | --- |
| `first_battle_progress` | 首战确认卡、规则短卡、首败复盘路径 | 当前赛季首战引导提示 | 赛季公告、周目标、规则回看、复盘入口 |
| `season_change_announcement_ack` | 平衡补丁、合法池变化、规则版本和赛季重置确认 | 某个公告版本的已读状态 | 首战引导、日常 / 周常回访、旧回放说明 |
| `season_loop_dismiss_state` | 今日课题、周目标、失败参与奖励提示 | 本日、本周或本赛季的回访提示 | 首战引导、规则变更公告、正式排位入口 |

优先级：规则 / 合法池变更公告最高，必须在受影响入口前展示；首战引导只在首次真人排位路径展示；赛季回访只能在不阻断入口的轻提示位置展示。

新手复杂度预算：

- 首次排位入口同屏最多解释 3 个概念：真人、正式积分、练习隔离。
- 每张规则短卡最多 4 行，超过则拆成可回看的规则详情。
- 首版推荐谱不得包含 0 费、免费释放、无限递归、额外回合、硬控、随机闪避、复杂检索。
- 每套推荐谱必须有明确弱点，避免新玩家误以为推荐谱就是最强答案。
- 首战引导奖励只能给外观、称号、练习徽章或非强度货币，不能给排位数值。

赛季变更必须让玩家提前知道影响范围：

| 变更类型 | 玩家可见内容 | 生效规则 |
| --- | --- | --- |
| 文案修正 | 只解释文字变化，不改变结算 | 可热更新，但要保持旧回放语义可读 |
| 平衡数值 | 改动前后、影响构筑、调整原因 | 只影响新入队对局，已锁定对局继续使用入队快照 |
| 合法池变化 | 新增、禁用、转写或解禁的内容和公开原因 | 不合法旧斗法谱进入排位前给替换建议 |
| 规则版本变化 | 回合、预算、终局、积分等规则差异 | 必须生成新规则版本，回放显示当局旧版本 |
| 赛季重置 | 定级、段位、奖励、历史记录保留范围 | 重置只影响赛季积分和荣誉，不清除玩家可读战报 |

补丁沟通底线：

- 任何影响正式排位胜负、构筑合法性、赛季积分或奖励边界的变化，都必须有玩家可见公告。
- 公告必须说清楚“为什么改、改了什么、什么时候生效、旧斗法谱怎么处理”。
- 已经进入匹配或对局的玩家使用入队时规则快照，不能中途被新版本改变规则。
- 旧回放必须显示旧规则版本，不能用新规则重新解释历史胜负。
- 旧斗法谱因合法池变化失效时，必须给出禁用原因和替换入口，而不是只报错。
- 赛季补丁优先修复不公平、不可读、拖局和异常局问题；不要为了频繁制造新鲜感破坏学习稳定性。

### 7.9 双方体验公平与负面体验预算

“没有被先手秒杀”只是底线，不等于体验已经好。真人 PVP 必须继续证明输赢双方都曾经有可理解、可执行、可复盘的选择。

双方体验公平审计至少跟踪以下指标：

| 指标 | 含义 | 阻断口径 |
| --- | --- | --- |
| `nonGameLossCount` | 失败方在没有主动投降、严重断线或明确托管的情况下，少于 2 个真实决策窗口就输掉 | 必须为 0 |
| `controlLockWindowCount` | 玩家因为控制、弱化、费用税或状态叠加而没有任何有意义选择，只能空过 | 必须为 0 |
| `unreadableBurstCount` | 单窗口高伤害没有上一行动窗口公开 setup，或复盘无法解释其来源 | 必须为 0 |
| `seatAgencyP05` | 每个座位低分位局中的有效选择数量，包括可打牌、防御、净化、结束回合收益和调谱建议 | 任一座位低于阈值时不能封板 |
| `responseWindowP05` | 面对公开 setup 后至少还能行动、净化、防御、施压或选择长局策略的窗口数量 | 低于阈值时需要调整内容或预算 |
| `lossExplanationCoverage` | 失败回放能否给出公开可见败因和下一步动作 | 必须为 100% |
| `rejectFrictionRate` | 玩家因规则不清、状态过旧或目标不可选产生的拒绝密度 | 超阈值时先改文案、目标提示或交互，不先怪玩家 |

负面体验标签必须进入打磨闭环：

- `burst_without_setup`：玩家觉得被突然秒杀。
- `no_meaningful_choice`：玩家活着但没有有效选择。
- `control_lock`：控制效果让玩家无法玩出一回合。
- `budget_confusing`：预算拦截发生了，但玩家看不懂为什么。
- `dragging_loop`：局面长时间没有推进。
- `network_unfair`：断线、重连或 pending 让一方觉得被系统坑。
- `reward_pressure`：日常、周常或奖励诱导玩家用不想玩的方式掉分。
- `social_discomfort`：表情、再战、举报或分享造成干扰。

处理顺序：

1. 先确认是否是内容或构筑问题，例如缺少防御、净化、收束或公开 setup。
2. 再确认是否是交互问题，例如目标提示、pending、拒绝文案、计时和复盘不清。
3. 再确认是否是匹配问题，例如宽跨度、低样本保护或连接健康。
4. 最后才调整规则真值，例如生命、起手、预算和长局公式。

任何一轮打磨若出现 `nonGameLossCount > 0`、`controlLockWindowCount > 0`、`unreadableBurstCount > 0` 或 `lossExplanationCoverage < 1`，不能封板，只能进入下一轮调参、文案或匹配修正。

## 8. 平衡验收

首版封板前必须通过仿真和真实浏览器验收。

最低仿真门槛：

- 至少 10,000 局 quick gate。
- 封板前 32,000 局 full gate。
- 先手总胜率 47%-53%。
- 任意基准对阵胜率 45%-55%。
- 后手首次行动前死亡次数为 0。
- 后手首次行动无有效行动线次数为 0。
- P95 对局时长不超过 12 分钟。
- P99 对局时长不超过 15 分钟。
- 第 14 整轮后仍未结束局数为 0。

浏览器验收必须覆盖：

- 匹配、准备、调息、行动、结束、复盘完整路径。
- 断线、重连、超时、投降、无效局。
- 移动端布局。
- 对方隐藏信息不泄露。
- 旧残影练习不被描述为正式排位。

### 8.1 独立测试赛道

旧 PVP 绿灯不能证明真人排位可用。V10 需要独立验收赛道。

| 赛道 | 证明内容 |
| --- | --- |
| 规则仿真 | 先后手胜率、预算拦截、长局收束、构筑生态 |
| 协议一致性 | 幂等行动、重连恢复、重复结算防护、隐藏信息脱敏 |
| 浏览器体验 | 匹配、准备、调息、对战、断线、结算、复盘 |
| 移动端体验 | 小屏下 HUD、手牌、行动反馈、复盘不遮挡 |
| 文案口径 | 排位和练习分离，失败解释不羞辱玩家 |
| 生产 smoke | 真实登录、真实匹配、真实结算、正式 API 健康 |

只有这些赛道都通过，才可以说真人 PVP 首版具备上线条件。

### 8.2 核心风险验收矩阵

每条核心风险都必须有唯一主证据，不能用“看起来有测试”替代。

| 风险 | 主证据 | 必须产物 | Freshness 要求 |
| --- | --- | --- | --- |
| 规则真值冲突 | 规则仿真和规则快照校验 | `pvp-live-v1` 规则摘要、冲突扫描结果 | 每次规则文档或内容包变化后重跑 |
| 后手体验失败 | opening script 压测 + 浏览器首轮审计 | 后手有效行动失败数、失败 replay | 每次改预算、起手、身份槽、卡池后重跑 |
| 协议重复结算 | 协议一致性测试 | 幂等 action、重复终局、重连补发报告 | 每次改匹配、行动、结算、重连后重跑 |
| 内容映射漂移 | 内容归一化校验 | 全量内容分类、禁用原因、公开解释标签 | 每次卡牌或身份槽变化后重跑 |
| 隐藏信息泄露 | 状态视图脱敏审计 | 对手手牌、牌库顺序、随机状态泄露检查 | 每次改状态视图或 replay payload 后重跑 |
| 旧练习污染排位 | 练习/排位隔离审计 | 练习胜负不改正式积分的证据 | 每次改 PVP 入口、结算、商店、历史后重跑 |
| UI 口径误导 | 文案审计 + 浏览器截图 | 排位/练习/无效局/预算拦截文案截图 | 每次改 PVP 文案或页面结构后重跑 |

旧 ghost / practice 赛道只能证明练习可用，不能证明 live ranked 可用。

### 8.3 打磨迭代闭环

V10 的平衡不能靠一次性拍数值完成，必须按“证据 -> 调整 -> 复验”的节奏不断打磨。

每一轮打磨都必须留下四类证据：

| 证据 | 用途 |
| --- | --- |
| 仿真报告 | 判断先后手、流派对阵、长局、预算拦截和极端爆发 |
| 代表回放 | 判断玩家是否看得懂 setup、反制窗口和败因 |
| 浏览器截图 | 判断移动端 HUD、手牌、toast、结算和复盘是否遮挡或误导 |
| 反馈标签 | 标记玩家主观问题：被秒、拖局、看不懂、无事可做、奖励无感 |

调参顺序必须固定，避免一遇到问题就改核心规则：

1. 先看是否是内容池问题：禁用、转写或替换风险牌。
2. 再看是否是基准谱问题：调整单套斗法谱的曲线、解牌、防御和收束比例。
3. 再看是否是单效果数值问题：微调伤害、治疗、护盾、过牌或 setup 数值。
4. 再看是否是匹配问题：保护新玩家和低样本玩家，不把强弱差异伪装成规则问题。
5. 最后才改规则真值：生命、起手、抽牌、灵力、伤害预算、长局判定。

核心规则真值一旦改动，必须重新跑完整平衡门槛，不能只跑局部 smoke。

打磨过程禁止：

- 暗中给连败玩家加伤害、加抽牌或降低对手强度。
- 只看总体胜率，不看后手首次有效行动、对局时长和败因可读性。
- 只用高熟练玩家样本解释新手挫败。
- 用旧残影练习数据证明真人排位公平。
- 在没有回放样本的情况下把“玩家觉得不公平”简单归因于不会玩。

一轮打磨结束时，必须回答五个问题：

1. 是否仍有后手首次行动前死亡或无有效行动线。
2. 是否有单套构筑综合胜率超过上限。
3. 是否有玩家看不懂的预算拦截、长局判定或 setup 爆发。
4. 是否有奖励推动玩家为了数值而不是为了学习和表达去刷。
5. 是否有新引入的规则漂移、文案误导或隐藏信息泄露。

只有五个问题都有证据回答，才算完成一轮有效打磨。

## 9. 分阶段路线

本阶段只保存整体方案，不绑定具体文件。

后续推荐按五段推进：

1. 合同冻结：确认规则版本、内容边界、UI 文案、仿真门槛。
2. 规则原型：先实现最小战斗内核，验证后手行动权、预算、回放和脱敏状态。
3. 真人链路：接入匹配、准备、行动、重连、超时和结算。
4. 体验打磨：补齐 PVP 页面、HUD、移动端、复盘和练习/排位口径。
5. 封板验证：跑仿真、Node、浏览器、生产 smoke，再决定是否上线。

进入开发前再单独生成实施计划；实施计划必须以当时仓库状态为准，不能复用过期文件路径。

## 10. 进入开发前必须再冻结的事项

整体方向可以先存档，但进入开发前还必须把以下事项机械化，避免不同实现各自解释。

| 事项 | 必须冻结的内容 | 不冻结的风险 |
| --- | --- | --- |
| 单一规则真值 | 伤害预算、抽牌/回灵上限、0 费合法性、练习是否共用 live 引擎 | 各模块按不同文档实现，测试都能自洽但线上互相矛盾 |
| 服务端协议闭环 | match 生命周期、HTTP/WS 双通道、重连补发、幂等 action、结算只写一次 | 重复结算、pending 卡死、重连后视图不一致 |
| 内容真值映射 | 当前卡牌 effect schema 到 PVP 规则的拒绝、禁用、安全转写、合法快照 | 后端规则、仿真、前端展示各用各的数据格式 |
| 后手有效行动权 | 不只验证“没死”，还要验证有可用行动线、可防守、可结束回合且不被托管误伤 | 后手形式上行动过，但体验仍像被先手打崩 |
| 匹配质量 | 真人、规则版本、段位区间、匹配评分、等待扩圈、新玩家保护和连接健康 | 匹配到真人但强弱差过大、连接差或新玩家连续被碾压 |
| 对局手感与社交安全 | pending、拒绝反馈、对方回合等待、行动计时、弱网状态、预设表情、静音、举报、再战和战报分享边界 | 规则公平但操作迟钝、等待焦虑、社交骚扰或战报泄露隐藏信息 |
| 赛季积分 | 匹配评分、赛季积分、段位徽章、终局类型、宽跨度匹配和奖励边界 | 积分刺激错误，玩家为了刷奖励而不是为了学习和表达去排位 |
| 娱乐性与再来一局 | 局内悬念、优势变化、结算后下一步、练习转化和再战意愿 | 规则公平但无聊，玩家打一局就离开 |
| 流派探索 | 每套谱的爽点、技能测试、公开弱点、替换卡位和练习课题 | 8 套谱只是模板，玩家没有探索和表达空间 |
| 首战引导与版本沟通 | 首战短引导、练习演武、推荐谱、规则短卡、补丁公告和赛季变更解释 | 新玩家不知道怎么打，老玩家觉得规则暗改或构筑被突然作废 |
| 双方体验公平 | 非游戏局、无选择窗口、不可读爆发、控制锁死、败因解释和负面体验标签 | 虽然没被秒杀，但一方仍觉得整局没有选择、看不懂或被系统坑 |
| 反作弊与争议处理 | 篡改、重放、自动化、刷分、拖局、漏洞利用、申诉证据包和处置结论 | 玩家相信结果不可信，异常局污染积分，或正常玩家被弱信号误伤 |
| 旧 PVP 边界 | 旧残影、票据、练习、商店、排行榜哪些保留，哪些不能参与排位 | 旧链路继续影响正式排位或污染结果口径 |
| 独立测试赛道 | live ranked 的 Node、浏览器、移动端、生产 smoke 单列，不用旧 PVP 绿灯代替 | 现有 release gate 通过但不能证明真人排位可用 |
| 支撑文档优先级 | 主方案、内容包、UI 文案、fixture、需求拆解之间的真值顺序、漂移处理和重生成条件 | 旧需求路径或旧 fixture 被误当成已冻结实现，导致开发计划和当前仓库结构脱节 |
| 上线灰度与回滚 | 候选版准入、生产 smoke、灰度范围、监控指标、阻断阈值和回滚动作 | 功能可用但正式服不稳定，玩家积分、奖励或隐藏信息被错误暴露 |

下一份开发计划应先冻结这些事项，再拆具体文件和任务。

### 10.1 非文件级开发合同包

进入开发前不先锁具体修改文件，但必须先冻结以下合同。它们是后续直接开发的依据，也是未来按最新仓库结构拆任务的边界。

| 合同 | 必须冻结的内容 | 完成证据 |
| --- | --- | --- |
| 规则快照合同 | `pvp-live-v1` 的生命、卡组、起手、抽牌、灵力、预算、长局、禁用机制和版本号 | 任意对局、仿真、回放都能输出同一规则摘要 |
| 构筑快照合同 | 斗法谱、身份槽、合法牌池、禁用原因、内容 hash、入队时锁定字段 | 入队后不能通过客户端改牌、换身份或绕过禁限表 |
| 对局生命周期合同 | `idle` 到 `finished/invalidated` 的状态、允许行为、退出条件和超时处理 | 任意状态都有唯一下一步，断线重连不会生成第二场或第二次结算 |
| 行动意图合同 | 出牌、结束回合、调息、投降、重连补发的最小字段、幂等 ID 和拒绝原因 | 重复提交同一行动只返回同一结果，不重复扣资源或播奖励 |
| 座位视图合同 | 本方完整信息、对方脱敏信息、公开状态、隐藏手牌、牌库顺序和随机数边界 | 浏览器侧无法读到对手手牌、牌库顺序或随机种子 |
| 事件回放合同 | 每次行动、预算拦截、setup、伤害、治疗、护盾、终局和托管动作的日志格式、可见性层级和复盘标记 | 战报能复现关键局面，且不泄露隐藏信息 |
| 匹配质量合同 | 规则版本、赛季、段位区间、匹配评分差、等待扩圈、连接健康、新玩家保护和拒绝原因 | 匹配到的是真人且双方体验跨度可解释，不靠残影和暗改补偿 |
| 对局手感合同 | pending、accepted、rejected、duplicate、sync_required、弱网、重连、动画跳过、行动计时和移动端确认 | 玩家知道行动是否可信、为什么被拒绝、现在轮到谁、还能做什么 |
| 双方体验公平合同 | 非游戏局、控制锁死、不可读爆发、有效选择低分位、败因解释覆盖率和负面体验标签 | 不只证明没有先手秒杀，也证明输赢双方都曾经有选择、看得懂结果 |
| 娱乐性合同 | stomp rate、close game rate、优势变化、结算后行动入口、调谱 / 练习转化和再战意愿 | 证明首版不是只公平可用，也有悬念、爽点和再来一局动机 |
| 流派探索合同 | 每套基准谱的爽点、技能测试、公开弱点、替换卡位、练习课题和代表回放 | 玩家能从基准谱走向个性化构筑，而不是被必带牌和模板锁死 |
| 社交安全合同 | 预设表情、静音、限频、举报、拉黑、再战、好友约战、战报脱敏分享和实时观战禁用 | 玩家有对手存在感，但不会被骚扰、窥牌或用社交绕过正式结算 |
| 结算合同 | 胜负、平局、无效局、投降、超时、断线、长局判定、段位和经济写入规则 | 结算只写一次；无效局不改正式积分 |
| 赛季积分合同 | 匹配评分、赛季积分、段位徽章、定级、宽跨度匹配、投降 / 超时 / 断线的积分处理 | 玩家知道积分为什么变化，奖励不提供排位强度 |
| 首战引导合同 | 排位入口确认、规则短卡、推荐谱、排位前演武、首败复盘和可回看入口 | 新玩家不读长文档也能完成第一场真人排位，且没有隐藏数值补偿 |
| 赛季变更合同 | 平衡补丁、合法池变化、规则版本变化、赛季重置、旧回放兼容和构筑替换建议 | 玩家知道为什么改、什么时候生效、旧斗法谱和旧战报怎么处理 |
| 风控争议合同 | 篡改、重放、自动化、刷分、拖局、漏洞利用、申诉证据包、风险标签和处置结论 | 异常局可隔离、可复核、可回滚，正常玩家不会被单一弱信号误伤 |
| 体验文案合同 | 排位、练习、约战、预算拦截、断线、失败建议、奖励边界的玩家可见文案 | 玩家不会把练习误认为正式排位，不会误解预算和奖励 |
| 打磨证据合同 | 仿真报告、代表回放、截图、反馈标签、调参记录和回滚依据 | 每轮调参都有前后对比，不靠主观判断上线 |
| 支撑文档漂移合同 | 主方案、内容包、需求拆解、UI 文案样例、fixture 合同的优先级、冲突记录、重生成条件和引用入口 | 后续开发知道哪些是冻结合同，哪些只是旧路径、样例或素材 |
| 上线灰度合同 | 候选版准入、灰度人群、生产 smoke、监控阈值、回滚动作和正式结论 | 正式排位只在证据通过后开放，异常时能先保住玩家数据和公平口径 |

这些合同可以被映射到任意未来代码结构，但合同本身不能随着实现方便而漂移。

### 10.2 可开发用户故事与验收场景

以下用户故事是进入实施计划前的最小开发 backlog。它们只冻结体验和验收结果，不冻结具体修改文件。

| ID | 用户故事 | 验收场景 |
| --- | --- | --- |
| PVP-US-01 | 作为想打排位的玩家，我能选择合法斗法谱并进入真人匹配 | 选择合法斗法谱后进入 `queueing`；界面显示规则版本、本方构筑锁定状态和取消入口；匹配到的对手必须是真实在线玩家，排位不会在开局前暴露对手构筑 hash、身份槽或完整牌表 |
| PVP-US-02 | 作为排位玩家，我不会在真人不足时被系统悄悄塞进残影局 | 匹配超时后只能继续等待、取消或进入练习；若进入练习，文案和结算都明确不影响正式积分 |
| PVP-US-03 | 作为构筑玩家，我能在入队前知道哪些牌不能用于排位 | 含 0 费、复制、造牌、额外回合、硬控、随机闪避等禁用内容时，入队被拒绝，并显示公开禁用原因 |
| PVP-US-04 | 作为公平对局玩家，我的入队构筑会被锁成权威快照 | 进入 `queueing` 后锁定斗法谱、身份槽、合法牌池和内容 hash；重连、刷新或重复提交不能换谱、换身份或绕过禁限表 |
| PVP-US-05 | 作为对局双方，我能在准备阶段完成一次调息换牌 | 双方进入 `setup` 后可换 0-2 张且每局一次；确认后不能继续改斗法谱；准备超时进入明确的无效局或判定流程 |
| PVP-US-06 | 作为行动方，我的每次出牌都有 pending、成功或拒绝反馈 | 合法行动推进状态版本并写事件；非法行动返回稳定错误码，不扣资源、不改状态、不让客户端自行补算 |
| PVP-US-07 | 作为后手玩家，我不会在第一次真实行动前死亡或只能空过 | opening script 覆盖所有基准对阵；后手首次行动前死亡数为 0，无有效行动线次数为 0 |
| PVP-US-08 | 作为进攻方，我能打出压迫感，但不能靠不可读爆发秒杀 | 超过预算的伤害被截断，行动仍按 accepted 推进并生成 `budget_clamped` 公开事件；上一行动窗口没有公开 setup 的高爆发不能绕过预算 |
| PVP-US-09 | 作为防守方，我能理解为什么对手的爆发被拦截或命中 | UI 和复盘显示 setup 来源、预算拦截、实际伤害和被截断伤害；不显示隐藏手牌或牌库顺序 |
| PVP-US-10 | 作为真人对局玩家，我不会从实时状态、重连或回放里看到对手隐藏信息 | 本方能看到完整手牌和牌库摘要；对方视图只显示公开状态、手牌数量、牌堆数量和公开事件；任何 payload、回放、截图都不能泄露对手手牌、牌库顺序或随机种子 |
| PVP-US-11 | 作为断线玩家，我能在短断线后恢复对局状态 | 重连后收到服务端权威状态和已接受行动结果；客户端 pending 不能覆盖服务端状态 |
| PVP-US-12 | 作为未断线玩家，我不会被对方反复断线拖到无体验 | 对局中玩家侧多次超时或断线默认进入托管后判负；`invalidated` 只用于准备阶段无效局或服务端异常；托管不能打出决定性高爆发，且托管动作写入复盘 |
| PVP-US-13 | 作为认输或超时的一方，我看到的结算与对方一致 | 投降、超时、断线、击杀、长局判定都进入唯一终局；重复终局消息不重复发奖励 |
| PVP-US-14 | 作为排位玩家，我的正式结果只会被服务端写入一次 | 段位、赛季记录、正式历史和经济由服务端按终局写一次；重连、刷新、重复终局消息不能重复发奖励，也不能漏写已完成对局 |
| PVP-US-15 | 作为复盘玩家，我能知道这一局输在哪里，并立刻进入下一步行动 | 复盘至少显示终局原因、最大单回合伤害、最大 setup 来源、预算拦截、首次优势回合和下一局建议；建议能带上下文进入练习、斗法谱调整、继续排位或好友复现 |
| PVP-US-16 | 作为练习玩家，我能用残影或托管练反制但不污染排位 | 练习胜负不写正式积分、段位、正式历史；练习可产出学习建议和样本回放 |
| PVP-US-17 | 作为好友约战玩家，我能低压力测试构筑 | 约战可指定玩家、不影响正式积分；仍使用合法池和脱敏视图，不能绕过规则产生正式战绩 |
| PVP-US-18 | 作为追求赛季的玩家，我能获得荣誉和表达奖励 | 排位奖励只给段位、称号、头像框、回放徽章、熟练度或非强度货币；不能提高生命、伤害、抽牌、灵力或起手质量 |
| PVP-US-19 | 作为移动端玩家，我能完成完整 PVP 路径 | 小屏下匹配、准备、手牌、行动反馈、断线提示、结算和复盘都可读、可点、不遮挡关键信息 |
| PVP-US-20 | 作为排位玩家，我能匹配到体验跨度合理的真人对手 | 匹配只使用同规则版本真人；等待扩圈有提示；宽跨度匹配被记录并在结算中解释；匹配不到真人时不自动塞残影 |
| PVP-US-21 | 作为赛季玩家，我能理解积分和段位为什么变化 | 每局正式结算展示终局类型、积分变化原因和下一步入口；练习、约战、无效局不改正式赛季积分 |
| PVP-US-22 | 作为正式服玩家，我只会在通过灰度和生产验证的版本中进入排位 | 未通过候选版证据时正式排位入口保持关闭或测试赛季隔离；生产 smoke、监控阈值和回滚口径通过后才开放正式积分 |
| PVP-US-23 | 作为公平排位玩家，我能在异常局后提交争议并获得可复查结论 | 结算页和战报页可提交异常反馈；证据包能复核规则版本、事件日志、状态 hash、结算写入和风险标签；弱信号不能直接扣分或封禁 |
| PVP-US-24 | 作为第一次打真人 PVP 的玩家，我能用短引导完成首战，并理解首战异常分支 | 排位入口解释真人、练习和正式积分区别；规则短卡可折叠和回看；推荐谱无隐藏加成；排位前演武不写正式积分；120 秒无真人、宽跨度匹配、找到对手后断线、ready timeout、`refresh_required` 都有不写错积分的解释；首败后给练习和调整斗法谱入口 |
| PVP-US-25 | 作为赛季玩家，我能理解补丁和赛季变更对我的影响 | 平衡、合法池、规则版本和赛季重置都有公告；入队后不被中途切规则；旧回放显示旧规则；不合法旧斗法谱有替换建议 |
| PVP-US-26 | 作为实时对局玩家，我能清楚感知每次行动、等待和同步状态 | 提交行动后有 pending；接受、拒绝、重复提交和需要同步都有不同反馈；对方回合显示思考计时和连接状态；弱网重连不会让旧状态覆盖服务端状态 |
| PVP-US-27 | 作为真人 PVP 玩家，我能安全地表达、再战、举报或屏蔽对手 | 只开放预设表情且可静音和限频；结算后可再战、加好友、分享脱敏战报或举报异常；实时观战不开放；社交动作不影响积分、匹配评分或奖励 |
| PVP-US-28 | 作为后续开发负责人，我能判断主方案和支撑文档冲突时该以哪个为准 | 文档入口列出优先级；旧文件路径、模块名、fixture 路径只作为素材；进入开发前必须按最新仓库结构重生成实施计划；冲突项有漂移审计记录 |
| PVP-US-29 | 作为赛季回访玩家，我每天和每周都有不逼迫掉分的回访目标 | 当日课题来自复盘主题或推荐练习，不强迫玩家用弱构筑打正式排位；失败参与奖励有每日上限；周目标只给荣誉、展示、练习徽章或非强度货币；赛季循环审计能证明奖励不影响生命、伤害、抽牌、灵力、起手或匹配评分 |
| PVP-US-30 | 作为输赢任一方，我都能感到本局有选择、有反制窗口、输赢原因可解释 | 非投降、非严重断线局中失败方至少经历 2 个真实决策窗口；没有控制锁死、不可读爆发或无解释败因；复盘能给出公开败因和下一步动作 |
| PVP-US-31 | 作为打完一局的玩家，我能自然选择再来一局或去练习调整 | 结算页按终局和败因提供继续排位、问道练习、调谱、好友复现或收藏高光；常见败因都有下一步入口；娱乐性审计能证明碾压局比例、悬念局比例和优势变化不失衡 |
| PVP-US-32 | 作为构筑探索玩家，我能从基准谱发展出自己的打法 | 每套基准谱都有爽点、技能测试、公开弱点、2-4 张替换卡位、练习课题和代表 replay；复盘建议能落到替换卡位；流派熟练度和徽章只给表达不提供排位强度 |

每个用户故事进入开发前都要补三类验收材料：

- 主路径：玩家正常操作时应看到的状态、文案和结果。
- 失败路径：非法构筑、超时、断线、重复提交、匹配不到真人等异常结果。
- 证据路径：需要用仿真、协议测试、浏览器截图、回放样本或生产 smoke 中哪一类证明。

如果某个开发切片无法对应至少一个用户故事，它不应进入首版真 PVP 范围。

### 10.3 开工准备与完成验收

每个用户故事进入开发前，必须先达到 Definition of Ready。未满足 Ready 的故事只能继续策划，不能直接开工。

| Ready 项 | 必须具备 | 不满足时的处理 |
| --- | --- | --- |
| 规则锚点 | 关联到 `pvp-live-v1` 中的具体规则项、禁用项或状态机节点 | 先补规则真值，不能让实现自行解释 |
| 玩家文案 | 主路径、失败路径、拒绝原因、行动状态、等待状态、社交状态、首战引导、补丁公告、结算提示都有玩家可见口径 | 先补文案合同，避免实现后再补解释 |
| 数据输入 | 需要的斗法谱、内容分类、身份槽、状态标签或回放字段已经定义 | 先补内容映射或合同字段 |
| 风险标签 | 明确是否涉及后手行动权、隐藏信息、重复结算、断线、行动计时、社交表达、奖励边界、首战引导、版本变更、风控争议、移动端布局 | 风险未标记时不能估算工作量 |
| 验收证据 | 确认用仿真、协议测试、浏览器截图、回放样本、生产 smoke 中哪几类证明 | 没有证据路径的故事不能进入开发 |
| 回滚口径 | 明确失败时回退到练习、无效局、禁用内容、旧入口隔离或规则版本回滚哪一种 | 没有回滚口径时不能上线试运行 |
| 文档优先级 | 确认本故事涉及的主方案、内容包、需求拆解、文案样例和 fixture 是否存在冲突 | 冲突未处理时不能进入开发，只能先做漂移审计 |

每个用户故事完成后，必须达到 Definition of Done。

| Done 项 | 必须证明 |
| --- | --- |
| 主路径可用 | 玩家能按用户故事完成完整体验，状态推进符合协议 |
| 失败路径可解释 | 非法构筑、重复提交、超时、断线、匹配失败、预算拦截都有明确反馈 |
| 服务端权威 | 胜负、抽牌、随机、隐藏信息、结算、奖励不依赖客户端自报 |
| 公平底线 | 不引入先手秒杀、后手无行动线、完整回合剥夺、隐藏爆发或排位数值奖励 |
| 回放可读 | 关键行动、预算拦截、setup、终局、托管动作能在复盘中解释 |
| 模式隔离 | 排位、练习、约战、残影、商店、赛季记录没有互相污染 |
| 手感可验证 | pending、接受、拒绝、重复提交、同步恢复、对方回合等待、行动计时和移动端确认都可感知、可操作 |
| 社交不干扰 | 表情、静音、限频、举报、拉黑、再战和战报分享不影响规则裁定、匹配评分、正式积分或隐藏信息边界 |
| 新手可进入 | 首战引导、推荐谱、排位前演武和首败复盘能帮助新玩家完成第一场真人排位，且没有隐藏战斗补偿 |
| 变更可理解 | 平衡补丁、合法池变化、规则版本和赛季重置都有公告、旧回放兼容和斗法谱替换建议 |
| 风控可复核 | 异常对局能生成证据包；篡改、重放、刷分、恶意拖局和争议结论都有稳定处置 |
| 文档不漂移 | 当前实现依据能追溯到主方案和支撑文档优先级；旧路径、旧模块名或旧 fixture 不被误当成冻结事实 |
| 移动端可用 | 小屏下关键信息和操作不被遮挡，按钮和手牌可触达 |
| 证据可复查 | 仿真报告、截图、回放、测试输出或 smoke 结果可被后续版本复核 |

若一个故事只满足主路径但缺失败路径、回放解释或证据产物，只能算内部半成品，不能标记为首版完成。

切片推进规则：

- V10-S0 未冻结前，不能开始任何会影响排位结果的开发。
- V10-S1 和 V10-S2 未通过前，不能开放真人排位入口。
- V10-S4 未通过前，不能写正式积分、赛季记录或经济奖励。
- V10-S5 未通过前，不能宣称“失败可学习”。
- V10-S8 未通过前，不能宣称首版可上线。

允许并行的范围：

- V10-S2 的仿真脚本、V10-S3 的准备房间体验、V10-S5 的复盘文案可以在 V10-S1 有稳定事件日志后并行。
- V10-S6 和 V10-S7 可以在正式结算合同稳定后并行，但不得提前写正式积分。
- 移动端验收可以从 V10-S3 开始持续跟随，不必等到最后一次性补。

若某个切片发现规则真值冲突，必须回到 V10-S0 修正规则快照，再重新跑受影响切片的 Done 证据。

### 10.5 验收证据产物模板

后续具体文件结构可能变化，但每个切片的验收证据不能变化。证据可以是文档、测试输出、审计页面、截图集合或自动报告；本方案只冻结证据内容，不冻结保存位置。

| 证据类型 | 证明问题 | 必要字段 | 覆盖切片 | 失败即阻断 |
| --- | --- | --- | --- | --- |
| 规则快照报告 | 当前所有入口、仿真和回放读取的是同一套 `pvp-live-v1` 真值 | 规则版本、生命、卡组、起手、抽牌、灵力、预算、回合阶段、长局判定公式、禁用机制、生成时间、参与校验的规则摘要 | V10-S0 / V10-S1 / V10-S8 | 阻断所有正式排位开发和上线宣称 |
| 内容映射报告 | 首版合法池、禁用池、转写池和未来候选没有漂移 | 内容 ID、PVP 类别、原始效果摘要、PVP 结算语义、公开解释标签、隐藏信息边界、构筑快照字段、仿真标签 | V10-S0 / V10-S2 / V10-S6 | 阻断入队、匹配、仿真和构筑相关开发 |
| 构筑快照校验 | 玩家入队后不能改谱、换身份或绕过禁限表 | 玩家席位、规则版本、斗法谱摘要、身份槽、合法池 hash、禁用原因、快照生成时间、重复提交结果 | V10-S0 / V10-S3 / V10-S4 | 阻断真人匹配和正式结算 |

### 10.6 上线灰度与生产验证合同

本节只冻结上线口径，不冻结具体部署脚本、测试文件或监控实现。正式环境以 `https://080305.xyz/` 为玩家访问目标；进入开发后可以按当时生产架构映射执行方式，但不能降低以下门槛。

发布候选版准入：

- V10-S0 至 V10-S8 的阻断证据全部通过。
- `pvp-live-v1` 规则版本、内容版本、合法池、禁用池和结算口径已冻结。
- 没有未解释的后手首次行动前死亡、无有效行动线、隐藏信息泄露、重复结算或正式积分污染样本。
- 生产 smoke 的账号、测试赛季、正式积分隔离和回滚动作已确认。
- 回滚后玩家能继续使用练习、约战、战报和商店中不依赖 live ranked 的功能。

生产 smoke 必须覆盖：

| 场景 | 通过标准 | 失败处理 |
| --- | --- | --- |
| 正式站点可达 | 正式域名首页返回可访问状态，静态资源加载正常 | 阻断开放排位，先恢复基础站点 |
| API 健康 | 同源 API health 返回健康状态，版本可追溯 | 阻断所有在线链路 |
| 注册与登录 | 新账号可注册、登录、刷新后保持会话 | 阻断真人匹配和存档写入 |
| 存档读写 | 登录态能保存、读取、冲突处理且不污染本地离线档 | 阻断正式服长期进度写入 |
| 残影上传与拉取 | 练习素材可上传和拉取，但不写正式排位 | 阻断练习资料入口，不影响排位判断 |
| 测试赛季真人 PVP | 两个受控账号能进入同规则版本测试赛季、完成匹配、行动、终局、复盘和隔离结算 | 阻断正式排位入口 |
| 正式积分隔离 | 测试赛季和 smoke 账号不改变正式赛季积分、正式奖励和公开榜单 | 立即停止 smoke 写入并检查结算合同 |

灰度阶段：

| 阶段 | 开放范围 | 排位结果 | 退出条件 |
| --- | --- | --- | --- |
| 内部测试赛季 | 仅受控账号和开发验证入口 | 不写正式积分、不发正式奖励 | 生产 smoke 全通过，0 个阻断事故 |
| 小范围自愿灰度 | 少量明确选择参与的玩家 | 可写灰度赛季积分，不进入正式赛季榜 | 连续观察窗口内无公平、结算、隐藏信息事故 |
| 正式赛季预开放 | 正式入口逐步开放，保留一键停排能力 | 写正式积分和荣誉奖励 | 监控稳定且玩家复盘、申诉和异常样本可追溯 |
| 全量开放 | 常规赛季入口开放 | 正式写入 | 继续保留版本回滚和停排开关 |

线上监控至少跟踪：

- 匹配成功率、取消率、等待时间 P50/P95、宽跨度匹配占比。
- ready timeout、行动 timeout、断线判负、重连成功率。
- `budget_clamped` 次数、预算截断后终局分布、无公开 setup 爆发阻断数。
- 后手首次真实行动前死亡数、后手无有效行动线数。
- `match_finished` 与 `settlement_written` 的一对一关系、重复结算数、无效局数。
- 隐藏信息审计失败数、回放脱敏失败数、未授权状态同步数。
- 篡改拒绝数、重放提交数、异常投降链、重复匹配刷分风险、争议申诉量和结论分布。
- 生产 API health、错误率、关键请求 P95、前端控制台错误和移动端阻断截图。

立即阻断条件：

- 任意隐藏手牌、牌库顺序、随机种子或服务端私密字段泄露到对手视图、回放公开视图或截图。
- 任意正式真人 PVP 中出现后手首次真实行动前死亡。
- 任意正式真人 PVP 中出现后手无合法非投降行动线且未被规则标记为已终局。
- 任意正式对局出现重复结算、重复发奖励、客户端自报胜负改正式积分。
- 正式排位被残影、托管练习或测试赛季结果污染。
- 出现可复现的客户端篡改改写权威状态、批量刷分或申诉证据包缺失。
- 生产 API health 失败或登录、存档、结算任一关键链路持续异常。

回滚动作按轻重分层：

1. 关闭正式排位入口，保留练习、约战、战报和商店的安全部分。
2. 冻结新匹配，允许已进入 active 的对局按规则结束；若服务端状态不可信，进入 `server_invalidated`，不改正式积分。
3. 禁用新内容或新身份槽，回到上一版已验证的规则版本和合法池。
4. 暂停正式赛季积分写入，只保留测试赛季和只读复盘。
5. 回退前端入口和服务端规则版本；已完成的正式结算不得静默改写，必须通过审计记录和补偿流程处理。

上线结论只能有三类：

- `go`: 全部阻断证据通过，可以开放或扩大正式排位。
- `hold`: 主路径可用但存在非阻断风险，只能继续灰度或测试赛季。
- `rollback`: 触发阻断条件，必须停排、隔离或回退，不能用文案解释代替修复。

### 10.7 支撑文档优先级与漂移处理合同

本节只冻结文档真值顺序，不冻结具体文件路径。进入开发前，必须先完成一次文档漂移审计。

真值顺序：

| 冲突类型 | 裁定顺序 |
| --- | --- |
| 玩法目标、公平底线、上线门槛 | 本文优先 |
| 合法牌池、禁用内容、基准斗法谱、身份槽 | 内容包优先，但不得突破本文禁用机制 |
| 玩家可见文案、toast、按钮、页面短句 | UI 文案样例优先，但不得改变本文规则和奖励边界 |
| 仿真 fixture、golden replay、报告字段 | fixture 合同优先，但失败阻断以本文证据模板为准 |
| 旧开发路径、模块名、接口草案 | 只能作为素材，必须按开发时最新仓库结构重生成实施计划 |

漂移审计必须列出：

- 当前使用的主方案、内容包、需求拆解、UI 文案样例、fixture 合同版本。
- 每个支撑文档中是否存在具体文件路径、模块名、接口路径或测试文件名。
- 这些路径是否仍符合当前仓库结构。
- 若不符合，是否只保留语义合同并重生成实施路径。
- 是否存在规则数字冲突，例如生命、卡组、起手、抽牌、灵力、预算、长局判定。
- 是否存在范围冲突，例如某支撑文档开放了本文禁用机制。
- 是否存在验收冲突，例如某支撑文档把旧 PVP 绿灯当成真人排位门禁。
- 最终开发依据：哪些条目被保留、哪些被替换、哪些必须重新冻结。

开发计划生成规则：

1. 先做文档漂移审计。
2. 再按当时仓库结构生成实施计划。
3. 实施计划可以引用支撑文档的 schema、文案和 fixture 语义，但不能无审计复制旧文件路径。
4. 若开发中发现支撑文档和主方案冲突，必须先更新方案或记录裁定，再继续实现。
5. 若代码结构已经变化，优先保持本文的体验和证据目标，不为迁就旧路径降低 PVP 公平、隐藏信息或上线门槛。

### 10.8 跨文档真值索引

本节用于保存整体方案的裁定结果，不绑定具体开发文件。后续开发计划可以重映射文件路径，但不能改变下列真值。

| 真值主题 | 裁定口径 | 支撑合同 | 开发前必须复查 |
| --- | --- | --- | --- |
| 长局终局 | 第 14 整轮按公开多指标分数结算；`round14_score` 需要分差至少 5；不足为 `round14_draw` | 主方案、需求拆解、fixture | 不得重新引入旧压力伤害或生命护盾简化公式 |
| 匹配公平 | 评分差 `0-99` 严格近分，`100-199` 只在长等待后扩圈，`200+` 当前首版不自动匹配，未来若开放必须先有显式确认 | 主方案、需求拆解、UI 文案 | 最近候选选择、宽跨度样本比例、等待时长和结算解释必须进入证据 |
| 首战引导 | 真人排位、问道练习、好友约战三模式必须区分；跳过只影响本赛季提示 | 主方案、需求拆解、UI 文案 | 多端恢复、赛季重置和首败复盘入口不能丢 |
| Replay 可见性 | 玩家回放不能返还完整对手手牌、牌库顺序、未公开 RNG 或未公开意图 | 主方案、需求拆解、fixture | 公开战报建议必须能从 `replay_public` 或 `audit_safe` 推导 |
| 稳定拒绝码 | 首版拒绝码只允许使用冻结清单；旧 alias 只能保留迁移说明 | 主方案、需求拆解、UI 文案 | UI toast、协议、测试 fixture 不能各自发明新码 |
| 内容合法性 | 首版无 0 费牌、无额外回合、无硬控、无无限抽回灵循环 | 主方案、内容包、fixture | 卡牌效果变化必须重跑合法池和仿真 |
| 双方体验公平 | 非投降、非严重断线局中失败方至少经历 2 个真实决策窗口，且败因可公开解释 | 主方案、fixture、UI 文案 | 软锁死、不可读爆发、无解释败因任一出现都阻断封板 |
| 赛季循环 | 赛季任务只给表达和非强度奖励，不强迫弱构筑进正式排位 | 主方案、需求拆解、UI 文案 | 日/周/赛季提示、奖励 ledger 和关闭状态必须可审计 |
| 争议处理 | 争议只用事件日志和安全视图复核；不静默改分，不泄露风险字段 | 主方案、需求拆解、UI 文案 | 奖励暂挂、撤销、补偿和重复工单都有玩家可见结论 |

### 10.9 当前未冻结范围

为了避免在仓库结构变化前锁死实施路径，当前只冻结整体方案与开发合同，不冻结：

- 具体要修改哪些源码文件、目录或模块边界。
- 具体接口、测试文件和产物目录的最终落点。
- 线上灰度批次、配置开关名称和运营后台入口。
- 最终 UI 排版、组件命名、资源文件名和动画实现方式。

进入开发前必须先完成一次实施计划重生成：以本方案为体验和证据边界，再按当时最新仓库结构拆分具体文件、模块、接口、测试和上线步骤。

### 10.10 实施计划生成门禁

下一阶段可以开始生成开发计划，但不能直接跳到源码改动。生成实施计划前必须先交付一份不绑定具体文件的输入包，证明本方案已经能被拆成任务。

实施计划输入包必须包含：

| 输入 | 必须回答的问题 | 不通过时 |
| --- | --- | --- |
| 合同冻结记录 | `pvp-live-v1` 的规则、内容、状态机、结算、回放、奖励和上线门槛是否已经有唯一口径 | 回到 V10-S0，继续打磨主方案 |
| 用户故事覆盖矩阵 | PVP-US-01 至 PVP-US-32 是否都映射到至少一个切片、至少一类主证据和至少一条失败路径 | 缺口故事不能进入开发，只能补合同 |
| 风险域映射 | 每个故事涉及后手行动、隐藏信息、重复结算、匹配公平、奖励边界、移动端或争议处理中的哪些风险 | 风险未标记时不能估算工期或拆任务 |
| 证据门禁矩阵 | 每个切片退出时需要哪些报告、截图、仿真、回放、smoke 或审计字段 | 没有证据的功能不能被标记完成 |
| 回滚和隔离口径 | 每个切片失败时回到练习隔离、停排、无效局、禁用内容、规则回滚或测试赛季中的哪一种 | 没有回滚口径时不能开放给真实玩家 |
| 文档漂移审计 | 主方案、需求拆解、内容包、UI 文案和 fixture 是否仍一致，旧路径是否仍只是素材 | 冲突未裁定前不能生成文件级计划 |
| 当前仓库状态摘要 | 当前实现中哪些旧 PVP、残影、客户端胜负、本地结算和旧票据路径必须被替换或隔离 | 若无法区分旧实现和 V10 目标，不能开工 |

实施计划输出只能先拆到以下层级：

- 体验切片：匹配、准备、战斗、重连、结算、复盘、练习、约战、赛季、上线。
- 责任域：规则真值、状态同步、可见性、内容映射、证据产物、玩家文案、风控争议、灰度验证。
- 验收门槛：每个切片的主路径、失败路径、证据路径和回滚口径。

实施计划在完成文档漂移审计前不得写入：

- 最终源码文件名、目录结构或模块 owner。
- 最终接口路径、数据库迁移文件名或测试文件名。
- 最终部署脚本、配置键名或监控面板地址。

允许出现“建议候选落点”，但必须标注为可替换素材，并在开发前按当前仓库结构重新确认。

如果实施计划生成时发现任何阻断级冲突，应优先更新本文或支撑合同，而不是为了赶进度让实现自行解释。

## 11. 当前完成定义

作为整体方案，本文完成后应满足：

1. 明确 V10 的核心卖点是真人、公平、可复盘 PVP。
2. 明确排位和练习分离，旧残影不能充当正式排位。
3. 明确首版规则、节奏、回合结构、反先手秒杀防线。
4. 明确首版开放内容和暂不开机制。
5. 明确服务端权威和隐藏信息边界。
6. 明确日常、周常、赛季、奖励、练习、约战、复盘如何提高可玩性。
7. 明确 PVP 成长只给表达和学习，不给排位数值优势。
8. 明确复盘、平衡、长局判定和浏览器验收门槛。
9. 明确单一规则真值，避免实现时多文档口径冲突。
10. 明确抽象协议状态机、行动意图、稳定拒绝码、事件日志和结算幂等底线。
11. 明确后手有效行动权，不只停留在“不被秒杀”。
12. 明确内容映射类别，避免卡牌效果被静默降级。
13. 明确独立测试赛道，不用旧 PVP 绿灯替代真人排位验收。
14. 明确打磨迭代必须用仿真、回放、截图和反馈标签闭环，不靠一次性拍数值。
15. 明确非文件级开发合同包，后续可按最新仓库结构直接拆实施计划。
16. 明确可开发用户故事和验收场景，让后续实施计划能按体验切片拆分。
17. 明确每个用户故事开工前的 Ready 条件和完成后的 Done 证据。
18. 明确首版实施切片、依赖顺序、并行边界和退出证据。
19. 明确各类验收证据产物的证明问题、必要字段、覆盖切片和阻断口径。
20. 明确真人匹配质量、等待扩圈、新玩家保护和宽跨度匹配解释。
21. 明确匹配评分、赛季积分、段位徽章和奖励边界。
22. 明确对局手感、等待状态、行动计时、弱网恢复、动画跳过和移动端确认。
23. 明确预设表情、静音、限频、举报、拉黑、再战、战报分享和实时观战禁用边界。
24. 明确首战引导、推荐谱、排位前演武、首败复盘和新手复杂度预算。
