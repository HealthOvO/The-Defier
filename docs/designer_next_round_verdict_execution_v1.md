# The Defier《逆命者》下一轮功能执行策划 V1

## 0. 文档定位

本文不是新的大版本总策划，而是对 `V9.0《命盘战争》` 当前最值得继续推进的下一轮功能，做一份可直接进入开发拆分的执行蓝图。

它服务于以下已有文档：

- `docs/designer_major_upgrade_planning_v7.md`
- `docs/designer_major_upgrade_requirements_v6.md`

本文只回答一件事：

> 在当前 `seasonBoard / sanctum agenda / aftereffect / challenge weekly` 已经具备基础闭环的前提下，下一轮最值得继续开发什么，才能真正提升可玩性，而不是继续堆摘要层。

---

## 1. 下一轮结论

下一轮主切片应明确锁定为：

## `周结算 -> 押卷结果 -> 债账包 -> 主/旁验证状 -> 回写下周目标`

一句话解释：

> 把已经做出来的“赛季裁定、债账包、主/旁验证状、weekly side verification”从单轮说明层，推进成真正会占用下一轮资源、改变下一轮目标密度、并留下长期记录的硬结算闭环。

这条切片比“继续补观察站筛面”“继续加 PVP 档案字段”“继续加 reward 文案解释”更值得先做，因为：

1. 当前项目最缺的是 `10-20 小时` 结构，而不是单页信息量。
2. `seasonBoard` 已经能产出 `正卷 / 险卷 / 欠卷`、`debtPack` 与 `verificationOrders`，底层并不空白。
3. 玩家现在已经能看见“这周发生了什么”，但还没有足够强地感受到“这周欠下的东西，会真实挤压下一轮怎么打”。
4. 这条链一旦成立，后续继续做 Challenge / PVP / 无尽深化，都会变成同一套押注结构的放大，而不是继续新增平行功能页。

---

## 2. 为什么是这一刀

### 2.1 当前已经成熟的基础

当前代码与 UI 已经具备以下基础：

1. `seasonBoard` 能区分 `sampling / lockline / ranking`，并能生成 `押卷中 / 正卷 / 险卷 / 欠卷`。
2. `debtPack`、主验证状、旁验证状、`weekly challenge` 旁验证入口都已落地。
3. Reward / Sanctum / Build / payload / 浏览器门禁已经能围绕这些字段同步读写。
4. `aftereffect`、`lineage`、`heavenly mandate` 都已具备可消费这些结果的宿主。

### 2.2 当前还不够“好玩”的地方

当前最明显的缺口不是“看不到”，而是“结果没有真正咬住下一轮”：

1. `欠卷` 已经能被生成，但还不像真实会占用下一轮目标位的长期负担。
2. `主验证 / 旁验证` 已经能显示并点击，但成功、失手、拖延对下一周安排的影响还不够硬。
3. `lineage` 和 `aftereffect` 已有记录能力，但还没有明确承接“玩家是怎么清账、怎么押注、怎么拖延”的长期风格。
4. `heavenly mandate` 作为周目标层，仍然更像读题组织层，而不是被硬结算结果反向塑形的排班层。

---

## 3. 用 MDA 看这一轮该补什么

### 3.1 Mechanics

下一轮真正要补的 Mechanics 只有 4 组：

1. `周结算快照`
   把当前周的 `seasonBoard` 结果压成一个稳定的周结算摘要，而不只是章节内即时状态。
2. `债账包生命周期`
   给 `欠卷` 一个真实的打开、占位、清账、降级、拖延状态流。
3. `验证状结果回写`
   让 Challenge / PVP / 无尽的验证结果回写到 `seasonBoard / lineage / mandate`。
4. `强目标位占用`
   当玩家拖延债账或放弃验证时，下一周要被真实占位，而不是只多一条说明文案。

### 3.2 Dynamics

补完这些 Mechanics 后，玩家行为会发生 3 类关键变化：

1. 玩家会开始判断“我现在先补债，还是继续押榜”。
2. 玩家会开始感受到“主验证和旁验证不是两个按钮，而是两种资源分配策略”。
3. 玩家会开始形成长期风格：
   - 稳扎稳打清债
   - 先冲榜再回头补债
   - 经常逆押
   - 经常拖延导致下周排班被挤压

### 3.3 Aesthetics

这一轮应该强化的体验不是“信息更丰富”，而是：

1. 押注感
2. 追账压力
3. 长期身份感
4. 证明一条路线值不值得继续经营的决断感

---

## 4. 下一轮主玩法定义

## 4.1 主目标

让玩家在一周结束后，不只看到“这周结果是什么”，而是必须面对：

1. 我这周押成了什么
2. 我还有哪笔账没清
3. 这笔账会不会挤掉我下周的强目标
4. 我现在该去做主验证，还是先补旁验证
5. 这些选择会不会留下长期风格记录

## 4.2 本轮不追求的事

这一轮不应该追求：

1. 再做一个新页面
2. 再给观察站补更多筛面
3. 再给 PVP / 无尽补更多档案字段
4. 再给奖励页补一层更厚的摘要
5. 大规模重写 `battle / map / pvp / endless`

---

## 5. 系统设计

## 5.1 周结算快照

新增一个轻量 `week verdict ledger` 概念，但不另起新长期根状态机。

建议字段：

1. `weekTag`
2. `seasonId`
3. `phaseId`
4. `verdictResult`
   - `positive_sheet`
   - `risky_sheet`
   - `debt_sheet`
5. `sourceRunId`
6. `primaryDebtPackId`
7. `primaryVerificationOrderId`
8. `sideVerificationOrderId`
9. `resolvedStatus`
   - `open`
   - `verified`
   - `deferred`
   - `degraded`
10. `carryIntoNextWeek`

它的目标不是替代 `seasonBoard`，而是把当前一周“最终押成什么”这件事稳定化，供：

- `heavenlyMandate`
- `lineage`
- `aftereffect`
- 下周 UI 摘要

读取。

## 5.2 债账包生命周期

当前 `debtPack` 已经存在，但下一轮要明确它的生命周期。

建议状态：

1. `open`
   已生成欠卷，尚未进入清账验证。
2. `verification_ready`
   当前周已分配主验证状，可去高压入口处理。
3. `cleared`
   已完成清账验证，退出强目标占位。
4. `deferred`
   本周未处理，下一周继续占用强目标位。
5. `degraded`
   连续拖延后降级成较弱尾账，减少收益但不再无限占用主舞台。

建议规则：

1. 同时活跃的 `open / verification_ready / deferred` 债账包最多 `3` 个。
2. 单个债账包默认清账窗口为 `2` 章或 `1` 周。
3. 同一债账连续 `2` 次拖延后自动降级。
4. 降级不是“没事了”，而是：
   - 奖励变差
   - 谱系记录一次“拖延清账风格”
   - 后效保留轻压强

## 5.3 主验证状 / 旁验证状

下一轮不新增验证入口，只强化“验证结果如何回写”。

建议口径：

1. 主验证状
   - 用于 `清债` 或 `定榜`
   - 成功后会真实改变该周结论
2. 旁验证状
   - 用于补第二份证明
   - 不能替代主验证，但能影响：
     - 结论稳定度
     - 谱系证明质量
     - 下周目标推荐

建议优先保持：

1. `challenge weekly` 作为最轻量旁验证入口
2. `PVP focus` 和 `endless mirror` 作为高压主验证入口

## 5.4 拖延回流

这是下一轮最该真正做硬的点。

拖延不应只是“保留说明文案”，而应至少造成：

1. 下一周 `heavenly mandate` 被占用 `1` 个强目标位
2. 周结算摘要中明确标记“本周先清哪笔账”
3. `seasonBoard` 的推荐动作优先级被改写
4. `lineage` 记录一次：
   - `debt_recovery`
   - `risky_push`
   - `deferred_cleanup`
   三类风格中的一种

## 5.5 回写层

下一轮要明确结果回写去哪里：

1. `seasonBoard`
   - 负责展示当前结果、验证状、下一步建议
2. `heavenlyMandate`
   - 负责把未结债账变成下周强目标位占用
3. `aftereffect`
   - 负责持续施压，不负责定义债账状态
4. `lineage`
   - 负责记录玩家如何清账、如何押榜、如何拖延

---

## 6. Owner / Precedence Matrix

这一轮必须继续坚持单一 owner 原则。

### 6.1 单一 owner

1. `agenda`
   只负责生成债账来源与原始责任。
2. `seasonBoard`
   只负责组织押卷结果、主/旁验证状和当前验证优先级。
3. `heavenlyMandate`
   只负责把未清事项占用成“下周目标排班”。
4. `aftereffect`
   只负责跨章持续压力与风险偏置。
5. `lineage`
   只负责长期记录“你怎么处理这些事”。

### 6.2 绝对禁止

1. `aftereffect` 直接定义“你本周欠了什么账”
2. `lineage` 反过来驱动当前验证建议
3. `heavenlyMandate` 直接改写债账来源解释
4. `seasonBoard` 同时承担“债账来源 + 长期记录 + 周排班”三种职责

---

## 7. 玩家可见界面建议

下一轮 UI 不求多，但要更硬。

## 7.1 洞府总览

必须新增或强化：

1. `本周押卷结果`
2. `待清研究债账`
3. `当前主验证状`
4. `当前旁验证状`
5. `若继续拖延，将挤占下周目标位` 的明确提示

## 7.2 奖励页

奖励页不是继续加说明，而是要成为“本周结算确认页”。

至少包含：

1. 本周押成什么
2. 这次结果是否清掉债账 / 升级结论 / 只是补旁证
3. 下一周会不会被强制排进清账目标

## 7.3 周目标页 / heavenly mandate

下一轮要让玩家看见：

1. 哪个强目标位被欠账占用
2. 哪个目标是自由推进位
3. 当前是“先清债”还是“可继续冲榜”

## 8. 建议的开发切片

## 8.1 主切片 A：周结算快照 + 拖延规则

### 目标

先把“本周结算的结果会留下什么”做硬。

### 最小交付

1. `week verdict ledger`
2. 债账包 `open / deferred / degraded / cleared` 流程
3. `heavenly mandate` 强目标位占用

### 不碰

1. 不新增 Challenge 页面
2. 不重做 PVP / endless 详细 UI

## 8.2 主切片 B：验证结果回写

### 目标

让现有高压入口的结果不再只影响“当前一局”，而能明确回写到长期结构。

### 最小交付

1. 主验证成功 -> 清债 / 升级结论
2. 旁验证成功 -> 提升证明质量 / 强化下周推荐
3. 失手 -> 留高质量反例样本并降级

## 8.3 补位切片 C：谱系记录最小扩展

### 目标

只加最小必要记录，不做奖励层膨胀。

### 最小交付

1. 新增清账风格记录
2. 新增押榜风格记录
3. 新增拖延风格记录

## 9. 工程边界

下一轮建议以以下文件为主：

1. `js/game.js`
   作为 `seasonBoard / heavenlyMandate / payload` 主集成入口
2. `js/core/collection_hub.js`
   负责洞府与 Build 的消费层表达
3. `tests/sanity_season_board_system_checks.js`
   作为主 Node 合同 chokepoint
4. `tests/browser_meta_screen_audit.mjs`
   作为主浏览器门禁 chokepoint

尽量不要在下一轮同时大改：

1. `js/core/map.js`
2. `js/core/challenge_hub.js`
3. `js/core/expedition_hub.js`
4. PVP 全链条

这些模块只做“消费现有结果”的最小接线，不做重构。

## 10. 下一轮验收标准

如果下一轮做完，至少应满足：

1. 玩家能明确看到“这周押成了什么、哪笔账还没清、下一步该去哪里”。
2. 未清债账会真实占用下一周一个强目标位，而不是只在摘要里提醒。
3. `正卷 / 险卷 / 欠卷` 的后续处理不再只是文案，而会影响：
   - `seasonBoard`
   - `heavenlyMandate`
   - `lineage`
4. 主验证与旁验证成功、失手、拖延三条分支，都会有稳定回写。
5. owner / precedence 在冲突场景下仍只有一个系统解释“现在先做什么”。

---

## 11. Node / 浏览器门禁建议

### 11.1 Node 合同

至少新增或强化：

1. `week verdict ledger` 生成与迁移
2. 债账包 `open -> deferred -> degraded -> cleared`
3. 主验证成功 / 失手 / 拖延三路回写
4. 旁验证成功对证明质量与推荐方向的影响
5. `heavenlyMandate` 强目标位占用规则
6. owner / precedence matrix

### 11.2 浏览器审计

至少新增或强化：

1. 洞府页看到：
   - 本周押卷结果
   - 待清债账
   - 主/旁验证状
   - 下周占位提示
2. 奖励页看到：
   - 本次结算结果
   - 是否清账 / 升级 / 拖延
3. payload 与 UI 三处一致：
   - reward
   - collection / sanctum
   - render_game_to_text

---

## 12. 最终建议

下一轮不要再问“还能往哪个页面加东西”，而要只问：

> 现在这条押注链，哪里还没有真正改变玩家下一轮的选择密度？

答案就是：

1. 周结算还不够硬
2. 债账拖延还不够痛
3. 验证结果还不够会回写

因此，下一轮应把开发和策划都集中到这条硬结算闭环，而不是把资源分散到更多并排系统深化上。
