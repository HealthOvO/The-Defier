# The Defier《逆命者》下一大版本策划案 V6

## 0. 文档定位

本文为《逆命者》`V8.0《观星归卷》` 的完整策划案。

目标不是继续横向堆叠新模式，而是把当前已经成型的：

- 挑战 / 观察站样本读题
- 远征路线执行
- 章节结算复盘
- PVP 对手档案读题

升级成一条真正可持续的“修行主线”。

本策划案包含：

1. 版本定位与目标
2. 当前版本诊断
3. 设计支柱
4. 核心玩法升级模块
5. 难度与反馈目标
6. 开发切片与交付节奏
7. QA 与上线门禁
8. 版本验收标准
9. 风险与回滚策略
10. 里程碑结果定义

## 1. 版本定位（V8.0《观星归卷》）

### 1.1 一句话定位

> 让玩家带着课题进章，按样本作答，在结算时真正“交卷并复盘”。

### 1.2 版本主题

`V8.0《观星归卷》` 的主题，不是“再开一个新系统”，而是把当前版本最强的系统语言：

- 读题
- 执行
- 复盘

收束成一个玩家能直接感知的大闭环。

如果说 `V7.0《命途裂变》` 完成的是：

> 把挑战观察站、远征、PVP 的读题语言逐步打通。

那么 `V8.0《观星归卷》` 要完成的是：

> 把这套语言变成每一局都能被执行、被评价、被归档、被继续训练的修行主线。

### 1.3 版本主目标

1. **玩法目标**
   让每章不再只是“推进主线”，而是像完成一份带题目的答卷。
2. **体验目标**
   让玩家不仅知道“这章推荐怎么打”，还知道“我这章到底打成什么样”。
3. **成长目标**
   让失败不只留下结果，而能明确反推下一次该练什么。
4. **联动目标**
   让挑战、观察站、远征、PVP 不再只是并列模块，而是围绕同一条课题相互供血。
5. **稳定性目标**
   版本核心闭环必须具备 Node 门禁、浏览器门禁、截图证据与归档一致性校验。

## 2. 当前版本问题诊断（为 V8.0 定义升级靶点）

### 2.1 现状优势

1. **观察站已经具备长期训练器雏形**
   观察站不再只是样本列表，而是拥有留痕、回放、对比轴、训练标签、演练目标、筛面、排序与预设。
2. **观察站到远征的桥已经打通**
   精选命盘已经能冻结成章节观星线索，并进一步影响推荐路线、命盘共鸣、路线合卷、开战 bonus 与章节答卷收益。
3. **PVP 已经拥有成熟的读题层**
   对手档案、焦点约战、赛季账本、历史交手留痕、多场趋势对照、风险主轴和赛后复盘都已上线。
4. **当前版本的数据与门禁能力较强**
   `render_game_to_text`、构筑快照、章节归档、Node sanity 与浏览器审计已经能支撑高频迭代。

### 2.2 当前关键瓶颈

1. **训练闭环还差最后一段“归卷”**
   玩家已经能读到样本、执行样本，但还没有一个足够强、足够中心化的结算舞台来告诉玩家“这次到底答得怎样”。
2. **多模式之间还缺共同长期目标**
   挑战、远征、PVP 已经互相能看见彼此，但仍更像“并排很强的模块”，而不是同一条修行主线上的不同阶段。
3. **当前系统很会诊断，但还不够会塑造身份**
   现有系统擅长告诉玩家“发生了什么”，但还不够擅长告诉玩家“你这一局到底在主修什么”。
4. **执行反馈尚未足够集中**
   推荐路线、一键锁线、训练标签、路线合卷都已存在，但玩家在章内仍需要自行拼接“我正在按题作答”的感受。

## 3. V8.0 设计支柱

### 支柱 A：每章都要有一张“答卷”

章节开始时，玩家必须明确知道这章的主修课题、建议路线和关键训练目标。

### 支柱 B：每个关键决策都要留下“作答痕迹”

路线、关键节点、资源预留、风险处理、样本贴合度，都必须在章内持续可见，而不是只在结算后回忆。

### 支柱 C：每次结算都要变成下一次训练入口

结算页不仅给奖励和分数，还要给出“完成了什么、偏离了什么、下一次该练什么”。

### 支柱 D：模式之间要围绕同一课题供血

观察站提供题目，远征执行题目，PVP 或挑战提供结业验证，归档系统沉淀修行轨迹。

## 4. 核心玩法升级（必做模块）

## 4.1 修行课题卡（主驱动模块）

### 功能描述

允许玩家把一份观察站 guide，或一份 PVP 焦点 dossier，设为当前章节 / 当前周期的主修课题，并生成简化后的训练目标。

### 设计细则

1. 课题来源至少支持：
   - 观察站精选命盘
   - 观察站历史留痕
   - PVP 焦点对手档案
2. 每张课题卡至少产出：
   - `2-3` 条训练目标
   - 推荐路线
   - 推荐节点
   - 建议资源预留
   - 结业条件
3. 课题卡不应是硬任务，而应是：
   - 明确目标
   - 可偏离
   - 可评价
   - 可回收成建议

### 对可玩性的提升

把“我这局准备练什么”从模糊理解，升级成可执行的主修方向。

## 4.2 章节答卷系统（主体验框架）

### 功能描述

在章节开始时，根据当前课题卡和章节状态，生成一张本章答卷。

### 设计细则

1. 每张答卷至少包含三类目标：
   - 路线目标
   - 执行目标
   - 资源目标
2. 路线目标示例：
   - 优先经过 `营地 / 商店 / 观星台`
   - 优先锁定某条推荐支线
3. 执行目标示例：
   - 第 3 重前保留一次爆发或兜底
   - 在高压段前完成某条训练动作
4. 资源目标示例：
   - 进入高压窗口前保持血线 / 护阵 / 命环节奏处于安全线
5. 答卷必须允许：
   - 完成
   - 部分完成
   - 偏离但成功
   - 彻底偏题

### 对可玩性的提升

玩家不再只是“看推荐”，而是能真正理解本章作答逻辑。

## 4.3 远征作答面板（章内执行中枢）

### 功能描述

在现有远征面板上新增“答卷进度区”，持续反馈玩家当前作答状态。

### 设计细则

1. 面板至少显示：
   - 当前课题名
   - 当前答卷目标
   - 已完成 / 进行中 / 偏离状态
   - 当前路线与样本贴合度
   - 下一条关键提醒
2. 面板必须避免信息过载：
   - 只展示最关键的 `3` 条目标
   - 偏离提示保持 `1` 条主提示
3. 面板状态要与：
   - `observatoryLink`
   - 章节答卷数据
   - 构筑快照
   - 章节结算
   保持一致。

### 对可玩性的提升

玩家章内始终知道自己是在贴题作答，还是已经开始偏题。

## 4.4 观星回响总结卡（结算主舞台）

### 功能描述

在章节结算页 / 奖励页新增版本级主卡“观星回响总结卡”，作为整个大版本的中心反馈舞台。

### 设计细则

1. 总结卡必须回答以下问题：
   - 这章读的是哪份样本
   - 最终走了哪条路线
   - 哪些目标完成了
   - 哪些地方偏离了
   - 这份样本最终带来了什么收益
2. 总结卡要显示：
   - 课题来源
   - 答卷评级
   - 偏题原因
   - 奖励回响
   - 下一次训练建议
3. 视觉上要保证：
   - 一眼能看懂
   - 信息密度高但不凌乱
   - 可以成为截图传播点

### 对可玩性的提升

把当前版本最缺的“最后一锤”补上，让玩家真正感到自己是在交卷，而不是只看一串分数。

## 4.5 答卷评级与偏题诊断

### 功能描述

在既有章节分数之外，新增一套更贴近训练价值的答卷评级。

### 设计细则

1. 首发建议采用 `4` 个维度：
   - 路线贴合
   - 执行稳定
   - 风险处理
   - 收束质量
2. 首发建议采用 `4` 个结果标签：
   - 稳答
   - 险答
   - 偏答
   - 超纲
3. 评级要能解释，而不是只给结论。
4. 偏题诊断至少要覆盖：
   - 路线偏离
   - 高压资源使用错误
   - 关键节点顺序错误
   - 样本目标理解偏差

### 对可玩性的提升

失败和成功都能变成可学习的结构，而不是只剩“赢了 / 输了”。

## 4.6 训练建议回流（失败价值化）

### 功能描述

把章节结算结果自动转成下一次的训练建议，让失败也拥有明确价值。

### 设计细则

1. 每次结算至少输出 `1` 条主建议：
   - 路线类建议
   - 资源类建议
   - 节奏类建议
   - 风险处理建议
2. 建议必须可执行，避免空泛表述。
3. 建议必须能回流到：
   - 观察站
   - 章节答卷
   - 历史归档

### 对可玩性的提升

增强“输了也愿意再来一局”的公平感和成长感。

## 4.7 归卷档案与修行书架（长期目标模块）

### 功能描述

把当前观察站历史留痕与章节答卷结果合并成一条“修行书架”。

### 设计细则

1. 归档至少支持按以下维度筛查：
   - 课题主题
   - 章节
   - 路线
   - 评级
   - 偏题原因
   - 结局
2. 书架至少提供三种阅读视角：
   - 按课题看成长
   - 按章节看答卷史
   - 按偏题原因看训练缺口
3. 书架应支持：
   - 快速回放
   - 重新设为课题
   - 对比最佳样本

### 对可玩性的提升

把“单局复盘”升级成“赛季修行轨迹”，增强长期留存。

## 4.8 结业验证战（挑战 / PVP 联动模块）

### 功能描述

把当前课题引到一个更具象的“结业验证点”，而不是只停留在观察站或远征内部。

### 设计细则

1. 结业验证至少支持两类入口：
   - 挑战验证
   - PVP 焦点验证
2. 玩家完成一组课题后，可获得：
   - 结业标签
   - 课题完成记录
   - 高质量样本归档权重
3. 结业验证不应取代日常流程，而应作为：
   - 训练成果展示
   - 风格证明点
   - 高价值结算节点

### 对可玩性的提升

让 challenge、expedition、pvp 从“互相关联”升级成“互相证明”。

## 4.9 奖励页与章节结算决策化升级

### 功能描述

让奖励页与章节结算页不只是发奖励，还能告诉玩家这次选择与作答结果之间的关系。

### 设计细则

1. 奖励区要显示：
   - 当前补的是哪类短板
   - 更适合下一章还是立即兑现
   - 是否与当前课题方向一致
2. 章节结算区要显示：
   - 本章答卷摘要
   - 关键偏题原因
   - 推荐下一步去向

### 对可玩性的提升

提升“打完这一章之后要去做什么”的清晰度。

## 4.10 UI/UX 可读性升级（信息层）

### 功能描述

统一章前、章内、章后的信息层，让玩家在任何阶段都能快速理解“当前课题”和“当前答题状态”。

### 设计细则

1. 章前重点展示：
   - 本章课题
   - 三条答卷目标
   - 推荐路线
2. 章内重点展示：
   - 当前答卷进度
   - 核心偏离提示
   - 下一条建议动作
3. 章后重点展示：
   - 总结卡
   - 评级
   - 训练建议
4. 移动端必须做到：
   - 两步内看到课题核心信息
   - 不出现横向爆布局
   - 结算卡不因文本量而崩坏

## 4.11 稳定性工程与门禁扩展

### 功能描述

把 `V8.0` 的新闭环做成真正可回归的主链路，而不是只靠人工确认。

### 必做项

1. 关键数据结构统一：
   - 课题卡
   - 章节答卷
   - 答卷评级
   - 偏题诊断
   - 训练建议
2. 新链路必须贯穿：
   - 观察站
   - 远征
   - 奖励页 / 结算页
   - 归档
3. 新系统必须配套：
   - Node sanity
   - 浏览器审计
   - fresh 截图证据目录

## 5. 难度综合评定方案（版本级）

## 5.1 难度分层目标

1. **新手层**
   第一次接触课题系统时，应能在 `30 秒` 内理解“这章推荐怎么打”。
2. **熟练层**
   玩家应能明确知道“自己偏题在哪里”，而不是只体感这局打得乱。
3. **挑战层**
   高水平玩家可以主动偏题、冒险、超纲完成更高收益答卷。

## 5.2 平衡目标（建议值）

1. 标准章节答卷“完整完成”比例：`25% - 40%`
2. 标准章节答卷“部分完成”比例：`35% - 50%`
3. 标准章节答卷“偏题但通关”比例：`15% - 25%`
4. 高压课题“结业完成”比例：`10% - 20%`

> 注：本版本追求的不是“人人满分”，而是“人人都能看懂自己为什么没有满分”。

## 5.3 难度校验门禁

1. 任何高压课题都必须给出至少 `1` 条清晰应对信息。
2. 任何“偏题”判断都必须具备可解释依据，不能只给黑箱结果。
3. 奖励强度必须与答卷难度对等，避免“高难低回报”。

## 6. 开发切片与交付节奏

## 6.1 切片原则

1. 每个切片都要独立可运行、可验证、可回退。
2. 先做统一的数据骨架，再做体验层与长线层。
3. 先把“大闭环”做完整，再继续扩长周期历史聚合。

## 6.2 建议切片

1. **Slice A：课题与答卷骨架**
   - 修行课题卡
   - 章节答卷数据结构
   - 基础 payload 串联
2. **Slice B：章前与章内执行层**
   - 章前课题摘要
   - 远征作答面板
   - 路线贴合与偏离提示
3. **Slice C：章后归卷层**
   - 观星回响总结卡
   - 答卷评级
   - 偏题诊断
4. **Slice D：训练回流与归档**
   - 训练建议回流
   - 修行书架
   - 归卷筛查
5. **Slice E：结业验证与跨模式证明**
   - 挑战结业验证
   - PVP 焦点结业验证
6. **Slice F：可读性与发布收口**
   - 移动端适配
   - 文案统一
   - QA 门禁补齐

## 6.3 首发承诺范围（建议）

如果要把 `V8.0` 压成一个真正可交付、可验收的首发包，建议首发只承诺以下 `4` 件玩家强感知的大事：

1. 章前修行课题卡
2. 远征作答状态面板
3. 章节观星回响总结卡
4. 答卷评级 + 训练建议

与这 `4` 件配套的底座能力同步纳入首发：

- 统一课题 / 答卷数据总线
- 贴题 / 偏题即时反馈

建议延后到第二阶段的内容：

- 归卷档案的深筛与长期书架扩展
- 挑战 / PVP 结业验证的完整跨模式证明链

这样做的目的不是缩水，而是先保证 `V8.0` 最核心的体验闭环：

> 选题 -> 开章定标 -> 章内作答 -> 章节归卷 -> 训练回流

先完整成立，再继续拉长长期深度。

## 7. QA 与“无阻塞缺陷”上线标准

## 7.1 测试矩阵

1. **语法层**
   相关脚本修改必须通过 `node --check`。
2. **逻辑层**
   课题卡、章节答卷、评级、训练建议至少各有 1 组 sanity 合同。
3. **浏览器层**
   章前、章内、章后至少各有 1 条真实 UI 审计链路。
4. **Playwright 冒烟**
   必须能完成：
   - 读取课题
   - 进入章节
   - 作答推进
   - 结算交卷
5. **人工抽检**
   每个核心切片至少保留：
   - 1 张桌面图
   - 1 张移动端图
   - 1 个 fresh 产物目录

## 7.2 上线门禁（必须全部满足）

1. Node 门禁全通过。
2. 浏览器审计全通过。
3. 新总结卡、作答面板、归卷书架无关键 UI 缺失或挤爆。
4. `render_game_to_text`、构筑快照、章节归档三处口径一致。
5. 核心流程无 P0 / P1 阻塞缺陷。

## 8. 版本验收标准（产品视角）

1. 玩家在开章前能一句话说出“这章建议我怎么打”。
2. 玩家在章内能随时知道自己是在贴题还是偏题。
3. 玩家在结算时能看懂“为什么会是这个结果”。
4. 玩家失败后能拿到一条明确、可执行的训练建议。
5. 玩家会觉得自己是在完成一条修行主线，而不是在多个模块间来回跳转。

## 9. 风险与回滚策略

## 9.1 主要风险

1. 跨模块改动过多，容易变成“每个地方都加一点”。
2. 信息量增加后，章内界面可能过载。
3. 评价系统若解释不足，容易让玩家觉得“被系统误判”。

## 9.2 控制策略

1. 以“观星回响总结卡”为版本中心，所有模块围绕它收束。
2. 章内仅展示最关键的 `3` 条目标与 `1` 条偏离提示。
3. 所有评级与建议都必须具备可追溯的解释文本。

## 9.3 回滚预案

1. 若跨模式结业验证风险过高，可先保留章节答卷主链，临时关闭 PVP / 挑战结业验证。
2. 若章内信息层过载，可保留章前摘要与章后总结，收缩章内实时提示密度。
3. 若评级反馈争议过大，可先保留答卷目标与总结卡，延后完整评级权重上线。

## 10. 里程碑结果定义

`V8.0《观星归卷》` 成功的标志，不是“多了更多面板”，而是：

1. 玩家会说“这一章我是在按一份题目打”。
2. 玩家会说“我知道这次为什么答偏了”。
3. 玩家会说“我下一次想再试一把，把这份课题答好”。
4. 挑战、观察站、远征、PVP 首次像同一条修行主线的不同阶段。
