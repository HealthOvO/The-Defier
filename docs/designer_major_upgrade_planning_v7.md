# The Defier《逆命者》下一大版本策划案 V7

## 0. 文档定位

本文为《逆命者》`V9.0《命盘战争》` 的大版本策划案。

这次大版本的目标，不是继续横向堆更多孤立功能页，而是把当前已经成熟的：

- 观察站样本读题
- 远征章节作答
- 洞府议程 / 锁线契约 / 残卷回收
- 地图工程偏置
- 无尽赛季读题
- PVP 风险画像与复盘

收束成一条真正的跨模式主循环。

本文包含：

1. 版本定位与主目标
2. 当前版本诊断
3. 设计支柱
4. 核心升级模块
5. 玩家 10-20 小时循环设计
6. 开发切片与交付节奏
7. QA 与上线门禁
8. 风险与反目标
9. 版本验收标准

## 1. 版本定位（V9.0《命盘战争》）

### 1.1 一句话定位

> 让玩家围绕同一赛季题面，在观察站读题、在远征作答、在洞府押注、在 PVP / 无尽验证，并把结果沉淀成长期谱系。

### 1.2 为什么是现在做

当前项目的 `1-3 小时` 单次体验已经很强：

- 观察站能给训练标签、样本筛面、失手剖面与对照轴
- 远征能承接观星线索与路线锁线
- 洞府议程已经形成“立项 -> 章中处置 -> 锁线契约 -> 结题 / 残卷回收”的完整承诺循环
- PVP 与无尽也已经拥有风险主轴、对策提示、赛季账本和复盘语言

当前真正缺的，不是更多单点功能，而是 `10-20 小时` 维度的中长期结构：

- 玩家为什么要连续玩 5-10 把
- 为什么本周和下周的练习重心会不同
- 为什么远征、观察站、洞府、PVP、无尽不是各自精彩，而是在经营同一件事

`V9.0《命盘战争》` 要解决的，就是这层“长期修行结构”的缺口。

### 1.3 版本主目标

1. **主轴目标**
   把观察站、远征、洞府、PVP、无尽统一到同一条“赛季命盘主轴”上。
2. **中期目标**
   给玩家稳定的 `周循环 -> 长期追账 -> 跨章后果 -> 赛季结算` 结构。
3. **选择目标**
   让玩家不只是“看很多信息”，而是持续做“押哪条题面、签哪类契约、承担什么后果”的选择。
4. **身份目标**
   让玩家留下“我是靠哪种命盘、哪种契约风格、哪类节点偏好打出来的”长期档案。
5. **工程目标**
   让大版本新增内容尽量沿洞府议程这条成熟状态机主链推进，而不是无控制地多线并发重写。

## 2. 当前版本问题诊断

### 2.1 当前版本优势

1. **局部系统已经非常成熟**
   观察站、洞府议程、章节奖励页、PVP 档案、无尽账本都已经具备完整的信息表达能力。
2. **多系统语言已经部分统一**
   风险主轴、对策、预留、样本、训练、归卷、工程这些概念已经跨页面复用。
3. **洞府议程是最强的新主链**
   它已经具备状态机、资源消耗、章中分支、契约代价、失败回收与奖励 / payload 投影。
4. **自动化门禁已经能承接大版本迭代**
   Node 合同测试、save migration、browser meta audit 都已经具备迭代基础。

### 2.2 当前关键缺口

1. **系统之间还是“并排联动”，不是“共同目标”**
   玩家能看到它们互相影响，但还不够明确自己这一周到底在经营什么。
2. **失败已经有价值，但还不是长期资源**
   残卷回收、失手剖面、赛后复盘都有了，但还没有统一沉淀成中长期成长燃料。
3. **长期身份感不足**
   现在更擅长告诉玩家“这把发生了什么”，还不够擅长告诉玩家“你长期主修什么”。
4. **下一步该玩什么仍不够集中**
   功能很多，但缺一个能明确组织玩家时间投入的“大目标面板”。

## 3. V9.0 设计支柱

### 支柱 A：所有模式都围绕同一条赛季题面供血

观察站负责读题，远征负责答题，洞府负责押注与转化，PVP / 无尽负责验证，图鉴与谱系负责沉淀。

### 支柱 B：新版本继续沿洞府议程主链纵深

大版本的骨架不另起炉灶，而是继续把议程、契约、失败回收、结题工程做成跨模式状态中枢。

### 支柱 C：失败不是补偿，而是研究燃料

失败样本、残卷、崩盘账本、PVP 复盘都要能回流到下一轮目标和长期成长。

### 支柱 D：长期进度记录“你怎样赢”，而不只记录“你赢过”

角色、流派、节点偏好、议程倾向、契约风格都应被长期记录。

### 支柱 E：每一层都必须可验证、可回滚、可收口

大版本设计必须天然考虑 save migration、payload 投影、文本同步与浏览器审计，不允许最后阶段再补救。

### 支柱 F：长期结构必须通过硬结算驱动下一轮押注

周循环、界痕后效、赛季天道盘与命盘谱系都不能只做“会总结玩家”的组织层；它们必须通过统一的结算层，把成功、失手、欠账和验证结果压成下一轮真实可经营的押注。

## 4. 核心升级模块

## 4.1 天道敕令周循环板

### 功能描述

每周生成一组跨模式目标，把远征、观察站、PVP、无尽统一到同一个“本周修行安排”里。

### 设计细则

1. 周循环至少分为三条主线：
   - `远征线`：章节答卷、节点组合、议程结题、契约兑现
   - `训练线`：观察站复刻、训练标签、历史样本对照
   - `对抗线`：PVP、镜像演武、无尽赛季表现
2. 每周给出 `6-9` 条任务，允许玩家自由选择完成顺序。
3. 奖励优先产出：
   - 谱系推进点
   - 赛季资源
   - 洞府研究材料
   - 轻量收藏 / 外观 / 称号类目标
4. 周循环必须天然读取既有结算字段，而不是要求新增一整套重复埋点。

### 玩家体验目标

让玩家明确知道“这周回来玩什么”，而不是面对很多强系统却没有明确起手点。

## 4.2 命盘谱系

### 功能描述

把玩家长期使用的角色、流派、节点倾向、议程路线和契约风格，沉淀成长期专精档案。

### 设计细则

1. 首发建议做四条谱系：
   - `角色谱系`
   - `流派谱系`
   - `节点谱系`
   - `研究谱系`
2. 研究谱系中必须记录：
   - 议程类型偏好
   - 处置倾向
   - 契约风格
   - 残卷回收 / 结题风格
3. 奖励优先以：
   - 展示型成就
   - 检索便利
   - 轻量预设特权
   - 非破坏性长期奖励
   为主，避免首发就做重数值压制。

### 玩家体验目标

把“我这把怎么玩”升级成“我长期主修什么、擅长怎样打成”。

## 4.3 界痕抉择与契约后效

### 功能描述

把洞府议程从“本章承诺”扩成“跨章后果系统”。结题、未成、残卷回收与契约兑现，都能留下持续 `2-3` 章的界痕后果。

### 设计细则

1. 首发只做有限模板，建议控制在 `3-4` 类：
   - `路线偏置型`
   - `敌情 / 追猎型`
   - `资源税负型`
   - `档案 / 归卷收益型`
2. 契约必须不仅有签约代价，还要有后续追账：
   - 兑现时得到工程倾斜
   - 未兑现时留下负债 / 偏题后果
   - 残卷回收时留下较轻但仍可追踪的补卷义务
3. 后果必须能投影到：
   - 洞府总览
   - 远征面板
   - 奖励页 / payload
   - 章节摘要与推荐路线提示

### 玩家体验目标

让“这一章的选择”真的会延伸到后面几章，而不是章末结算后立刻蒸发。

## 4.4 赛季天道盘

### 功能描述

用一个更高层的赛季结构，把周循环、谱系推进、界痕后果和跨模式验证统一到同一个赛季盘面。

### 设计细则

1. 每季必须有一个总题面：
   - 决定本季重点风险
   - 决定本季推荐样本方向
   - 决定本季 PVP / 无尽 / 远征的重点读题
2. 每季至少拆成三个阶段：
   - `采样期`
   - `锁线期`
   - `定榜期`
3. 赛季天道盘不是新模式，而是各模式的组织层：
   - 观察站产出样本
   - 远征执行样本
   - 洞府压成议程与契约
   - PVP / 无尽给出结业验证
   - 谱系吸收长期成果

### 玩家体验目标

让玩家感受到“这个赛季我们在打什么”，而不是只是无尽与 PVP 各自换一批数字。

## 4.5 模式供血关系

### 目标

把现有模式串成统一的供血关系，而不是继续做页面并排扩展。

### 关系定义

1. `观察站`
   提供课题、历史样本、训练标签、失手剖面。
2. `远征`
   承担章节作答、节点执行、议程推进、契约兑现与失败回收。
3. `洞府`
   把章节答卷压成议程、契约、界痕与工程。
4. `PVP / 无尽`
   提供极端压力验证、镜像验证、赛季成绩与反例样本。
5. `图鉴 / 谱系`
   沉淀长期身份与中长期目标。

## 4.6 赛季押卷与结业验证（首发后优先扩展）

### 功能描述

把已经落地的天道敕令、洞府议程、界痕后效与赛季天道盘，推进成一条真正可结算的 `周结算 -> 季押卷 -> 结业验证 -> 清账回流` 主链。

这块不是再开一个新模式，而是给当前已经存在的“训练 / 远征 / 契约 / 后效 / ranking”补上一个更硬的结算层，让玩家知道：

- 这周到底押成了什么
- 哪些账已经结清
- 哪些账被转成了下一轮风险
- 什么时候值得把当前路线投向挑战 / PVP / 无尽去做真正验证

### 设计细则

1. 每次章节结算或周结算后，系统都应把当前结果压成 `押卷结果`：
   - `正卷`：当前主轴被稳定兑现，可直接转成赛季推进或谱系证明
   - `险卷`：虽然完成了主目标，但留下了明显偏题代价或后续压力
   - `欠卷`：契约 miss、残卷回收或验证失败留下的研究债账
2. `欠卷` 不应只是失败描述，而要形成可追踪的 `研究债账包`，至少记录：
   - 来源议程 / 契约
   - 欠账主题
   - 建议验证入口
   - 清账窗口
   - 清账收益
   - 失败后追加负担
3. 洞府议程继续作为 `研究债账包` 的单一 owner：
   - 议程负责生成债账
   - `seasonBoard` 负责把债账组织成 `验证状`
   - `aftereffect` 负责提供持续偏压
   - `lineage` 负责记录玩家如何清账与如何押注
4. `seasonBoard` 进入 `ranking` 后，不应只是给“去试炼 / 精英 / 战斗 / 禁术”的路线提示，还应生成 `1` 个主验证状与 `1` 个可选旁验证状：
   - `清债验证`：优先处理欠卷与后效债务
   - `定榜验证`：把已经成形的路线投向更高价值的赛季证明
   - `逆押验证`：允许玩家用更高风险换更高研究价值，但失败会留下更重欠卷
5. 论证入口首发建议只复用现有高压场景，不新增独立大页面：
   - `挑战验证`
   - `PVP 焦点验证`
   - `无尽镜像验证`
6. 成功、失手与拖延都必须产生真实回流：
   - 成功：清空对应债账、提升季盘阶段结论、补入谱系证明记录
   - 失手：保留一份高质量反例样本，并把债账降级成较轻的恢复目标
   - 拖延：不立即判死，但会在下一周目标里占用强目标位，挤压其他成长机会

### 建议参数锚点

1. 每周只突出 `2-3` 个强目标，不把所有债账都摊给玩家。
2. 同时处于活跃态的 `研究债账包` 最多 `3` 个，避免局外总览变成纯清单管理。
3. 单个债账包默认给 `2` 章或 `1` 周的清账窗口，超过窗口后自动降级为较弱的长期尾账。
4. 每周默认只给 `1` 个主验证状与 `1` 个旁验证状，控制选择密度与 QA 复杂度。
5. 若玩家连续 `2` 次未能完成同一债账，系统应自动给出更保守的恢复建议，避免长期锁死。

### 玩家体验目标

把“我知道下一步该练什么”升级成“我知道现在该不该押、押了会拿到什么、没押成会欠下什么、什么时候值得去打一场真正的证明局”。

## 5. 玩家 10-20 小时循环

`V9.0` 后，玩家的中期循环应当演进为：

1. 在观察站 / 赛季盘读取本周题面
2. 从天道敕令中选本周主目标
3. 带着目标进入远征
4. 在洞府立项议程、处理中段处置、补签契约
5. 章节结算后把结果压成 `正卷 / 险卷 / 欠卷`，并决定是否形成研究债账
6. 将结果回流到谱系、周循环与赛季盘，并生成下一轮 `验证状`
7. 用挑战 / PVP / 无尽去验证当前路线是否站得住、是否值得清账或定榜
8. 再回到观察站与洞府，围绕更明确的押卷结论开启下一轮押注

这个循环的核心，不再只是“打一把变强一点”，而是：

> 每一把都在为同一条赛季命盘战争线提供样本、押注、欠账、反证和长期身份。

## 6. 开发切片与交付节奏

## 6.1 第一阶段：主链扩容

### 玩法目标

1. 沿洞府议程主链补更多议程候选
2. 增加契约后效 / 界痕字段
3. 把周循环与谱系需要的最小数据源先接到状态机里

### 工程目标

1. 先在 `js/game.js` 完成状态源扩容
2. 先扩 Node 合同测试，再放 UI
3. 先定义字段，再决定展示方式

### 通过标准

1. `sanity_dongfu_agenda_system_checks`
2. `sanity_save_migration_checks`
3. `sanity_expedition_state_checks`
4. 相关 payload / reward meta 断言

## 6.2 第二阶段：玩家可见化与跨页同步

### 玩法目标

1. 让天道敕令、谱系、界痕后效在洞府 / 奖励页 / 远征摘要中可见
2. 让玩家知道当前长期主线、当前债务、当前赛季目标

### 工程目标

1. 统一 `collection_hub / expedition_hub / render_game_to_text` 的投影口径
2. 扩浏览器 probe，覆盖成功 / 失败 / 追账三条链路
3. 收口字段命名和文案一致性

### 通过标准

1. `tests/browser_meta_screen_audit.mjs` 至少覆盖：
   - 一条成功兑现链
   - 一条失败回收链
   - 一条后效 / 追账链
2. 所有 UI 与 payload 字段保持一致

## 6.3 第三阶段：赛季封板与发布硬化

### 玩法目标

1. 接入赛季天道盘
2. 把周循环、谱系、界痕后果与跨模式验证统一收口
3. 做最终外宣级版本表达

### 工程目标

1. 不再大规模新增状态机分支
2. 专注 save migration、清档、战败分支、浏览器 pageerror 与发布门禁
3. 统一版本说明、对外文案和游戏内更新页

### 通过标准

1. 全量 Node 清单
2. 浏览器元页面审计
3. 存档迁移回归
4. 文案三处同步

## 6.4 当前建议优先切片：周结算 / 季押卷 / 结业验证

### 玩法目标

1. 让天道敕令不再只给任务，而是能在周结算后产出明确的押卷结果。
2. 让 `seasonBoard ranking` 从“路线提醒”升级成“验证状组织层”。
3. 让失败不只是留下后效文案，而是形成可恢复、可清账、可再押注的研究债账。

### 工程目标

1. 继续沿 `sanctum agenda -> aftereffect -> seasonBoard` 主链推进，不额外并发创建新的长期根状态机。
2. 先定义 `研究债账包 / 验证状 / 押卷结果` 的最小字段，再决定 Challenge / PVP / 无尽各自投影细节。
3. 先补 owner / precedence matrix，再落 UI，避免 `agenda / seasonBoard / aftereffect / mandate / lineage` 抢解释权。

### 通过标准

1. 玩家能在洞府或奖励页一眼看到：本周押成了什么、欠着什么、下一步该去哪里验证。
2. 同时存在议程焦点、界痕后效和季盘指引时，只有一个 owner 真正驱动地图或验证方向。
3. 成功、失手、拖延三条分支都能把结果稳定写回周循环、谱系和季盘，而不是只留文案摘要。

## 7. QA 与上线门禁

### 7.1 Node 合同测试必须扩的方向

1. 周循环任务结算
2. 谱系计数口径
3. 界痕后果跨章存续
4. 契约未兑现追账
5. 残卷回收与长期资源回流
6. `正卷 / 险卷 / 欠卷` 三类押卷结果的收口与降级
7. owner / precedence matrix 在冲突场景下仍保持单一驱动口径

### 7.2 浏览器审计必须扩的方向

1. 洞府总览展示长期主线与追账状态
2. 奖励页与 payload 同步展示界痕 / 赛季目标
3. 失败路径不丢状态
4. 新 UI 在桌面和移动端都可读
5. 验证状、押卷结果与欠账提示在洞府 / 奖励页 / payload 中口径一致

### 7.3 文档同步门禁

规划与发布至少同步四处：

1. `docs/designer_major_upgrade_planning_v7.md`
2. `docs/designer_major_upgrade_requirements_v6.md`
3. `progress.md`
4. 封板后再更新 `game-intro.html` 与游戏内更新页

## 8. 风险与反目标

### 8.1 本版本最需要规避的方向

1. 继续只给观察站加更多筛面和排序
2. 继续只给 PVP 加更多档案字段
3. 继续只给洞府议程加更多文案分支但没有长期后果
4. 同时重写 battle、地图生态、PVP、无尽和新局外系统
5. 把结业验证做成一层新奖杯或新徽章，却没有真正影响下一轮押注与清账

这些方向看起来很“大”，但容易做成“信息越来越厚、可玩选择没更深”。

### 8.2 工程反目标

1. 不做一套完全脱离议程主链的新长期系统
2. 不让多个高耦合文件同时无边界改口
3. 不在测试口径未定之前先改对外版本说明
4. 不让 `seasonBoard / heavenlyMandate / aftereffect / agenda / lineage` 在同一问题上同时做 owner

## 9. 版本验收标准

如果 `V9.0《命盘战争》` 成功，玩家应能在 10-20 小时内清楚感受到：

1. 我这一周有明确修行目标
2. 我这一季在经营一条具体命盘主轴
3. 我签过什么契约、欠着什么账、留下了什么界痕
4. 我的长期身份不是“打过很多把”，而是“擅长怎样押注和怎样打成”
5. 观察站、远征、洞府、PVP、无尽不是五个系统，而是一条完整的赛季修行链
6. 我知道当前哪些赌注值得现在去证明，哪些账必须先补，再决定下一次押注

如果以上五点有三点以上不成立，那么这个版本仍然只是“功能很多”，还不是一次真正的大版本升级。
